# v1.0 回滚手册

> 口径说明
>
> 文中若出现旧的 `主数据` 说法，当前统一理解为 `词典建设`。

## 1. 文档目的

本手册用于在 `v1.0` 发布后出现问题时，指导如何回滚到上一已知稳定版本。

## 2. 适用场景

适用场景包括：

- 发布后出现阻断级运行风险
- 关键 runtime 接口异常
- 关键节点无法收敛
- 线上效果与发布预期明显不一致

## 3. 回滚原则

1. 回滚是发布域动作，不是主数据修复动作
2. 回滚对象是已知稳定 release
3. 回滚前要先记录当前问题和当前正式版本
4. 回滚后仍要补充证据和问题归档

## 4. 回滚前检查

回滚前必须确认：

1. 当前正式版本 releaseId
2. 目标回滚版本 releaseId
3. 目标回滚版本为已知稳定版本
4. 当前问题是否达到回滚阈值
5. 回滚负责人和审核负责人已确认

## 5. 执行步骤

### 5.1 确认目标版本

1. 进入“版本发布 -> 版本列表”
2. 找到上一已知稳定版本
3. 打开版本详情
4. 确认版本状态、发布时间和历史稳定性

### 5.2 执行回滚

1. 由具备发布权限的角色执行回滚动作
2. 记录：
   - 当前正式版本
   - 目标回滚版本
   - 执行时间
   - 执行人

### 5.3 回滚后核查

回滚后立即检查：

1. 当前正式版本是否已切换
2. 运行节点是否开始向目标版本收敛
3. 关键 runtime 接口是否恢复
4. 发布后风险是否下降
5. 工作台是否仍存在阻断级问题

## 6. 证据留档

建议保留：

- 回滚前 release 详情
- 回滚目标 release 详情
- 回滚执行结果
- 回滚后运行节点状态
- 回滚后接口验证结果
- 问题说明与处置结论

## 7. 回滚后动作

回滚完成后应继续：

1. 记录问题根因
2. 标记本次发布为失败发布
3. 确认是否需要冻结后续发布窗口
4. 进入问题修复与重新发布流程

## 8. 当前用途

本文档当前用于：

- `JOB-030` 的发布准备文档
- `v1.0` 线上问题处置的回滚执行基线
