## 1. 文档目的

本文件在以下文档基础上继续细化新后台的技术改造方案：

- [15-后台管理系统重构方案](./15-后台管理系统重构方案.md)

- 新后台需要新增哪些聚合接口
- 如何与当前 `/admin` MVP 并行共存

## 2. 核心约束

### 2.1 不破坏现有 MVP

以下现有内容必须继续保留：

- `/admin`
- `/test-client`
- 当前 `/api/admin/*`
- 当前 `/api/runtime/*`
- 当前 `/api/simulate`
- 当前 `/ws/runtime/correct`

新后台建设不得以破坏式重构现有原型为前提。

### 2.2 复用现有核心词典模型

新后台必须继续复用当前词典主数据模型：

- `terms`
- `aliases`
- `term_rules`
- `pinyin_profiles`

不能为了页面重构而重新定义一套平行词典真值表。

### 2.3 新增中间层，不直接污染主表语义

对于批量导入、聚合视图、页面工作流等需求，优先新增：

- 批次层
- staging 层
- 聚合接口层

而不是把所有中间态直接塞进 `terms` 或 `aliases`。

## 3. 当前实现的可复用基础

当前原型已经有较稳定的底层对象，可直接复用。

### 3.1 词典主数据

- `terms`
- `aliases`
- `term_rules`
- `pinyin_profiles`

### 3.2 审核与发布

- `review_tasks`
- `releases`
- `release_terms`
- `gray_policies`

### 3.3 验证与回流

- `validation_cases`

### 3.4 审计与运行统计

- `audit_logs`
- `runtime_hourly_stats`
- `runtime_hourly_terms`
- `runtime_peak_stats`

这些表仍然是新后台的底层 source of truth。

## 4. 新后台新增数据对象建议

新后台建议新增一组“导入中心”与“聚合展示”相关对象。

## 4.1 导入批次主表 `import_jobs`

作用：

- 表示一次完整导入任务
- 承接上传、清洗、预览、确认、落库、审核衔接

建议字段：

- `job_id`
- `job_type`
- `source_type`
- `template_code`
- `template_version`
- `status`
- `submitted_by`
- `confirmed_by`
- `summary`
- `created_at`
- `confirmed_at`
- `finished_at`

建议状态：

- `uploaded`
- `parsed`
- `preview_ready`
- `confirmed`
- `imported`
- `failed`
- `cancelled`

## 4.2 导入文件表 `import_job_files`

作用：

- 记录导入批次关联的原始文件
- 支撑原始文件归档、回放与追踪

建议字段：

- `file_id`
- `job_id`
- `file_role`
- `original_name`
- `stored_path`
- `content_type`
- `file_size`
- `checksum`
- `uploaded_at`

`file_role` 建议支持：

- `source_file`
- `error_report`
- `preview_snapshot`

## 4.3 导入行表 `import_job_rows`

作用：

- 保存清洗后的逐行记录
- 支撑预览、错误定位、局部确认

建议字段：

- `row_id`
- `job_id`
- `row_no`
- `raw_payload_json`
- `normalized_payload_json`
- `target_term_key`
- `status`
- `error_code`
- `error_message`
- `decision`
- `created_at`
- `updated_at`

`status` 建议支持：

- `parsed`
- `warning`
- `error`
- `ready`
- `imported`
- `skipped`

`decision` 建议支持：

- `pending`
- `accept`
- `skip`
- `merge_existing`

## 4.4 导入结果表 `import_job_results`

作用：

- 保存导入结果汇总
- 为页面、统计和审计提供稳定结果对象

建议字段：

- `result_id`
- `job_id`
- `new_term_count`
- `updated_term_count`
- `new_alias_count`
- `updated_alias_count`
- `skipped_count`
- `error_count`
- `imported_by`
- `imported_at`

## 4.5 模板定义表 `import_templates`

作用：

- 管理导入模板元信息
- 支撑模板下载、模板版本和格式说明

建议字段：

- `template_code`
- `template_name`
- `template_version`
- `import_type`
- `file_format`
- `schema_json`
- `example_file_path`
- `template_file_path`
- `enabled`
- `created_at`
- `updated_at`

说明：

- 这张表可以先用 repo 配置文件启动，后续再演进为 DB 管理
- 关键目标是让“模板和字段格式”成为正式产品对象，而不是散落在文档里

## 5. 现有表如何在新后台中使用

## 5.1 `terms`

继续作为标准词主表。

新后台中：

- 词条列表页的主数据来自 `terms`
- 词条详情页的基础信息来自 `terms`

## 5.2 `aliases`

继续作为错误词/别名表。

新后台中：

- 错误词管理区直接读写 `aliases`
- dashboard 上的“错误词库规模”仍以 `aliases` 聚合得到

## 5.3 `term_rules`

继续作为词条规则表。

新后台中：

- 词条详情页规则区直接读写该表

## 5.4 `pinyin_profiles`

继续作为拼音治理表。

新后台中：

- 词条详情页拼音治理区直接读写该表

## 5.5 `review_tasks`

继续作为统一审核任务表。

新后台中：

- 审核中心直接聚合该表
- 导入批次确认任务可考虑复用该表，或先单独实现确认流程，再决定是否统一

## 5.6 `releases` / `release_terms` / `gray_policies`

继续作为发布中心底层表。

新后台中：

- 发布中心直接读这些表
- 但页面不直接暴露表结构，而是走聚合接口

## 5.7 `validation_cases`

继续作为验证样本库。

新后台中：

- 样本与回流中心主数据来自该表
- 词条详情页会关联展示该表中与词条相关的记录

## 6. 导入链路的目标数据流

建议新导入链路如下：

1. 用户下载模板
2. 用户上传原始文件
3. 创建 `import_jobs`
4. 保存原始文件到 `import_job_files`
5. 解析并清洗为 `import_job_rows`
6. 用户查看预览并确认
7. 系统把确认通过的数据写入：
   - `terms`
   - `aliases`
   - 必要时默认初始化 `term_rules`
   - 必要时默认初始化 `pinyin_profiles`
8. 生成 `import_job_results`
9. 进入审核链路

关键原则：

- 导入中间态不直接污染正式词典数据
- 用户确认前，正式词典不变
- 导入完成后，正式词典与导入批次可互相追踪

## 7. 新后台接口分层建议

建议把新后台接口单独放到新命名空间，避免与现有 `/api/admin/*` 语义混杂。

建议命名空间示例：

- `/api/console/*`

具体路径名后续可再定，但核心原则是：

- 与现有 `/api/admin/*` 隔离
- 现有接口继续保留

## 7.1 词条中心聚合接口

目标：让前端以“词条对象”工作，而不是自己拼多张表。

建议接口：

- `GET /api/console/terms`
- `GET /api/console/terms/{termId}`
- `POST /api/console/terms`
- `PUT /api/console/terms/{termId}`
- `POST /api/console/terms/{termId}/submit-review`
- `POST /api/console/terms/{termId}/disable`

其中 `GET /api/console/terms/{termId}` 应返回聚合结果：

- 基本信息
- aliases
- rules
- pinyinProfile
- reviewSummary
- releaseSummary
- validationSummary
- sourceSummary

## 7.2 导入中心接口

目标：让文件导入成为正式产品流程。

建议接口：

- `GET /api/console/import/templates`
- `GET /api/console/import/templates/{templateCode}/download`
- `POST /api/console/import/jobs`
- `GET /api/console/import/jobs`
- `GET /api/console/import/jobs/{jobId}`
- `GET /api/console/import/jobs/{jobId}/rows`
- `POST /api/console/import/jobs/{jobId}/confirm`
- `POST /api/console/import/jobs/{jobId}/cancel`
- `GET /api/console/import/jobs/{jobId}/errors/download`

说明：

- `POST /api/console/import/jobs` 负责上传文件并创建批次
- `confirm` 负责把 staging 数据写入正式词典表
- 模板下载是必须能力

## 7.3 审核中心接口

建议接口：

- `GET /api/console/reviews`
- `GET /api/console/reviews/{taskId}`
- `POST /api/console/reviews/{taskId}/approve`
- `POST /api/console/reviews/{taskId}/reject`

说明：

- 底层可复用现有 `review_tasks`
- 新接口主要负责聚合视图和更稳定的前端返回结构

## 7.4 发布中心接口

建议接口：

- `GET /api/console/releases`
- `GET /api/console/releases/{releaseId}`
- `POST /api/console/releases/build`
- `POST /api/console/releases/{releaseId}/submit-review`
- `POST /api/console/releases/{releaseId}/publish`
- `POST /api/console/releases/{releaseId}/rollback`
- `GET /api/console/releases/{releaseId}/gate`
- `GET /api/console/releases/{releaseId}/validation`

说明：

- 这里重点增加 gate/validation 聚合视图
- 让页面能直接解释“为什么能发、为什么不能发”

## 7.5 样本与回流中心接口

建议接口：

- `GET /api/console/validation-cases`
- `GET /api/console/validation-cases/{caseId}`
- `POST /api/console/validation-cases`
- `POST /api/console/validation-cases/import`
- `POST /api/console/validation-cases/{caseId}/disable`

补充聚合接口：

- `GET /api/console/terms/{termId}/validation-cases`
- `GET /api/console/validation-cases/{caseId}/related-terms`

## 8. 聚合接口与现有实现的关系

建议原则是：

1. 现有 `/api/admin/*` 继续服务 MVP 页面
2. 新后台优先新增 `/api/console/*`
3. `/api/console/*` 内部可以调用现有 service / DB 方法
4. 对外不再暴露“让前端自己拼多表”的细碎接口

也就是说：

- 现有实现作为底层能力
- 新后台通过聚合接口重新编排

## 9. 模板与文件格式能力落地建议

模板能力不能只停留在文档，建议产品化为正式接口和资产。

### 9.1 第一阶段建议

先把模板文件放在 repo 中，配套配置清单。

例如：

- `prototype/config/import_templates/`

包含：

- 模板文件
- 示例文件
- schema 定义
- 字段说明

### 9.2 第二阶段建议

后续再演进为：

- DB 中维护模板元信息
- 页面动态读取模板列表
- 支持模板版本管理

### 9.3 模板内容要求

每种模板至少应定义：

- 字段名
- 字段含义
- 是否必填
- 示例值
- 合法值范围
- 重复处理规则
- 导入后的目标对象说明

## 10. 导入确认时的入库策略建议

导入确认写入正式表时，建议采用以下策略：

### 10.1 `terms`

按 `category_code + canonical_text` 判断是否已存在。

- 不存在：新建 `terms`
- 已存在：进入“合并/更新”逻辑

### 10.2 `aliases`

按 `term_id + alias_text` 去重。

- 不存在：新增 alias
- 已存在：忽略或更新来源信息

### 10.3 `term_rules`

若模板未提供规则：

- 初始化默认规则

若模板提供规则：

- 写入标准化后的规则对象

### 10.4 `pinyin_profiles`

默认不强制批量导入复杂拼音画像。

首阶段建议：

- 自动生成默认画像
- 人工后续在词条详情页调整

## 11. 审计与追踪建议

新后台必须保证以下追踪关系可查：

- 某个词条来自哪个导入批次
- 某个导入批次产生了哪些词条
- 某个错误词来自哪个导入文件
- 某个验证失败样本关联哪些词条
- 某个 release 包含哪些导入批次形成的词条变更

建议：

- 重要入库动作继续写入 `audit_logs`
- 导入批次与词条建立来源映射关系

## 12. 实施顺序建议

建议按以下顺序实施。

### 阶段 1：聚合接口只读层

目标：

- 先做新后台可读视图
- 不改变现有写入流程

产出：

- `/api/console/terms`
- `/api/console/terms/{termId}`
- `/api/console/releases/*`
- `/api/console/validation-cases/*`

### 阶段 2：导入批次模型

目标：

- 把导入工作台所需的 DB 模型补齐

产出：

- `import_jobs`
- `import_job_files`
- `import_job_rows`
- `import_job_results`
- 模板配置与模板下载接口

### 阶段 3：导入确认写入流程

目标：

- 打通“上传 -> 预览 -> 确认 -> 入词典库”

产出：

- 上传接口
- 清洗预览
- 确认导入
- 错误报表

### 阶段 4：审核/发布/验证联动

目标：

- 完成导入后到发布验证的完整闭环

产出：

- 导入任务到审核任务联动
- 词条详情页展示来源与验证关联
- 发布中心展示变更来源与验证结果

## 13. 当前不建议做的事情

- 直接推翻当前 `terms/aliases` 主数据模型
- 直接用 `validation feed` 取代导入中心
- 让新后台前端直接拼现有所有零散接口
- 一开始就把模板管理做成复杂后台配置系统
- 在没有导入批次对象前直接做多类型文件上传

## 14. 下一步建议

在本文件确认后，建议进入实现前的最后一步准备：

1. 明确新后台前端技术选型和目录结构
2. 明确 `/api/console/*` 的首批最小接口集
3. 明确导入模板第一批支持的文件格式
4. 明确导入批次 4 张表的最终字段定义
