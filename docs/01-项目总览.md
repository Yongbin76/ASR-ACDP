# 项目总览

## 1. 背景

在中文 ASR 实时纠错链路中，热词规则层位于排序缓冲之后、LLM 纠错之前，承担“低延迟、高精度、可解释”的第一层纠错职责。

现有 Demo 已证明 AC 自动机可用于：

- 多模式字符串精确匹配
- 冲突词的最长匹配优先
- 按词典映射完成替换

但 Demo 仍有明显缺口：

- 词典来源仍是 Excel，缺少平台化管理
- 仅支持单机脚本构建，不支持生产级热更新、灰度、回滚
- 对中文缺少拼音与近音增强能力
- 缺少词条元数据、审计、统计、治理闭环

因此需要建设一个独立的“ACDP”项目，统一承接词典治理、发布和运行时接入。

## 2. 建设目标

### 2.1 业务目标

- 提升已收录热词变体的命中率
- 降低中文 ASR 在地名、政务专有词、机构名、专有名词上的误识别
- 为 LLM 层提供更干净、更稳定的前置纠错结果和候选标注
- 建立词典从发现到上线再到评测回流的闭环

### 2.2 工程目标

- 支撑 CG1 热词规则层在线处理延迟目标
- 实现不停服热更新
- 支持版本管理、灰度发布与快速回滚
- 支持词条级审计与效果追踪
- 支持中文拼音增强能力

### 2.3 产品目标

- 让运营、算法、开发在同一平台完成词典协作
- 让“新增词条、修复误替换、下线风险词条”成为标准化动作
- 让词典维护从人工表格变成可校验、可发布、可追溯的流程

## 3. 适用范围

### 3.1 纳入 AC 平台的词典类型

- 上海地名/路名
- 政务专有词
- 专有名词
- 百家姓专项中可结构化管理的拆字词典、姓氏纠正规则词典

### 3.2 不直接纳入 AC 主匹配的场景

- 整体姓名全文盲匹配
- 强依赖上下文语义的复杂纠错
- 数字和英文字母的主流程格式化
- 长句语义改写

## 4. 核心原则

- 高精度优先：热词层首先要减少误伤，而不是盲目追求召回
- 延迟受控：运行时以毫秒级处理为目标，不引入重模型依赖
- 词典不可变快照：运行时只加载已发布快照，不直接读运营底表
- 发布可回滚：任何词典版本上线都必须可灰度、可回滚
- 中文优先：平台设计以中文 ASR 为主，不采用英文搜索引擎式的默认假设
- 拼音增强内建：拼音不是附加能力，而是中文词典平台的一级能力

## 5. 目标用户

- AI 运营：新增词条、修正词条、热点词维护
- 算法/规则工程师：配置上下文规则、风险等级、拼音增强策略
- 平台工程师：发布、回滚、灰度、监控
- 质检与数据治理团队：回收误判样本、形成迭代闭环

## 6. 成功标准

### 6.1 功能层

- 支持词典录入、审核、发布、回滚、灰度、停用
- 支持词条变体、类别、优先级、上下文规则、拼音规则配置
- 支持运行时热更新
- 支持命中、替换、候选、误伤、漏召回的统计

### 6.2 运行层

- 词典加载不阻塞在线流量
- 新版本构建失败不影响当前运行版本
- 单版本异常可在分钟级回滚

### 6.3 治理层

- 对新增词条有审核流程
- 对线上替换结果有版本追踪
- 对高风险词条可快速停用
- 对中文拼音变体有专门的治理入口

## 7. 项目产物

本项目最终应形成两类产物：

### 7.1 控制面产物

- 管理后台
- 词典审核与发布 API
- 词条治理与统计报表
- 词典版本与审计记录

### 7.2 数据面产物

- 运行时词典快照
- AC 自动机编译产物
- 拼音索引和拼音增强配置
- 规则配置快照

## 8. 首阶段落地范围

首阶段优先做“能支撑生产试点”的最小闭环：

- 三类词典管理
- 词条审核与版本化
- 快照编译
- 热更新与回滚
- 命中与替换统计
- 拼音增强一期

拼音增强一期以“编译期变体扩展 + 运行期拼音候选召回”为主，不直接做大范围高风险自动替换。
