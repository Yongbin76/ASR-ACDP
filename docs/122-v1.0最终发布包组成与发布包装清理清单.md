# v1.0 最终发布包组成与发布包装清理清单

## 1. 文档目的

本文档用于把以下问题一次性说清楚：

- `v1.0` 的“最终发布包”到底包含什么
- 发布前需要做哪些打包与清理动作
- 开发过程文档、测试数据、测试报告何时清理
- 哪些内容会影响正式发布交付物

## 2. 最终发布包的 3 类交付物

`v1.0` 最终发布时，必须明确区分以下 3 类交付物：

### 2.1 运行制品包

这是给 runtime 真正消费的 release 制品。

当前代码里的核心产物包括：

- `snapshot.json`
- `manifest.json`
- `package.tar.gz`
- 对应 `artifact metadata`

当前代码依据：

- [build-snapshot.js](/Codex/ACDP/prototype/src/cli/build-snapshot.js)
- [server.js](/Codex/ACDP/prototype/src/server.js)
- [snapshot-builder.js](/Codex/ACDP/prototype/src/lib/snapshot-builder.js)
- [artifact-store.js](/Codex/ACDP/prototype/src/lib/artifact-store.js)

### 2.2 部署交付包

这是给运维和部署环境消费的交付物。

至少包括：

- admin 镜像
- runtime 镜像
- K8S YAML / 部署清单
- 环境变量模板
- Secret 与 artifact store 配置说明

### 2.3 发布证据包

这是给 go / no-go、发布审批、留档复盘使用的证据，不属于运行制品。

至少包括：

- 真实部署验证证据
- 真实并发与吞吐验证证据
- console 回归结果
- release 导出结果
- 发布说明
- 回滚手册
- `summary.json`
- `go-no-go.md`

当前依据：

- [85-R7证据包目录与go-no-go模板](./85-R7证据包目录与go-no-go模板.md)
- [86-v1.0发布手册](./86-v1.0发布手册.md)
- [87-v1.0回滚手册](./87-v1.0回滚手册.md)
- [89-v1.0-go-no-go清单模板](./89-v1.0-go-no-go清单模板.md)

## 3. 生成最终发布包需要做的事情

建议按以下顺序执行：

### 3.1 冻结候选 release

前提：

1. release 已 `built`
2. 发布审核已满足
3. 版本校验通过
4. `JOB-006` 完成
5. `JOB-009` 完成
6. `JOB-031 ~ JOB-033` 已完成
7. `JOB-090` 本地与真实环境阶段已收口

### 3.2 生成运行制品包

目标：

- 生成 release 目录
- 写出 `snapshot.json`
- 写出 `manifest.json`
- 生成 `package.tar.gz`
- 生成 artifact metadata
- 同步到 artifact store

### 3.3 生成部署交付包

目标：

- 固定 admin 镜像
- 固定 runtime 镜像
- 固定 K8S YAML
- 固定配置与 Secret 注入说明
- 固定部署命令与校验命令

目录模板依据：

- [124-v1.0镜像标签与部署交付包目录模板](./124-v1.0镜像标签与部署交付包目录模板.md)

### 3.4 生成发布说明包

目标：

- 发布说明
- 部署手册
- 回滚手册
- 帮助手册快照
- 接口手册快照

文档清单依据：

- [125-v1.0发布说明包与文档快照清单](./125-v1.0发布说明包与文档快照清单.md)

当前仓库已提供本地可执行生成脚本：

- `npm run prepare:release-bundle -- --version <v> --release-id <releaseId> --image-registry <registry> --output-dir <dir>`

### 3.5 生成发布证据包

目标：

- `deployment/`
- `concurrency/`
- `console/`
- `release/`
- `docs/`
- `summary.json`
- `go-no-go.md`

### 3.6 记录封板信息

必须记录：

- 版本号
- 对应 `releaseId`
- 发布时间
- 发布负责人
- 回滚负责人
- 证据包目录

## 4. 当前过程性内容会不会影响最终发布包

结论：

- **不会直接污染 runtime release 制品**
- **会影响 admin/runtime 镜像和最终交付边界**

### 4.1 不会直接污染的部分

以下内容通常不会直接进入 runtime release 制品：

- 开发过程讨论文档
- 测试批次报告
- `prototype/workspace/*` 下的运行产物
- `host_verification/*` 下的验证产物

原因：

- 当前 runtime release 构建链只围绕 release snapshot 与 artifact metadata 工作
- `.dockerignore` 已排除：
  - `prototype/workspace`
  - `prototype/workspace-*`

### 4.2 会影响的部分

当前已改成：

- `Dockerfile` / `Dockerfile.admin`
  - 保留：
    - `package.json`
    - `prototype/`
    - `console/`
    - `data_sources/`
    - `docs/help_manuals/`
    - `docs/25-console宿主环境联调与smoke执行说明.md`
    - `docs/26-console内部试用说明.md`
- `Dockerfile.runtime`
  - 仅保留：
    - `package.json`
    - `prototype/`

当前依据：

- [Dockerfile](/Codex/ACDP/Dockerfile)
- [Dockerfile.admin](/Codex/ACDP/Dockerfile.admin)
- [Dockerfile.runtime](/Codex/ACDP/Dockerfile.runtime)
- [/.dockerignore](/Codex/ACDP/.dockerignore)

同时 `.dockerignore` 已收紧为：

- 忽略：
  - `project_management`
  - `prototype/tests`
  - `SESSION_HANDOFF.md`
  - `NEXT_STEPS.md`
  - `STAGE_SUMMARY_*.md`
  - `docs/**`
- 仅放行：
  - `docs/help_manuals/**`
  - `docs/25-console宿主环境联调与smoke执行说明.md`
  - `docs/26-console内部试用说明.md`

这意味着当前主要影响已被收敛为：

1. 正式镜像只保留运行和在线帮助所需的最小文档集合
2. 过程性交接文档不再进入正式镜像
3. 测试代码不再进入正式镜像
4. 镜像边界已经从“整仓资料打包”收紧到“运行所需最小集”
5. 可以先通过 `prepare:release-bundle` 生成部署交付包和发布说明包模板目录，再由运维补齐真实镜像地址与环境变量

## 5. 清理应该在什么时候做

### 5.1 `/console` 产品面封板后

也就是 `JOB-031 ~ JOB-033` 完成后，优先清理：

- 旧导航命名
- 旧页面标题
- 旧 breadcrumb
- 旧帮助入口
- 旧过渡态文案
- 页面中的英文品牌与试用态提示

### 5.2 `JOB-030` 前

在发布前，必须做一次交付物清理：

- 明确哪些文档要进入正式镜像
- 明确哪些文档只保留在仓库，不进入镜像
- 明确哪些测试脚本保留在仓库，但不进入部署交付
- 明确最终证据包目录

### 5.3 发布后

发布后不立即删除仓库中的过程资料，而是：

- 保留开发文档用于追溯
- 保留测试报告用于复盘
- 只把最终证据包作为正式发布留档

## 6. 建议的发布包装清理动作

建议至少做以下 6 项：

1. 收紧 Docker 打包边界
2. 区分运行制品、部署交付包、发布证据包
3. 将 `SESSION_HANDOFF.md`、`NEXT_STEPS.md` 排除出正式镜像
4. 评估 `docs/` 是否整包进入镜像，或仅保留帮助文档子集
5. 评估 `prototype/tests`、开发脚本是否应进入正式镜像
6. 增加发布前“打包边界复核”检查
