# v1.0 完整重构讨论清单

> 历史说明
>
> 本文档保留完整重构讨论过程，其中包含大量旧口径问题陈述。
>
> 当前直接编码时，不应把本文中的旧称当成现行页面或接口名称。

## 1. 文档目的

本文档用于在进入正式重构前，把必须先讨论清楚的问题列成清单。

目标不是立即编码，而是避免边讨论边改代码、越改越乱。

## 2. 必须先拍板的 11 个问题

### 2.1 `v1.0` 的硬门槛是什么

需要确认：

- `JOB-006` 是否为 `v1.0` 硬门槛
- `JOB-009` 是否为 `v1.0` 硬门槛
- `JOB-007` 是否延期到 `v1.1`

### 2.2 主数据是否完全与 release 解耦

建议答案：

- 是

需要明确：

- 词条中心和导入中心都只改主数据
- release 永远是快照，不直接被页面“编辑”

### 2.3 审核中心是否只保留内容审核

建议答案：

- 是

需要明确：

- 词条审核
- 拼音审核

而：

- release publish review 归口发布中心

### 2.4 发布中心是否彻底改成生命周期视图

建议答案：

- 是

需要明确：

- `built`
- `canary`
- `published`
- `发布后风险`

以及：

- `gate` 只作为升级前置条件

### 2.5 运行节点中心的主对象是什么

建议答案：

- 以“备案节点”为主对象

然后叠加：

- 注册状态
- 在线状态
- 版本收敛状态
- 异常状态

### 2.6 用户 / 角色 / 权限 / 治理策略怎么持久化

建议答案：

- 当前 `v1.0` 阶段以统一配置文件持久化
- 不强制先做数据库 IAM 平台

### 2.7 中文化和术语统一的边界是什么

建议答案：

- 用户主文案统一中文
- 技术码只在二级信息或技术详情中暴露

### 2.8 在线帮助手册做到什么程度

建议答案：

- 作为 `v1.0` 硬门槛

需要明确：

- 页面级帮助
- 流程级帮助
- 运维手册
- 排障手册
- runtime 对外接口帮助
- admin / console 对外接口帮助

### 2.9 runtime / admin 对外接口文档边界是什么

建议答案：

- 必须单独成体系，不混在页面文档里

需要明确：

- runtime 正式接口
- runtime 候选接口
- WebSocket 接口
- admin / console 管理接口的定位和适用对象

### 2.10 前端是原地修，还是新壳重建

建议答案：

- 先收口后端 contract
- 再决定是否重建前端页面壳层

如果继续原地 patch，风险很高。

### 2.11 一级、二级目录名称是否要再收口，并改成配置驱动

建议答案：

- 是

需要明确：

- 一级目录最终名称
- 二级目录最终名称
- 是否减少“中心”类泛化命名
- 导航是否改由统一配置文件驱动

## 3. 建议的工作包

### W01 目标与范围包

输出：

- `v1.0` 目标
- `v1.0` 非目标项
- 时间计划
- go / no-go 标准

### W02 领域边界包

输出：

- 主数据域
- 内容审核域
- 版本发布域
- 运行治理域
- 系统管理域

### W03 状态机包

输出：

- 主数据状态机
- 审核状态机
- release 状态机
- runtime 节点状态机

### W04 后端统一读模型包

输出：

### W05 前端 IA 包

输出：

- 一级导航
- 二级目录
- 页面职责
- 页面之间跳转关系

### W06 实现与验证包

输出：

- 编码任务拆解
- 回归策略
- 真实环境验证策略

## 4. 建议讨论顺序

建议按以下顺序讨论：

1. `v1.0` 目标与硬门槛
2. 5 个域边界
3. 4 套状态机
4. 运行治理域主对象
5. 发布生命周期
6. 配置驱动与系统管理
7. 前端重建策略
8. 在线帮助手册与接口文档体系
9. 前端中文化边界
10. 导航命名与配置化
11. 里程碑和 joblist

## 5. 当前结论

在这些问题没有拍板前，不建议直接进入下一轮大规模编码。

文档应该先作为讨论基线，再进入重构实现。
