# 运行治理域详细设计

- 文档状态：active
- 适用版本：v1.0
- 文档类型：design
- 所属工作区：2026-04-17-v1.0正式文档工作区
- 最后更新时间：2026-04-18
- 责任对象：节点备案、运行节点、runtime-control、运行验证

## 1. 设计目标

运行治理域负责：

1. 管理哪些 runtime 可以接入。
2. 观察哪些 runtime 正在运行。
3. 下发目标版本并观察收敛。
4. 提供运行验证入口。

## 2. 主对象

1. `runtime_node_registry`
2. `runtime_nodes`
3. `runtime_control_state`
4. runtime stats

## 3. 备案与接入链

### 3.1 节点备案

1. 在后台创建 registry 记录
2. 生成或轮换 `registrationSecret`
3. 节点启动时使用该密钥接入

### 3.2 节点注册

1. runtime 调 `POST /api/runtime-nodes/register`
2. admin 校验 runtime token
3. admin 校验 registry
4. 写 `runtime_nodes`

### 3.3 心跳

1. runtime 周期性 heartbeat
2. admin 更新节点状态和时间戳

## 4. 目标版本下发链

1. admin 设置 `desiredRelease`
2. 写 `runtime_control_state`
3. 更新节点 `desired_version`
4. runtime 拉 `/api/runtime-control/me`
5. 下载制品
6. 安装并切换
7. 回传 apply result

## 5. 实时与历史状态分层

当前设计要求：

1. 备案状态和运行状态分开
2. 实时在线状态和历史错误分开
3. 节点对齐状态和应用失败状态分开

## 6. 运行验证

当前运行验证分两层：

1. `runtime-demo`
   - 首页轻量模拟
2. `runtime-verify`
   - 独立验证页
   - 当前只支持 `cluster_current`

## 7. 关键边界

1. 未备案节点不应成为正式主对象。
2. `runtime_control_state` 当前是全局单控制状态。
3. 运行治理负责“让版本生效”和“观察节点状态”，不负责词条录入与发布审批。

## 9. 修改风险

1. 改 registry 校验会影响 runtime 接入。
2. 改 control view 结构会影响 runtime 下载链。
3. 改节点状态表达会影响控制台页面和异常判断。
