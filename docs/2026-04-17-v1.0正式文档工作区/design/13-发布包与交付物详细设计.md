# 发布包与交付物详细设计

- 文档状态：active
- 适用版本：v1.0
- 文档类型：design
- 所属工作区：2026-04-17-v1.0正式文档工作区
- 最后更新时间：2026-04-18
- 责任对象：镜像、交付目录、发布说明包

## 1. 设计目标

把“运行镜像”“部署交付包”“发布说明包”边界彻底分清。

## 2. 当前交付物分类

### 2.1 运行镜像

1. `Dockerfile`
2. `Dockerfile.admin`
3. `Dockerfile.runtime`

### 2.2 部署交付包

模板目录：

1. `release_bundle_templates/deployment_bundle/`

### 2.3 发布说明包

模板目录：

1. `release_bundle_templates/release_notes_bundle/`

## 3. 真实环境最小交付口径

当前最务实的真实环境交付建议如下：

1. 不同步整个项目目录
2. 只交付 `acdp-admin` / `acdp-runtime` 镜像
3. 同时交付 `release bundle`
4. `acdp-prototype` 仅保留给本地一体联调或演示

推荐拓扑：

1. `admin` 单实例
2. `runtime` 多实例
3. 默认使用 `admin_http_signed` 下发快照

部署顺序：

1. 先部署 admin
2. 在 admin 中完成节点备案
3. 再部署 runtime
4. 最后验证 register / heartbeat / desiredVersion / apply-result

## 4. 打包边界

当前保留进正式镜像的文档仅限：

1. `docs/help_manuals/`
2. `docs/25-console宿主环境联调与smoke执行说明.md`
3. `docs/26-console内部试用说明.md`

不再整包复制 `docs/`。

## 5. 关键代码入口

1. [`prepare-release-bundle.js`](/Codex/ACDP/prototype/src/cli/prepare-release-bundle.js)
2. `Dockerfile*`
3. `.dockerignore`

## 6. 修改风险

1. 改镜像打包边界会影响交付物纯度。
2. 改 release bundle 模板会影响交付目录结构。
