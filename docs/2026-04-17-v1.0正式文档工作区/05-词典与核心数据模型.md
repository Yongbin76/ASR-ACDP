# ACDP 词典与核心数据模型

- 文档状态：active
- 适用版本：v1.0
- 文档类型：baseline
- 所属工作区：2026-04-17-v1.0正式文档工作区
- 最后更新时间：2026-04-17
- 责任对象：核心实体、关系、生命周期、状态

## 1. 设计目标

本模型解决 5 个核心对象域：

1. 词典建设。
2. 审核与准入。
3. 批量导入。
4. 版本发布。
5. runtime 节点治理。

## 2. 实体总览

### 2.1 词典建设实体

4. `alias_source`

5. `term_rule`

6. `pinyin_profile`

### 2.2 审核实体

1. `review_task`

   - 拼音候选审核任务。

## 3. 词典记录模型

### 3.1 词典记录的主键与业务主语

词典记录的业务主语是 `term_id`。

核心字段包括：

1. `category_code`
2. `canonical_text`
3. `status`
4. `priority`
5. `risk_level`
6. `replace_mode`
7. `base_confidence`
8. `source_type`
9. `pinyin_runtime_mode`

### 3.2 别名模型

别名是附属于 `term_id` 的从属集合。

一个标准词可以有多个别名，但 `(term_id, alias_text)` 必须唯一。

### 3.3 来源追踪模型

标准词和别名都可以追踪来源，来源字段包括：

1. 来源类型。
2. `import_job_id`。
3. 源文件名。
4. 源行号。
5. 源引用说明。

### 3.4 规则模型

每个词条可绑定一套规则，控制：

1. 是否仅作为 candidate。
2. 最小与最大文本长度。
3. 边界策略。
4. 左右文允许与阻断。
5. 正则允许与阻断。

### 3.5 拼音画像模型

每个词条可绑定一套拼音画像，核心包括：

1. 主读音。
2. 首字母串。
3. 音节数组。
4. runtime 模式。
5. 多音字模式。
6. 自定义主读音。
7. 额外候选读音。
8. 备注。

## 4. 词典生命周期

### 4.1 词典状态

当前词典状态主要为：

1. `draft`
2. `pending_review`
3. `approved`
4. `disabled`

### 4.2 状态迁移

1. 手工新建或导入确认后的新词，默认进入 `draft`。
2. 提交审核后，进入 `pending_review`。
3. 审核通过后，进入 `approved`。
4. 审核驳回后，回到 `draft`。
5. 停用后，进入 `disabled`。

## 5. 审核模型

### 5.1 审核任务分类

当前审核任务类型包括：

1. `term_review`
2. `pinyin_candidate_review`
3. `release_publish_review`

### 5.2 审核目标类型

当前目标类型包括：

1. `term`
2. `pinyin_candidate`
3. `release`

### 5.3 审核状态

审核任务状态包括：

1. `pending`
2. `approved`
3. `rejected`

### 5.4 审核副作用

1. `term_review` 通过：
   - 词条状态更新为 `approved`。
2. `term_review` 驳回：
   - 词条状态回退为 `draft`。
3. `pinyin_candidate_review` 通过：
   - 将候选读音写入词条拼音画像。
4. `release_publish_review` 通过：
   - 只改变审核结果，不直接改变 release 状态。

## 6. 导入模型

### 6.1 导入批次主语

无论是 CSV 批量导入还是后续扩展的导入变体，统一以 `job_id` 作为导入批次主语。

### 6.2 导入文件

一个导入批次可挂多个文件，通常包括：

1. 主数据文件。
2. 错误词补录文件。
3. 兼容模式文件。

### 6.3 导入行

每一行导入记录具备：

1. 原始载荷。
2. 归一化载荷。
3. 问题列表。
4. 目标词主键。
5. 行状态。
6. 行决策。

### 6.4 导入批次状态

当前导入批次状态主要为：

1. `uploaded`
2. `parsed`
3. `preview_ready`
4. `imported`
5. `cancelled`

### 6.5 导入行状态

当前导入行状态主要为：

1. `ready`
2. `warning`
3. `error`
3. `imported`

### 6.6 导入行决策

当前导入行常见决策包括：

1. `pending`
2. `accept`
3. `merge_existing`

## 7. 验证样本模型

验证样本以 `case_id` 为主键，核心字段包括：

1. `description`
2. `sample_text`
3. `expected_canonicals_json`
4. `enabled`
5. `source_type`
6. `notes`

验证样本不属于词典记录本身，但会参与：

1. release 门禁。
2. 发布后风险观察。
3. 词条与样本的关联分析。

## 8. 发布模型

### 8.1 release 主语

release 以 `release_id` 为主键，代表一份可构建、可审核、可发布的词典版本。

### 8.2 release 关联词条

`release_terms` 记录某个 release 包含了哪些词条。

### 8.3 灰度策略

`gray_policies` 用于记录某个 release 的灰度配置，主要字段包括：

1. `scope_type`
2. `percentage`
3. `enabled`

### 8.4 发布状态

当前 release 状态主要为：

1. `built`
2. `canary`
3. `published`

## 9. runtime 治理模型

### 9.1 备案节点与运行节点分离

系统将 runtime 节点拆成两个对象：

1. `runtime_node_registry`
   - 管备案台账。
2. `runtime_node`
   - 管实际接入后的实时状态。

这两个对象必须分开，避免把未备案节点直接视为正式运行对象。

### 9.2 runtime 节点核心字段

运行节点核心关注：

1. `node_id`
2. `node_name`
3. `env`
4. `address`
5. `current_version`
6. `desired_version`
7. `status`
8. `last_heartbeat_at`
9. `last_apply_status`
10. `last_error`

### 9.3 控制状态模型

`runtime_control_state` 是 admin 对所有 runtime 的目标控制状态快照，核心包括：

1. `release_id`
2. `desired_version`
3. `artifact_metadata_json`
4. `config_version`
5. `issued_at`

## 10. 统计模型

统计模型分为两层：

1. 节点级统计。
2. 全局统计。

节点级统计包括：

1. 小时请求量。
2. 小时词条命中量。
3. 节点峰值并发。
4. 上传批次留痕。

全局统计包括：

1. 小时聚合请求量。
2. 小时聚合词条命中量。
3. 全局峰值并发。

## 11. 审计模型

`audit_logs` 用于记录关键动作的前后快照，主要包括：

1. 操作人。
2. 操作类型。
3. 目标类型。
4. 目标 ID。
5. 变更前快照。
6. 变更后快照。
7. 备注。

## 12. 当前数据模型的关键约束

1. 词条是词典建设的主对象，别名、规则、拼音画像均从属词条。
2. 审核主语是 `task_id`，不是页面本身。
3. 批量导入与手工创建的最终审核原子对象相同，都是 `review_task`。
4. `import_job_id` 用于标识“这条词条来自哪个导入批次”，不是另一套审核主语。
5. release 只管理“版本内容与发布生命周期”，不再承担主数据录入职责。
6. runtime 治理只以备案节点和运行节点为主对象，不以历史事件为主对象。
