## 5. 页面前后对照

改造前：

1. `确认导入`
2. 有 `error` 行时整批按钮置灰
3. 错误报表语义只停留在预览阶段
4. 导入结果卡片不足以区分：
   - 成功导入
   - warning 导入
   - 跳过阻断

改造后：

1. `导入可通过记录`
2. 只要 `importableRows > 0` 即允许导入
3. 阻断报表在导入完成后仍可下载
4. 导入结果明确区分：
   - `成功导入`
   - `warning 导入`
   - `跳过阻断`

## 6. 中文化要求

本页面最终呈现禁止直接暴露英文按钮和英文结果文案。

禁止示例：

1. `Confirm Import`
2. `Error Report`
3. `Skipped`
4. `Pending`

允许保留英文的范围：

1. 接口内部 JSON 枚举值
2. 测试断言中的原始字段值
3. 开发日志中的内部标识

## 7. 导入成功提示

导入成功后的提示必须包含：

1. title：`可通过记录已导入`
2. description：`成功导入 X 行，warning 导入 Y 行，跳过阻断 Z 行。`
3. actions：
   - `查看本批词条审核`
   - `下载阻断报表`
   - `返回批量导入`

## 8. 结果区块要求

导入完成后：

1. 被跳过的阻断行继续保留在“预览行结果”区块
2. 支持通过：
   - `行状态 = 错误行`
   - `处理决策 = 阻断跳过`
   定位这些记录
3. “影响词条”和“生成的审核任务”只反映实际入库的 `ready / warning` 行

## 9. JOB-106 批量处理方案

### 9.1 逐行系统建议

批量导入预览时，每一行都必须直接生成系统建议，不要求用户逐条自行判断。

建议动作固定为：

1. `按替换词条导入`
2. `按推荐候选导入`
3. `并入已有词条`
4. `补录到已有词条`
5. `阻断跳过`

### 9.2 主批量动作

批次详情页必须提供统一主动作：

1. `按系统建议处理`

执行结果为：

1. `save_replace` 行按替换词条入库
2. `save_candidate` 行按推荐候选入库
3. `merge_existing` 行更新已有词条
4. `append_alias_to_existing` 行补录到已有词条
5. `skip_blocked` 行跳过不入库

前提条件：

1. `merge_existing` 与 `append_alias_to_existing` 只有在存在唯一目标词条时才允许自动执行
2. 若存在多个目标候选，则不得自动并入或自动补录
3. 此类记录只能降级为：
   - `按推荐候选导入`
   - `阻断跳过`

### 9.3 页面冻结要求

1. 预览结果表格必须新增：
   - `系统建议`
   - `原因摘要`
   - `审核提示`
   - `目标词条`
2. 导入完成后的结果卡片应能区分：
   - `替换导入`
   - `候选导入`
   - `并入已有`
   - `补录已有`
   - `阻断跳过`
3. 页面不应要求用户逐条选择 `replace / candidate / merge / skip`
4. 预览结果区必须增加“建议动作”单独筛选项
   - `全部建议动作`
   - `建议替换导入`
   - `建议候选导入`
   - `建议并入已有`
   - `建议补录已有`
   - `建议阻断跳过`

### 9.4 结果字段冻结

导入结果汇总至少需要区分：

1. `替换导入`
2. `候选导入`
3. `并入已有`
4. `补录已有`
5. `阻断跳过`

### 9.5 reviewHints 呈现口径

1. 导入预览页的“审核提示”即 `reviewHints`
2. 因导入预览本身存在行级结果落盘能力，可保留当次建议摘要
3. 该摘要只服务当前批次预览和导入处理，不作为词条主数据永久字段
