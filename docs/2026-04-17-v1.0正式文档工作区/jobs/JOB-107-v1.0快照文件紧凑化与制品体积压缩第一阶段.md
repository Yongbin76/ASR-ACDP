# JOB-107 v1.0 快照文件紧凑化与制品体积压缩第一阶段

- 文档状态：active
- 实施状态：done
- 适用版本：v1.0
- 文档类型：job
- 所属工作区：2026-04-17-v1.0正式文档工作区
- 最后更新时间：2026-04-22
- 责任对象：snapshot 生成链、release 制品体积、runtime 下发效率

## 1. 目标

在**不改变 runtime 快照结构、不改变接口契约、不改变运行逻辑**的前提下，先把 `snapshot.json` 从“带缩进的 pretty JSON”改成“紧凑 JSON”写出，优先获得低风险、立刻可见的制品体积下降。

当前事实：

1. [latest/snapshot.json](/Codex/ACDP/prototype/workspace/releases/latest/snapshot.json) 当前约 `52.5MB`
2. 同一份数据若改成紧凑 JSON，可降到约 `35.35MB`
3. 预计单份快照可减少约 `16.37MB`

## 2. 范围

本 job 只处理第一档低风险方案：

1. 把 `snapshot.json` 输出从：
   - `JSON.stringify(snapshot, null, 2)`
   改成：
   - `JSON.stringify(snapshot)`
2. `manifest.json` 同步改成紧凑输出

## 3. 生效边界

本 job 的变更只影响**后续新生成的制品**：

1. 代码改完后，只有后续新 build / 新 release 生成的：
   - `releases/<releaseId>/snapshot.json`
   - `releases/<releaseId>/manifest.json`
   - `releases/latest/snapshot.json`
   - `releases/latest/manifest.json`
   会变成紧凑 JSON
2. 已存在的历史 release 制品不会自动重写
3. 若要让当前线上使用的快照体积变小，必须重新 build/release 后生效

## 4. 不在本 job 范围内

明确不处理：

1. 不改 `snapshot` 数据结构
2. 不改 `literalPatterns` 字段模型
3. 不改 `pinyinExactIndex` 字段模型
4. 不做 term meta 去重重构
5. 不改 runtime 匹配逻辑
6. 不改对外接口返回

结构去重和索引压缩若后续要做，另开新 job。

## 6. 预期收益

直接收益：

1. release 目录中的 `snapshot.json` 体积下降
2. `releases/latest/snapshot.json` 体积下降
3. runtime 安装时下载体积下降
4. runtime 启动/切换时的文件读取与 `JSON.parse` 开销下降

说明：

1. 这是**体积优化**
2. 不是结构重构
3. 对 steady-state 单请求性能提升有限，但对：
   - 下发
   - 安装
   - 启动
   - 切版本
   的改善更直接
