# JOB-104 v1.0 发布流水线自动化与交付产物标准化

- 文档状态：active
- 适用版本：v1.0
- 文档类型：job
- 所属工作区：2026-04-17-v1.0正式文档工作区
- 最后更新时间：2026-04-21
- 责任对象：发布流水线、镜像推送、bundle 生成、构建元数据

## 1. 目标

在 `JOB-100` 已完成“发布包边界收口”的基础上，继续把发布动作标准化为可重复执行的流水线。

本 job 的目标是：

1. 自动构建 `acdp-admin` / `acdp-runtime`
2. 自动推送镜像到目标 registry
3. 自动生成 `release bundle`
4. 自动落盘发布元数据、镜像 tag 清单和 checksum
5. 固定发布输入输出、目录命名和失败退出口径

## 2. 范围

本 job 处理：

1. 镜像命名与 tag 规则
2. registry 凭据注入口径
3. 发布脚本输入输出
4. bundle 目录生成
5. 构建元数据与 checksum 文件
6. 发布前校验和失败退出说明

## 3. 不在本 job 范围内

1. 不改业务代码
2. 不改运行时接口合同
3. 不做真实集群一键发布
4. 不把灰度策略实现和节点治理混入本批
5. 不在本批强制接入外部 SBOM / 漏扫平台，但允许预留挂点

## 4. 依赖

需要明确或补齐：

1. 镜像仓库地址
2. 镜像推送凭据
3. 版本号与 `releaseId` 生成规则
4. 构建时间、commit id 的写入来源
5. 可选的 checksum / signing / scan 输出挂点

## 5. 交付标准

至少达到以下标准：

1. 其他 Codex 不翻聊天记录也能直接执行发布流水线
2. 输入参数、输出目录、日志路径、失败码全部明确
3. admin/runtime 双镜像和 release bundle 可由同一批次自动生成
4. 发布元数据文件格式稳定，可供后续运维和审计复用
5. 文档、真源、模板和脚本同步

## 7. 建议实施顺序

推荐顺序：

1. 先冻结输入输出
2. 再实现脚本入口
3. 再补 bundle 与 metadata
4. 最后补测试和文档

理由：

1. 如果先写脚本，再补规则，容易反复返工
2. 如果先冻结输入输出，后续脚本和模板能一次性对齐

## 8. 风险与注意事项

1. 不要把 `JOB-104` 写成“真实集群一键发布平台”
2. 不要把 `JOB-104` 和 `JOB-030` 的真实环境验证混成一单
3. 不要重新扩大 `JOB-100` 范围
4. 发布流水线默认只覆盖最小真实交付物：`acdp-admin` / `acdp-runtime` + `release bundle`
