# JOB-103 v1.0 正式文档工作区详细设计化与 Codex 开发依据补齐

- 文档状态：active
- 适用版本：v1.0
- 文档类型：job
- 所属工作区：2026-04-17-v1.0正式文档工作区
- 最后更新时间：2026-04-17
- 责任对象：详细设计层、数据字典层、接口规范层、页面设计层、Codex 开发依据层

## 1. 目标

把当前工作区从“正式基线文档集”升级为：

1. ACDP 的详细设计说明书。
2. 其他 Codex 可直接编码的开发依据文档集。

## 2. 范围

本 job 不局限于已有 `00~13` 主册，而是在当前工作区内新增 5 个详细层目录：

1. `design/`
2. `data_dictionary/`
3. `api_specs/`
4. `pages/`
5. `codex_dev/`

## 3. 不在本 job 范围内

1. 不直接改业务代码。
2. 不直接改数据库结构。
3. 不直接改接口合同。
4. 不重开旧归档文档作为现行依据。

## 4. 本 job 的交付标准

至少达到以下标准：

1. 数据库字典不只写字段名，还写类型、默认值、可空性、约束、索引、案例和改造影响。
2. 接口文档不只写路由，还写调用方、参数、响应、错误码、副作用、样例和测试入口。
3. 页面文档不只写页面名，还写区块结构、按钮作用、数据来源、权限和状态规则。
4. Codex 开发依据文档能让其他 Codex 直接按阅读顺序开工。
5. 所有新增详细文档都收录进工作区索引，并接入单一真源。

## 6. 建议实施顺序

推荐顺序：

1. `data_dictionary/`
2. `api_specs/`
3. `design/`
4. `codex_dev/`
5. `pages/`

理由：

1. 数据字典和接口规范是后续编码的最底层依据。
2. 业务详细设计依赖数据和接口口径。
3. Codex 开发依据要建立在前面三层已经稳定的基础上。
4. 页面设计可最后收细，因为它同时依赖接口和状态机口径。

## 7. 风险与注意事项

1. 不能只扩写篇幅，必须提高开发可执行性。
2. 不能重新引用目录外旧文档。
3. 不能只写“应该如此”，必须以当前代码事实为准。
4. 对于当前代码里仍存在的模糊口径，要明确写“当前如此、后续如要改需同步哪里”。
