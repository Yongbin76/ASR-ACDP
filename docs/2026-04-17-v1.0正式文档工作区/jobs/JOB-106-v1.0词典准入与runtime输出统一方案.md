# JOB-106 v1.0 词典准入与 runtime 输出统一方案

- 文档状态：active
- 实施状态：done
- 适用版本：v1.0
- 文档类型：job
- 所属工作区：2026-04-17-v1.0正式文档工作区
- 最后更新时间：2026-04-22
- 责任对象：词典准入、词典录入页、批量导入页、runtime `correct / correct_cand`

## 1. 目标

把词典治理与 runtime 执行统一成一套逻辑，不再让“录入规则”和“运行时真实命中逻辑”各走各的。

统一目标为：

1. 准入层只保留 `blocked / ready`
2. `ready` 再按运行方式区分：
   - `replace`
   - `candidate`
3. 系统在录入时直接给出处理建议
4. 用户按建议批量处理，不再逐条自己判断

## 2. 当前实现事实

### 2.1 runtime 事实

1. literal 通道当前只以 `alias` 参与字面匹配
2. pinyin 通道当前会把 `canonical + alias` 一起写入拼音索引
3. `/api/runtime/correct` 和 `/ws/runtime/correct` 提供单一替换输出
4. `/api/runtime/correct_cand` 和 `/ws/runtime/correct_cand` 提供候选整句输出

### 2.2 治理事实

1. 当前准入已覆盖：字段合法性、脏数据、类别形态、拼音字段合法性、同名与 alias 冲突
2. 当前还没有明确判断“这个词是否适合进入 runtime 词典”
3. 当前还没有把“只能 candidate 的词”作为一等逻辑来治理

## 3. 统一原则

只有当一个词条能够形成稳定、有限、可审计的 `triggerText -> canonicalText` 映射时，才允许进入 runtime 词典。

其中：

1. `triggerText`
   - literal 通道：`alias`
   - pinyin 通道：`canonical + alias` 对应的拼音触发

2. `canonicalText`
   - runtime 最终输出文本

## 4. 统一后的准入模型

### 4.1 两层准入

1. `blocked`
2. `ready`

### 4.2 ready 的运行方式

1. `ready + replace`
2. `ready + candidate`

说明：

1. 不再把 `warning` 当成第三层状态
2. 原 `warning` 全部下沉为：
   - `reviewHints`
   - `riskNotes`
   - `recommendedAction`

## 5. blocked / ready 的判断树

### 5.1 直接 blocked

满足任一条件即 `blocked`：

1. 标准词本身不适合作为词典项
   - 说明词
   - 栏目词
   - 句子片段
   - 编号/脏数据
   - 明显强依赖上下文的泛词

2. 无法形成有效触发条件
   - alias 为空且无可用拼音触发
   - 录入后 runtime 也无法稳定命中

3. 冲突或歧义无法安全降级成 `candidate`
   - 冲突对象太多
   - alias 不可枚举
   - 输入太泛
   - 候选没有业务价值
   - 会把 `correct_cand` 污染成大面积噪声

### 5.2 ready + replace

满足以下条件：

1. 标准词稳定
2. alias 可枚举
3. 不会误改其他已正确标准词
4. 不会形成多标准词歧义
5. 风险足够低

### 5.3 ready + candidate

满足以下条件：

1. 有歧义
2. 但歧义有限、可解释
3. 仍有推荐价值
4. 不适合自动替换
5. 适合进入 `correct_cand`
6. 候选标准词数量必须 `< 4`

## 6. 系统处理建议模型

录入或导入时，系统必须直接给出建议，不要求用户逐条自己判断。

### 6.1 输出结构

每个词条或导入行都应输出：

1. `recommendedAction`
2. `reasonCodes`
3. `reasonSummary`
4. `reviewHints`
5. `runtimeSuitability`

### 6.2 推荐动作枚举

统一建议动作：

1. `save_replace`
2. `save_candidate`
3. `merge_existing`
4. `append_alias_to_existing`
5. `skip_blocked`

### 6.3 已冻结口径

以下口径已确认，不再作为开放讨论项：

1. 当 `triggerText` 命中其他已存在标准词时：
   - 一律不允许 `replace`
   - 系统建议优先顺序为：
     - `merge_existing`
     - `append_alias_to_existing`
     - `save_candidate`
     - `skip_blocked`
   - 只有在“没有明确并入目标、歧义范围有限且确有推荐价值”时，才允许降级成 `save_candidate`

2. `ready + candidate` 第一版不允许没有 `alias`、只靠拼音参与推荐

3. `replaceMode = candidate` 第一版仍保留 `candidateOnly` 规则兜底

4. `reviewHints` 第一版不落库，先按当前状态动态计算
   - 手工录入页、词条详情页、审核页均按当前词条状态和当前词典现状即时生成
   - 批量导入预览页因本身存在行级预览结果落盘能力，可保留当次建议摘要
5. `merge_existing` 与 `append_alias_to_existing` 第一版不做全自动落库
   - 采用“系统建议 + 用户一键确认批量执行”模式
6. `save_candidate` 的歧义阈值固定为 `< 4`
   - 即候选标准词数量最多允许 `3` 个
   - 达到 `4` 个及以上时不得进入 `candidate`
7. 批量导入页必须增加“建议动作”汇总卡片
   - 至少包括：
     - `建议替换导入`
     - `建议候选导入`
     - `建议并入已有`
     - `建议补录已有`
     - `建议阻断跳过`
8. 批量导入页必须增加“建议动作”单独筛选项
9. 手工录入页必须增加轻量版“系统建议说明”帮助卡片

### 6.4 目标词条解析规则

`merge_existing` 与 `append_alias_to_existing` 只有在系统能解析出唯一目标词条时，才允许进入“按系统建议处理”自动执行链。

统一规则如下：

1. 若可唯一定位目标词条，则允许：
   - `merge_existing`
   - `append_alias_to_existing`
2. 若存在多个候选目标词条，则不允许自动并入或自动补录
   - 只能降级为：
     - `save_candidate`
     - `skip_blocked`
3. 若不存在明确目标词条，则不允许：
   - `merge_existing`
   - `append_alias_to_existing`
4. `merge_existing` 与 `append_alias_to_existing` 的目标词条必须在页面上明确显示：
   - `targetTermId`
   - `targetCanonicalText`
5. 第一版即使存在唯一目标词条，也不直接自动落库
   - 仍需用户触发“按系统建议处理”后批量执行

### 6.5 存量词条收口口径

第一版统一方案只强制约束：

1. 新手工录入词条
2. 新批量导入批次

对存量词条的处理口径为：

1. 第一版不做全库自动回填或全库自动改模式
2. 存量词条继续按当前已发布状态运行
3. 后续若要对存量词条统一重评估，单独开重评估批次，不混入本次准入重构

## 7. 页面方案

### 7.1 手工录入页

页面要求：

1. 保存前直接显示系统建议
2. 如果建议是 `save_replace`
   - 允许按替换词条保存
3. 如果建议是 `save_candidate`
   - 允许保存
   - 但页面自动锁定为“推荐候选”
   - 不允许切回 `replace`
4. 如果建议是 `merge_existing`
   - 不再按新建保存
   - 页面直接提示并入已有词条
5. 如果建议是 `append_alias_to_existing`
   - 页面直接提示补录到现有词条
6. 如果建议是 `skip_blocked`
   - 禁止保存

### 7.2 批量导入页

页面要求：

1. 每一行预览都显示系统建议
2. 批量导入提供主动作：
   - `按系统建议处理`
3. 不要求用户逐条选择
4. 导入结果明确区分：
   - 直接替换导入
   - 推荐候选导入
   - 并入已有词条
   - 补录已有词条
   - 阻断跳过

### 7.3 candidate 的审核与发布边界

第一版固定为：

1. `candidate` 与 `replace` 走同一套词条审核链
2. `candidate` 审核通过后，同样进入 release 与 snapshot
3. 运行时是否直接替换，不由审核状态决定，而由：
   - `replaceMode`
   - `pinyinRuntimeMode`
4. 页面需要显式标识：
   - `仅推荐`
   - `不可直接替换`

### 7.4 页面中文化

所有用户可见文案必须中文化，包括：

1. 按钮
2. 状态 badge
3. 下拉 label
4. flash / callout
5. 卡片标题
6. 结果摘要

## 8. runtime 方案

### 8.1 `/correct`

1. 只允许 `replace` 生效
2. `candidate` 不参与主替换结果

### 8.2 `/correct_cand`

1. `replace` 形成主结果
2. `candidate` 进入候选集合
3. 当没有任何 `replace` 命中、只有 `candidate` 命中时：
   - `correctedTexts[0]` 固定返回原文
   - `candidate` 只出现在后续候选位
4. 不允许把 `candidate` 结果提升成主替换结果

### 8.3 `/ws/runtime/correct`

1. 与 `/correct` 保持一致

### 8.4 `/ws/runtime/correct_cand`

1. 与 `/correct_cand` 保持一致

### 8.5 当前实现必须修正的一点

当前 literal 通道没有真正尊重 `replaceMode = candidate`。

目前实现是：

1. `replaceMode = block` 才会 `block`
2. 其他值都会被当成 `replace`

因此 runtime 必须补改，才能和治理逻辑真正统一。

### 8.6 candidate 主结果规则

为避免 `correct` 与 `correct_cand` 的语义重新混乱，第一版冻结如下：

1. `/correct`
   - 只返回 `replace` 驱动的 `correctedText`
2. `/correct_cand`
   - 主结果位始终先看 `replace`
   - 无 `replace` 命中时主结果保持原文
   - `candidate` 只负责补充候选，不负责改写主结果

### 11.1 行级结果

1. `recommendedAction`
2. `targetTermId`
3. `targetCanonicalText`
4. `reasonSummary`
5. `reviewHints`

### 11.2 执行结果汇总

批次结果汇总至少需要区分：

1. `replaceImportedCount`
2. `candidateImportedCount`
3. `mergedExistingCount`
4. `aliasAppendedCount`
5. `skippedBlockedCount`

### 11.3 审核任务边界

1. `save_replace` 与 `save_candidate` 新建词条后，按词条审核链生成审核任务
2. `merge_existing` 与 `append_alias_to_existing` 若发生词条内容变更，也必须进入审核链
3. `skip_blocked` 不生成审核任务
