# JOB-105 v1.0 批量导入按行阻断与可通过记录导入

- 文档状态：active
- 适用版本：v1.0
- 文档类型：job
- 所属工作区：2026-04-17-v1.0正式文档工作区
- 最后更新时间：2026-04-21
- 责任对象：批量导入确认逻辑、导入详情页、导入结果汇总、错误导出

## 1. 目标

把当前“批次中只要存在 1 条 `error` 就整批不能确认导入”的逻辑，改成：

1. 行级阻断继续保留
2. `ready` / `warning` 行允许继续导入
3. `error` 行只跳过，不入库
4. 页面和结果汇总明确展示“导入成功 / warning 导入 / block 跳过”

## 2. 当前问题

当前 `confirmImportJob()` 的核心门禁是：

1. 只要任意 `import_job_rows.status === error`
2. 整个批次直接返回 `import_job_blocked_rows`

这会导致：

1. 小量阻断行拖死整批导入
2. 大量 `ready` 行不能及时入库
3. 用户只能反复改文件重传，操作成本高

## 3. 目标口径

### 3.1 仍然整批阻断的场景

保留给：

1. 模板错误
2. 文件损坏
3. 列缺失
4. 系统级异常

### 3.2 改为行级阻断的场景

包括但不限于：

1. `alias_conflict`
2. `gov_term_shape_blocked`
3. `poi_road_shape_blocked`
4. `dictionary_noise_blocked`
5. 单字别名阻断

### 3.3 导入确认后的行为

1. `ready` 行导入
2. `warning` 行导入
3. `error` 行跳过
4. `error` 行不入库
5. `error` 行不生成审核任务

### 3.4 页面按钮与下拉选择口径

本 job 必须同时冻结以下页面交互口径：

#### 批次详情页按钮

1. 主按钮不再叫“确认导入”
2. 主按钮改为“导入可通过记录”
3. 当 `job.status = preview_ready` 且 `importableRows > 0` 时，即使存在 `error` 行，也允许点击
4. 当 `importableRows = 0` 时，主按钮置灰
5. 置灰提示文案要明确说明：
   - 当前没有可导入记录
   - 仅剩阻断行，需修正后重新预览
6. “取消批次”按钮继续保留
7. “下载错误报表”按钮继续保留
8. 如果批次已导入但仍有被跳过阻断行，“下载错误报表”仍应可下载

冻结后的按钮文案建议如下：

1. 主按钮：
   - `导入可通过记录`
2. 次按钮：
   - `取消批次`
3. 报表按钮：
   - `下载阻断报表`
4. 无错误时禁用按钮：
   - `无阻断报表`

#### 导入批次列表页

本 job 不要求重构列表页状态筛选，但要明确：

1. 列表页批次状态下拉仍保留：
   - `全部批次状态`
   - `待确认导入`
   - `已导入`
   - `已取消`
2. 列表页摘要文案要从：
   - `可导入 / 需确认 / 错误`
   继续保留为预览统计
3. 当批次导入完成后，如果存在被跳过阻断行，列表页批次摘要应能显示：
   - `已导入（含阻断跳过）`
   或等价提示

#### 预览汇总卡片

预览态继续保留：

1. `预览总行数`
2. `可直接导入`
3. `需人工确认`
4. `错误行`
5. `可导入总数`

导入完成后，结果卡片需要改成：

1. `成功导入`
2. `warning 导入`
3. `跳过阻断`
4. `新增词条 / 更新词条 / 新增错误词 / 错误数`

冻结后的结果卡片建议为：

1. `成功导入`
2. `warning 导入`
3. `跳过阻断`
4. `新增词条`
5. `更新词条`
6. `新增错误词`
7. `错误数`

其中：

1. `错误数` 表示确认导入阶段真正失败的异常数
2. `跳过阻断` 表示预览阶段已知且被跳过的 `error` 行数
3. 不再把 `跳过阻断` 混进 `错误数`

#### 预览行结果区块

当前筛选区有两个下拉：

1. `行状态`
2. `处理决策`

本 job 必须明确：

1. `行状态` 是否继续使用：
   - `ready`
   - `warning`
   - `error`
   - `imported`
2. `error` 在导入完成后，展示语义要切成：
   - `阻断未导入`
3. `处理决策` 是否继续使用：
   - `accept`
   - `merge_existing`
   - `pending`
4. 是否新增决策值：
   - `skipped_blocked`

默认建议：

1. 保留现有 `rowStatus` 取值不变
2. 新增 `rowDecision = skipped_blocked`
3. 导入完成后，被跳过的阻断行展示为：
   - `状态 = error`
   - `决策 = skipped_blocked`

冻结后的筛选项建议如下：

1. `行状态`
   - `全部预览状态`
   - `可直接导入`
   - `需人工确认`
   - `错误行`
   - `已导入`
2. `处理决策`
   - `全部处理决策`
   - `直接接收（accept）`
   - `合并已有词条（merge_existing）`
   - `待人工确认（pending）`
   - `阻断跳过（skipped_blocked）`

说明：

1. 不新增新的 `rowStatus`
2. 只新增新的 `rowDecision`
3. 导入完成后，用户可以通过：
   - `行状态 = error`
   - `处理决策 = skipped_blocked`
   精确筛出“被跳过阻断行”

#### 成功提示与下一步引导

导入完成后的 flash 和 callout 文案不能继续只说“导入已确认并入库”。

需要改成包含：

1. 成功导入多少行
2. warning 导入多少行
3. 跳过阻断多少行
4. 下一步去哪里看：
   - 影响词条
   - 生成的审核任务
   - 阻断明细报表

冻结后的成功提示建议如下：

1. title：
   - `可通过记录已导入`
2. description：
   - `成功导入 X 行，warning 导入 Y 行，跳过阻断 Z 行。`
3. actions：
   - `查看本批词条审核`
   - `下载阻断报表`
   - `返回批量导入`

#### 结果汇总字段口径

本 job 需要明确 `resultSummary` 的展示字段至少包括：

1. `newTermCount`
2. `updatedTermCount`
3. `newAliasCount`
4. `updatedAliasCount`
5. `errorCount`
6. `importedReadyCount`
7. `importedWarningCount`
8. `skippedBlockedCount`
9. `createdReviewTaskIds`

其中新增字段建议为：

1. `importedReadyCount`
2. `importedWarningCount`
3. `skippedBlockedCount`

这样页面卡片、flash、详情 JSON 和后续测试口径可以统一。

1. 存在 `error` 行的批次仍可继续导入可通过记录
2. `error` 行不会进入主库
3. 页面提示清楚哪些行已导入、哪些行被跳过
4. `import_job_results` 能反映“跳过阻断行”数量
5. 测试、文档、真源同步
6. 页面按钮、下拉项和结果卡片口径一致
