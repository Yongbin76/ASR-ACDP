## 1. 工作区定位

本工作区用于承接 ACDP 当前版本的正式文档基线，同时定义后续衍生开发时的文档治理规则。

本工作区解决 4 个问题：

1. 当前版本到底以什么为准。
2. 后续新增需求时，文档如何增加、修改、归档。
3. 文档、JOBLIST、CHECKLIST、单一真源如何形成同一条治理链。
4. 阶段验收、后续开发、测试交接时，如何只看本目录就完成理解与执行。

## 2. 工作区边界

本工作区是一个封闭边界的正式文档集。

约束如下：

1. 本目录内的正式文档不引用本目录之外的旧文档。
2. 旧文档只作为整理输入，不再作为本工作区的外部依赖。
3. 本目录内的文档之间允许相互引用。
4. 若文档内容涉及代码、配置、数据库或真源文件，可以直接写仓库路径，但不依赖目录外 Markdown 文档完成解释。
5. 本目录是当前版本的正式文档工作区；后续版本需要建立新的工作区并按归档规则处理本目录。

## 7. 当前工作区内的治理逻辑

本工作区采用以下治理链：

`代码与配置事实` -> `正式基线文档` -> `JOB 与文档绑定` -> `CHECKLIST 与收尾标准` -> `单一真源同步` -> `归档与版本快照`

含义如下：

1. 基线文档只记录当前代码和配置已经存在的事实。
2. 后续每个 job 必须定义影响哪些正式文档。
3. job 关单前必须完成文档同步与回归验证。
4. 单一真源负责管理 job 状态与文档元数据关系。
5. 新版本建立新工作区时，当前工作区按归档规则冻结。

## 8. 后续新增文档的规则

1. 新增正式文档前，先判断是更新现有基线文档还是新增专题文档。
2. 新增专题文档时，必须同时更新本索引、`JOB与文档绑定矩阵` 和 `文档变更记录`。
3. 未被本索引收录的文档，不得视为正式依据。
4. 未绑定 job 的文档，只能是基线文档、模板或归档文档，不能作为实施文档存在。
5. 同一主题在同一工作区中只允许一份 `active` 的正式基线文档。
