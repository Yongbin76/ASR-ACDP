# ACDP 运行发布与治理机制

- 文档状态：active
- 适用版本：v1.0
- 文档类型：baseline
- 所属工作区：2026-04-17-v1.0正式文档工作区
- 最后更新时间：2026-04-17
- 责任对象：发布状态、审批状态、流量状态、节点治理、快照下发

## 1. 设计原则

运行发布与治理机制必须回答 5 个问题：

1. 当前 release 处于哪个生命周期。
2. 当前 release 是否完成审批。
3. 当前 release 是否已经下发到 runtime。
4. runtime 节点是否收敛。
5. 出现异常时如何定位是门禁问题、审批问题还是节点问题。

## 2. 发布状态分层

当前版本明确把以下状态分层处理。

### 2.1 版本状态

版本状态只描述 release 自身生命周期。

当前值为：

1. `built`
2. `canary`
3. `published`

含义：

1. `built`
   - 版本已构建完成，但尚未正式发布。
2. `canary`
   - 版本已进入灰度阶段。
3. `published`
   - 版本已正式发布。

### 2.2 审批状态

审批状态描述发布审核进展，而不是 release 生命周期。

当前值为：

1. `not_submitted`
2. `pending`
3. `partially_approved`
4. `approved`
5. `rejected`

### 2.3 流量状态

流量状态描述 release 当前是否承担线上流量。

当前典型值包括：

1. `no_gray`
2. `gray_enabled`
3. `gray_enabled_other`
4. `gray_closed`

### 2.4 门禁状态

门禁状态描述 release 是否存在阻断或风险项。

当前典型阻断包括：

1. `pending_term_review`
2. `pending_pinyin_candidate_review`
3. 发布后 rollout 未完成
4. validation 结果未达标

## 3. 发布审核规则

### 3.1 提交条件

release 只有在 `built` 或 `canary` 状态下才能提交发布审核。

### 3.2 审批通过规则

当前审批规则受治理策略控制，主要包括：

1. 提交人与审核人分离。
2. 双人审批时，审批人必须不同。

### 3.3 审批通过后的影响

审批通过后，release 获得被灰度、正式发布或下发的资格，但不会自动切换 release 状态。

## 4. 灰度与正式发布

### 4.1 灰度

灰度通过 `gray_policies` 记录，当前主要使用百分比策略。

灰度开启后：

1. runtime 根据 `trafficKey` 做 stable / canary 分流。
2. `canary` 版本承接命中的灰度请求。

### 4.2 正式发布

正式发布意味着：

1. release 状态切到 `published`。
2. 版本被视为正式线上版本。
3. admin 仍需继续观察 rollout、风险和节点收敛。

## 5. 节点备案与运行节点

### 5.1 节点备案

节点备案是 runtime 接入 admin 的前置条件。

备案信息包括：

1. `node_id`
2. `node_name`
3. `env`
4. `address`
5. `enabled`
6. `registration_secret_hash`
7. `secret_fingerprint`

### 5.2 运行节点

运行节点是 runtime 成功注册后形成的实时对象。

主要关注：

1. 是否在线。
2. 当前版本。
3. 目标版本。
4. 最近心跳。
5. 最近应用状态。
6. 最近错误。

### 5.3 治理原则

1. 主列表只以备案节点为主对象。
2. 未备案 runtime 只能被视为异常接入事件，不应成为正式主对象。
3. 实时状态与历史异常必须分开展示。

## 6. runtime 控制链路

当前 runtime 与 admin 的控制链路如下：

1. 节点启动。
2. runtime 使用备案信息与注册密钥向 admin 注册。
3. runtime 定期 heartbeat。
4. runtime 拉取 `/api/runtime-control/me`。
5. 若发现 `desired_version` 变化，则下载新快照。
6. 下载、安装、切换成功后，回传 apply 结果。
7. 后续继续上报统计。

## 7. 快照下发模式

### 7.1 `file`

1. 适合本地验证或共享文件模式。
2. runtime 直接读取文件路径或文件型下载地址。

### 7.2 `minio`

1. admin 通过对象存储提供下载地址。
2. runtime 通过预签名地址下载。

### 7.3 `admin_http_signed`

1. admin 直接提供下载路由。
2. 下载地址带签名、时效和节点绑定信息。
3. 当前版本已将该模式作为可落地模式之一。

## 8. rollout 与节点收敛

节点收敛判断主要依赖：

1. `desired_version`
2. `current_version`
3. `last_apply_status`
4. `last_heartbeat_at`

常见判断结果包括：

1. 已对齐。
2. 待收敛。
3. 待安装。
4. 离线。
5. 应用失败。

## 9. runtime 统计回传

runtime 会记录并回传：

1. 请求量。
2. HTTP / WebSocket 请求量拆分。
3. 词条命中量。
4. 小时粒度词条命中排行。
5. 峰值并发。

这些统计最终汇总到：

1. 节点级统计表。
2. 全局统计表。
3. `/console/runtime-nodes` 页面。

## 10. 运行验证

当前版本提供独立“运行验证”页面，对以下接口做验证：

1. `correct`
2. `correct_cand`

当前模式为：

1. 验证 cluster 当前口径。
2. 尚未扩展到指定 runtime 节点或细粒度灰度验证。

## 11. 当前机制中的关键约束

1. 发布状态、审批状态、流量状态、门禁状态必须分层展示，不能混成一个状态。
2. 节点备案与运行节点必须拆分。
3. 实时在线状态与历史异常必须拆分。
4. 快照下发模式只影响交付方式，不改变控制链主语义。
5. `published` 后仍然允许存在发布后风险观察，但不应重新定义 release 的生命周期状态。
