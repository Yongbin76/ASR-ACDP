# ACDP 中文拼音增强与词条准入设计

- 文档状态：active
- 适用版本：v1.0
- 文档类型：baseline
- 所属工作区：2026-04-17-v1.0正式文档工作区
- 最后更新时间：2026-04-17
- 责任对象：拼音增强、候选整句、词条准入规则

## 1. 设计目标

本模块解决两类问题：

1. 词条进入系统前，如何统一准入。
2. runtime 纠错时，如何结合拼音召回、候选输出和审核链路。

## 2. 词条准入的统一原则

当前系统不允许词条中心、导入中心和审核中心各自维护不同的准入口径。

统一准入由 `term-admission` 负责，作用于：

1. 手工创建词条。
2. 批量导入行校验。
3. 审核摘要展示。

## 3. 当前准入检查维度

准入检查至少包括：

1. 标准词是否为空。
2. 别名集合是否为空或重复。
3. 标准词与别名是否存在明显冲突。
4. 别名去空、去重后的长度是否合法。
5. 是否命中单字别名阻断规则。

## 4. 单字别名阻断规则

### 4.1 规则定义

当前规则以“错误词 / 别名集合”为核心判断：

1. 别名拆分后，先去空。
2. 别名去空后，再去重。
3. 以去空后的字符数判断是否为单字。
4. 只要存在任意一个单字别名，即阻断。
5. 如果别名集合为空，也阻断。

### 4.2 规则示例

1. 标准词=`王`，错误词=`汪`
   - `汪` 是单字
   - 结果：阻断
2. 标准词=`王`，错误词=`汪|忘`
   - 存在单字别名
   - 结果：阻断
3. 标准词=`欧阳`，错误词=`欧洋`
   - `欧洋` 不是单字
   - 结果：可通过
4. 标准词=`司马`，错误词=`司嗎|司玛`
   - 都不是单字
   - 结果：可通过
5. 标准词=`李`，错误词=`李某`
   - 别名不是单字
   - 结果：可通过
6. 标准词=`李`，错误词为空
   - 别名集合为空
   - 结果：阻断
7. 标准词=`王`，错误词=`汪|忘了`
   - 存在单字别名
   - 结果：阻断

### 4.3 当前规则边界

该规则与业务属性是否为 `surname` 无关。

它属于当前版本的统一准入规则，而不是某个类别的特例规则。

## 5. 拼音增强模型

### 5.1 runtime 拼音能力

runtime 的拼音能力不是独立于字面匹配的另一套系统，而是补充召回通道。

当前流程为：

1. 字面 AC 匹配先执行。
2. 若未关闭拼音通道，再执行拼音精确召回。
3. 命中的字面候选与拼音候选统一进入规则评估。
4. 规则评估决定 `replace`、`candidate` 或 `block`。

### 5.2 拼音画像字段

每个词条的拼音画像主要包括：

1. `full_pinyin_no_tone`
2. `initials`
3. `syllables_json`
4. `runtime_mode`
5. `polyphone_mode`
6. `custom_full_pinyin_no_tone`
7. `alternative_readings_json`
8. `notes`

## 6. 拼音候选审核链路

### 6.1 候选生成

系统可为词条生成拼音候选，并把候选快照提交为审核任务。

候选审核任务类型为：

1. `pinyin_candidate_review`

目标类型为：

1. `pinyin_candidate`

### 6.2 候选审核通过的副作用

拼音候选审核通过后：

1. 系统读取候选读音。
2. 若该读音不同于当前主读音，且不在备用读音列表中，则写入 `alternative_readings_json`。

## 7. runtime 纠错输出模型

### 7.1 标准纠错

`correct` 与 `ws correct` 走同一套底层纠错引擎。

当前输出只返回：

1. `correctedText`

### 7.2 候选纠错

`correct_cand` 与 `ws correct_cand` 同样复用同一套底层纠错引擎，但在结果层增加候选整句构造。

当前输出返回：

1. `correctedTexts`

其中：

1. 第一个元素是主修正结果。
2. 后续元素是基于替换槽位和候选项组合出的整句候选。

## 8. 候选整句构造逻辑

候选整句构造由 `runtime-candidates.js` 承担，当前逻辑如下：

1. 先从一次 `matchDetailed` 结果中识别所有替换槽位。
2. 对同一 span 的候选项按文本去重。
3. 优先保留高置信度项；同分时优先保留 `literal` 通道。
4. 每个槽位最多保留 3 个替代候选。
5. 主结果始终作为 `correctedTexts[0]`。
6. 后续优先尝试单槽位替代。
7. 若允许，再尝试双槽位组合。
8. 最终按得分排序并截断输出条数。

## 9. runtime 分流与拼音增强的关系

runtime 先决定走 stable 还是 canary，再执行纠错。

也就是说：

1. `trafficKey` 只决定使用哪个 runtime 快照。
2. 拼音召回、规则评估和候选整句是在选定 runtime 后执行。

## 10. 当前版本的设计边界

1. 当前拼音增强仍以精确召回和规则评估为主。
2. 当前没有引入复杂的语言模型排序。
3. 候选整句输出只基于单次纠错详情构造，不做跨请求学习。
4. 准入规则优先追求可解释和统一，不追求高度复杂的语义判词。
