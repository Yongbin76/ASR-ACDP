# ACDP 中文拼音增强与词条准入设计

- 文档状态：active
- 适用版本：v1.0
- 文档类型：baseline
- 所属工作区：2026-04-17-v1.0正式文档工作区
- 最后更新时间：2026-04-22
- 责任对象：拼音增强、候选整句、词条准入规则

## 1. 设计目标

本模块解决两类问题：

1. 词条进入系统前，如何统一准入。
2. runtime 纠错时，如何结合拼音召回、候选输出和审核链路。

## 2. 词条准入的统一原则

当前系统不允许词条中心、导入中心和审核中心各自维护不同的准入口径。

统一准入由 `term-admission` 负责，作用于：

1. 手工创建词条。
2. 批量导入行校验。
3. 审核摘要展示。

## 3. 当前准入检查维度

准入检查至少包括：

1. 标准词是否为空。
2. 别名集合是否为空或重复。
3. 标准词与别名是否存在明显冲突。
4. 别名去空、去重后的长度是否合法。
5. 是否命中单字别名阻断规则。
6. 是否命中类别形态规则。

## 4. 单字别名阻断规则

### 4.1 规则定义

当前规则以“错误词 / 别名集合”为核心判断：

1. 别名拆分后，先去空。
2. 别名去空后，再去重。
3. 以去空后的字符数判断是否为单字。
4. 只要存在任意一个单字别名，即阻断。
5. 如果别名集合为空，也阻断。

### 4.2 规则示例

1. 标准词=`王`，错误词=`汪`
   - `汪` 是单字
   - 结果：阻断
2. 标准词=`王`，错误词=`汪|忘`
   - 存在单字别名
   - 结果：阻断
3. 标准词=`欧阳`，错误词=`欧洋`
   - `欧洋` 不是单字
   - 结果：可通过
4. 标准词=`司马`，错误词=`司嗎|司玛`
   - 都不是单字
   - 结果：可通过
5. 标准词=`李`，错误词=`李某`
   - 别名不是单字
   - 结果：可通过
6. 标准词=`李`，错误词为空
   - 别名集合为空
   - 结果：阻断
7. 标准词=`王`，错误词=`汪|忘了`
   - 存在单字别名
   - 结果：阻断

### 4.3 当前规则边界

该规则与业务属性是否为 `surname` 无关。

它属于当前版本的统一准入规则，而不是某个类别的特例规则。

### 4.4 路名类别形态规则

当前 `ROAD_INFO / poi_road` 会做基础形态判断。

当前版本允许的常见后缀至少包括：

1. `路`
2. `街`
3. `巷`
4. `弄`
5. `大道`
6. `公路`
7. `桥`
8. `高速`
9. `立交`

这意味着像 `东海大桥`、`京沪高速`、`马陆立交` 这类词条，在其他字段合法时可通过准入，不应再因为“路名形态”被阻断。

### 4.5 政府部门类别形态规则

当前 `GOV_INFO / gov_term` 会做基础机构形态判断。

当前版本允许的强机构后缀至少包括：

1. `委员会`
2. `办公室`
3. `研究室`
4. `参事室`
5. `局`
6. `厅`
7. `办`
8. `法院`
9. `检察院`

这意味着像 `人民政府外事办公室`、`人民政府研究室`、`人民政府参事室` 这类词条，在其他字段合法时可通过准入。

同时，当前版本并没有把所有以 `室` 结尾的文本都直接放行。

例如：

1. `会议室`
2. `值班室`
3. `档案室`

这类更像房间、场所或内部分区的文本，仍应继续被拦截，避免把非机构名误录成政府部门词条。

## 5. 拼音增强模型

### 5.1 runtime 拼音能力

runtime 的拼音能力不是独立于字面匹配的另一套系统，而是补充召回通道。

当前流程为：

1. 字面 AC 匹配先执行。
2. 若未关闭拼音通道，再执行拼音精确召回。
3. 命中的字面候选与拼音候选统一进入规则评估。
4. 规则评估决定 `replace`、`candidate` 或 `block`。

### 5.2 拼音画像字段

每个词条的拼音画像主要包括：

1. `full_pinyin_no_tone`
2. `initials`
3. `syllables_json`
4. `runtime_mode`
5. `polyphone_mode`
6. `custom_full_pinyin_no_tone`
7. `alternative_readings_json`
8. `notes`

## 6. 拼音候选审核链路

### 6.1 候选生成

系统可为词条生成拼音候选，并把候选快照提交为审核任务。

候选审核任务类型为：

1. `pinyin_candidate_review`

目标类型为：

1. `pinyin_candidate`

### 6.2 候选审核通过的副作用

拼音候选审核通过后：

1. 系统读取候选读音。
2. 若该读音不同于当前主读音，且不在备用读音列表中，则写入 `alternative_readings_json`。

## 7. runtime 纠错输出模型

### 7.1 标准纠错

`correct` 与 `ws correct` 走同一套底层纠错引擎。

当前输出只返回：

1. `correctedText`

### 7.2 候选纠错

`correct_cand` 与 `ws correct_cand` 同样复用同一套底层纠错引擎，但在结果层增加候选整句构造。

当前输出返回：

1. `correctedTexts`

其中：

1. 第一个元素是主修正结果。
2. 后续元素是基于替换槽位和候选项组合出的整句候选。

## 8. 候选整句构造逻辑

候选整句构造由 `runtime-candidates.js` 承担，当前逻辑如下：

1. 先从一次 `matchDetailed` 结果中识别所有替换槽位。
2. 对同一 span 的候选项按文本去重。
3. 优先保留高置信度项；同分时优先保留 `literal` 通道。
4. 每个槽位最多保留 3 个替代候选。
5. 主结果始终作为 `correctedTexts[0]`。
6. 后续优先尝试单槽位替代。
7. 若允许，再尝试双槽位组合。
8. 最终按得分排序并截断输出条数。

## 9. runtime 分流与拼音增强的关系

runtime 先决定走 stable 还是 canary，再执行纠错。

也就是说：

1. `trafficKey` 只决定使用哪个 runtime 快照。
2. 拼音召回、规则评估和候选整句是在选定 runtime 后执行。

## 10. 当前版本的设计边界

1. 当前拼音增强仍以精确召回和规则评估为主。
2. 当前系统还未把“词典准入”和“runtime 输出模式”完全收口到同一套标准。

## 11. JOB-106 统一收口口径

### 11.1 准入只保留两层

统一后，词条准入层只保留：

1. `blocked`
2. `ready`

说明：

1. 原 `warning` 不再作为第三层准入状态。
2. 原 `warning` 下沉为：
   - `reviewHints`
   - `riskNotes`
   - `recommendedAction`

### 11.2 ready 的运行方式

`ready` 不再默认等于“可直接替换”。

统一后必须继续区分：

1. `ready + replace`
2. `ready + candidate`

判断标准为：

1. `replace`
   - 词条能形成稳定、有限、不会误改正确词的 `triggerText -> canonicalText` 映射。
2. `candidate`
   - 词条存在歧义，但歧义有限、可解释、仍有推荐价值。

### 11.3 triggerText 定义

为避免治理规则与 runtime 实际触发逻辑脱节，统一使用 `triggerText` 作为判断对象。

1. literal 通道的 `triggerText` 为 `alias`
2. pinyin 通道的 `triggerText` 为 `canonical + alias` 对应的拼音触发

### 11.4 candidate 的准入口径

以下情况不再直接按 `blocked` 处理，而是先判断能否降级成 `candidate`：

1. 当前词条的触发文本会碰到其他词条的标准词
2. 同一个输入可能落到多个 `canonical`

只有在以下条件成立时，才最终 `blocked`：

1. 歧义对象过多
2. alias 明显不可枚举
3. 输入过于泛化
4. 候选没有业务价值
5. 会把 `correct_cand` 污染成大面积噪声

### 11.5 系统处理建议

录入或导入时，系统必须直接给出处理建议，不要求用户逐条判断。

每个词条或导入行至少应输出：

1. `recommendedAction`
2. `reasonCodes`
3. `reasonSummary`
4. `reviewHints`
5. `runtimeSuitability`

建议动作范围固定为：

1. `save_replace`
2. `save_candidate`
3. `merge_existing`
4. `append_alias_to_existing`
5. `skip_blocked`
2. 当前没有引入复杂的语言模型排序。
3. 候选整句输出只基于单次纠错详情构造，不做跨请求学习。
4. 准入规则优先追求可解释和统一，不追求高度复杂的语义判词。
