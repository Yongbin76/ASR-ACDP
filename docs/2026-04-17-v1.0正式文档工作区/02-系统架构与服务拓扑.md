# ACDP 系统架构与服务拓扑

- 文档状态：active
- 适用版本：v1.0
- 文档类型：baseline
- 所属工作区：2026-04-17-v1.0正式文档工作区
- 最后更新时间：2026-04-17
- 责任对象：服务划分、控制面/数据面拓扑、主链路

## 1. 架构总述

ACDP 当前采用“控制面 + 数据面 + 文档与治理面”的结构。

### 1.1 控制面

控制面由 admin 服务承载，核心职责是：

1. 管理词典、审核、发布、验证样本和配置。
2. 为 `/console` 提供聚合接口和工作台能力。
3. 构建 release snapshot。
4. 管理 runtime 节点备案、注册、控制状态和统计汇总。
5. 提供 runtime 快照下载与控制接口。

### 1.2 数据面

数据面由 runtime 服务承载，核心职责是：

1. 对外提供文本纠错能力。
2. 执行 stable / canary 双 runtime 选择。
3. 下载并安装目标 snapshot。
4. 记录本地统计并回传 admin。

### 1.3 文档与治理面

文档与治理面由以下资产承载：

1. 本工作区正式文档。
2. 单一真源与 JobList 机制。
3. 帮助页与运维手册。
4. 测试脚本、runbook 和发布包模板。

## 2. 主要服务与资产

| 对象 | 类型 | 作用 |
|---|---|---|
| admin server | 服务 | 控制面、管理后台、聚合接口、runtime 控制接口 |
| runtime server | 服务 | 数据面、纠错接口、版本切换、本地统计 |
| `/console` | SPA | 主后台前端 |
| `/admin` | 页面 | 兼容后台入口 |
| `platform.db` | SQLite | 控制面主数据库 |
| `snapshot.json` | 文件 | runtime 加载的发布快照 |
| `manifest.json` | 文件 | release 元信息 |
| `runtime_state/current.json` | 文件 | runtime 当前安装版本状态 |
| `runtime_stats.db` | SQLite | runtime 本地统计缓冲 |
| `artifact store` | 存储 | release 快照分发载体 |

## 3. 启动拓扑

### 3.1 单进程模式

命令：`npm run start:prototype`

特点：

1. admin 与 runtime 在同一进程。
2. 使用同一套 workspace。
3. 适合本地开发、集成验证和快速演示。

### 3.2 拆分模式

命令：

1. `npm run start:admin`
2. `npm run start:runtime`

特点：

1. admin 和 runtime 分进程运行。
2. runtime 通过控制接口与 admin 同步。
3. 更接近真实部署形态。

### 3.3 多 runtime 模式

命令：`npm run start:runtime:instance -- ...`

特点：

1. 一个 admin 可对接多个 runtime。
2. 每个 runtime 实例有独立的端口、实例目录、节点标识和运行状态目录。
3. 适合验证多节点注册、版本下发和收敛观察能力。

## 4. 逻辑拓扑

### 4.1 控制面拓扑

`浏览器 -> /console -> admin-surface -> console-service/platform-db/config -> workspace`

这条链负责：

1. 页面展示。
2. 数据聚合。
3. 配置读取。
4. 审核与发布动作。

### 4.2 数据面拓扑

`调用方 -> runtime-surface -> runtime.match / runtime.matchDetailed -> snapshot bundle`

这条链负责：

1. 输入文本接收。
2. AC 字面匹配。
3. 拼音召回。
4. 规则评估。
5. 替换输出或候选整句输出。

### 4.3 控制同步拓扑

`runtime -> admin(runtime-control API) -> runtimeControlState -> artifact delivery -> runtime local install`

这条链负责：

1. runtime 注册。
2. 心跳。
3. 目标版本拉取。
4. 快照下载。
5. 应用结果回传。

### 4.4 统计回传拓扑

`runtime local stats -> stats upload -> admin aggregate tables -> /console/runtime-nodes`

这条链负责：

1. 请求量与峰值统计。
2. 词条命中统计。
3. 节点级与全局级汇总。

## 5. 核心主链路

### 5.1 词典建设到发布

1. 创建或导入词典记录。
2. 触发词条审核和拼音候选审核。
3. 通过审核后，词条进入可发布状态。
4. 构建 release。
5. 执行发布审核。
6. 进入 `built`、`canary` 或 `published` 生命周期。

### 5.2 发布到 runtime 生效

1. admin 更新 `runtime_control_state`。
2. runtime 心跳时读取目标版本。
3. runtime 下载 snapshot。
4. runtime 将新版本安装到本地目录。
5. runtime 更新 `current.json`。
6. runtime 回传 apply 结果。

### 5.3 runtime 请求执行

1. 调用方提交文本。
2. runtime 根据 `trafficKey` 和当前灰度策略选择 stable 或 canary。
3. runtime 执行字面匹配和拼音召回。
4. runtime 根据规则决定 `replace`、`candidate` 或 `block`。
5. `/api/runtime/correct` 输出单一 `correctedText`。
6. `/api/runtime/correct_cand` 输出整句候选集合 `correctedTexts`。

## 6. 灰度与分流

灰度分流基于以下逻辑：

1. 请求体中的 `trafficKey` 或 `callId` 作为分流键。
2. 对分流键进行 hash。
3. 将结果映射到 0 到 99 的 bucket。
4. 若 bucket 小于当前灰度百分比，则路由到 canary。
5. 否则路由到 stable。

若没有 `trafficKey`，当前请求默认进入 stable。

## 7. 快照下发模式

当前支持 3 种 runtime 快照下发模式：

1. `file`
   - 适合本地验证和共享文件模式。
2. `minio`
   - 适合以对象存储预签名链接下发。
3. `admin_http_signed`
   - 由 admin 直接生成带时效和节点绑定的下载地址。

这 3 种模式只影响“runtime 如何取得 snapshot”，不改变 release 构建和 runtime 安装的主链路。

## 8. 当前架构约束

1. 控制面主库当前仍是 SQLite，适合单节点和原型场景。
2. 当前 admin 是单控制面实例口径，不提供多 admin 协调。
3. runtime 可以多实例运行，但目标版本控制状态仍由单一控制状态来源决定。
4. `/admin` 兼容入口仍存在，但正式后台主入口是 `/console`。
5. 当前测试与发布准备链已经齐备，但真实 K8S 与真实吞吐证据仍受外部条件限制。
