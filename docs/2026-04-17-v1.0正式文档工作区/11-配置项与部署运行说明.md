# ACDP 配置项与部署运行说明

- 文档状态：active
- 适用版本：v1.0
- 文档类型：baseline
- 所属工作区：2026-04-17-v1.0正式文档工作区
- 最后更新时间：2026-04-17
- 责任对象：配置文件、启动命令、部署模式

## 1. 配置设计原则

当前版本明确要求：

1. 配置优先，不写硬编码。
2. 同类配置统一收口到 `prototype/config`。
3. 允许通过 `<key>Env` 方式从环境变量覆盖敏感值。
4. 同一能力的配置只保留一个正式入口。

## 2. 主要配置文件

### 2.1 `app.config.json`

作用：

1. 定义项目名和编码。
2. 定义 host、端口、adminPort、runtimePort。
3. 定义鉴权和 WebSocket 治理。
4. 定义 workspace 与数据目录路径。

关键分组：

1. `project`
2. `server`
3. `auth`
4. `paths`

### 2.2 `artifact_store.config.json`

作用：

1. 定义制品仓连接方式。
2. 定义 runtime 到 admin 的注册和同步配置。
3. 定义 runtime 快照下发模式。

关键分组：

1. `artifactStore`
2. `runtimeControl`
3. `runtimeDelivery`

#### `artifactStore`

主要字段：

1. `provider`
2. `endpoint`
3. `publicBaseUrl`
4. `bucket`
5. `region`
6. `accessStyle`
7. `useSsl`
8. `presignExpiresSeconds`
9. `rootUser`
10. `rootPassword`
11. `accessKey`
12. `secretKey`

#### `runtimeControl`

主要字段：

1. `adminBaseUrl`
2. `nodeId`
3. `nodeName`
4. `nodeEnv`
5. `nodeAddress`
6. `registrationSecret`
7. `heartbeatIntervalSeconds`
8. `syncIntervalSeconds`
9. `nodeOfflineThresholdSeconds`
10. `downloadTimeoutMs`
11. `statsFlushIntervalSeconds`
12. `statsFlushMaxBatchSize`
13. `statsRetentionHours`

#### `runtimeDelivery`

主要字段：

1. `mode`
2. `adminArtifactBaseUrl`
3. `signedUrlSecret`
4. `signedUrlExpiresSeconds`
5. `bindNodeId`
6. `bindConfigVersion`

### 2.3 `access_control.json`

作用：

1. 定义权限点。
2. 定义角色。
3. 定义用户与可分配角色。

### 2.4 `business_properties.json`

作用：定义业务属性配置。

### 2.5 `source_types.json`

作用：定义来源类型配置。

当前要求来源类型至少支持 `scope` 区分，如：

1. `dictionary`
2. `validation`

### 2.6 `governance_policies.json`

作用：定义审批与治理策略，例如：

1. 发布审核所需通过人数。
2. 提交人与审核人分离。
3. 是否要求不同审核人。

### 2.7 `console_navigation.json`

作用：定义 `/console` 左侧目录树。

当前配置内容包括：

1. 一级分组。
2. 二级页面。
3. `path`
4. `navKey`
5. `helpSlug`
6. 目录图标和缩写。

### 2.8 `console_help.json`

作用：定义帮助目录与 slug 元数据。

### 2.9 `validation_feed_connectors.config.json`

作用：定义验证回流接入配置。

当前可承载：

1. `file_inbox`
2. `http_pull_json`

4. `cursor`

## 3. 启动命令

### 3.1 单进程模式

```bash
npm run start:prototype
```

### 3.2 拆分模式

```bash
npm run start:admin
npm run start:runtime
```

### 3.3 多 runtime 实例

```bash
npm run start:runtime:instance -- --instance ACDP_AGENT_001 --port 8791 --node-id ACDP_AGENT_001 --node-name "ACDP_AGENT_001" --node-env test --node-address http://127.0.0.1:8791 --admin-base-url http://127.0.0.1:8788 --registration-secret <secret>
```

说明：

1. `instance` 用于实例目录隔离。
2. `port` 是该 runtime 实例监听端口。
3. `node-id` 必须与备案节点一致。
4. `registration-secret` 必须与备案信息一致。

## 5. 当前支持的部署模式

### 5.1 单机单进程

适合：

1. 本地开发。
2. 最小演示。

### 5.2 单机拆分

适合：

1. admin 与 runtime 分离验证。
2. 快照下发链路验证。

### 5.3 `1 admin + N runtime`

适合：

1. 多 runtime 节点治理验证。
2. 发布与收敛观察验证。

### 5.4 K8S

当前仓库已提供基础模板，但正式集群验证仍受外部条件限制。

## 6. 生产部署时的当前可行性说明

### 6.1 1 admin + 3 runtime，4 台独立服务器

当前架构逻辑上可行，前提是：

1. admin 对外可访问。
2. 每个 runtime 都有独立节点备案。
3. 每个 runtime 都能访问 admin 和快照下载地址。
4. 注册密钥、runtime token 等配置一致。

### 6.2 1 台物理机上部署多个 Pod，共享一个外部 IP

逻辑上可行，但前提是：

1. Pod 之间通过端口或 Service 区分。
2. runtime 对 admin 的注册地址使用可达地址而不是 `127.0.0.1`。
3. runtime 自身 `nodeAddress` 使用 admin 可识别的访问地址。

## 7. 当前配置与部署的关键约束

1. runtime 节点接入前必须完成人工备案。
2. runtime 注册时必须使用与备案记录匹配的 `registration-secret`。
3. 快照下发方式与制品存储方式是两个配置维度，不能混写成一个概念。
4. `/console` 左侧目录树、帮助目录、用户角色权限都已配置化，不应回到硬编码模式。
