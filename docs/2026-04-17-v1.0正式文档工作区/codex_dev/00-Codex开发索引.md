## 1. 作用

本目录不是讲业务，而是讲“如何改这个项目”。

目标是让后续 Codex 在不翻聊天记录的前提下，也能直接知道：

1. 改某个功能先读哪些文档。
2. 改某类代码先看哪些文件。
3. 必须同步哪些接口、页面、配置、表结构和测试。
4. 完成后要跑哪些回归。

## 3. 编写要求

每份文档应尽量写成“拿来即用”的开发依据，而不是泛泛说明。

至少包含：

1. 代码文件清单。
2. 文档清单。
3. 测试清单。
4. 常见风险。
5. 推荐顺序。
