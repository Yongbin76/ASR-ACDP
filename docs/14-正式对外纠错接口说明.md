# 正式对外纠错接口说明

## 1. 文档目的

本文件用于定义 ACDP 当前对外提供的正式纠错接口，供外部系统、调用方服务和后续对接文档使用。

当前正式对外纠错接口包括两类能力、两种协议形态：

1. 主结果纠错
2. 候选整句纠错
3. HTTP
4. WebSocket

两者能力一致，均用于：

- 输入：ASR 原始文本
- 输出：通过本系统修正/增强后的文本

说明：

- 当前正式对外接口已包含：
  - `correct`
  - `correct_cand`
  - `ws correct`
  - `ws correct_cand`

## 2. 接口总览

### 2.1 HTTP 接口

- 方法：`POST`
- 路径：`/api/runtime/correct`
- 方法：`POST`
- 路径：`/api/runtime/correct_cand`

### 2.2 WebSocket 接口

- 方法：`GET` 升级连接
- 路径：`/ws/runtime/correct`
- 方法：`GET` 升级连接
- 路径：`/ws/runtime/correct_cand`

## 3. 通用能力说明

### 3.1 功能定义

接口会基于当前加载的词典快照，对输入文本执行：

- 字面热词纠错
- 拼音增强召回
- 规则评估
- 替换结果输出

### 3.2 正式接口返回范围

正式对外接口按能力返回最小必要结果：

- `correct` / `ws correct`：
  - `correctedText`
- `correct_cand` / `ws correct_cand`：
  - `correctedTexts`

不返回调试明细：

- `matches`
- `candidates`
- `blocked`
- `dictVersion`
- `route`

如果需要调试明细，应使用内部调试接口 `/api/simulate`，不建议对外暴露。

### 3.3 鉴权

如果服务配置了运行时 Bearer Token，则调用方必须携带：

```http
Authorization: Bearer <token>
```

当前环境变量：

```bash
ACDP_RUNTIME_TOKEN=your-token
```

## 4. HTTP 正式纠错接口

### 4.1 路径

```text
POST /api/runtime/correct
```

### 4.2 请求头

必需：

- `Content-Type: application/json`

可选：

- `Authorization: Bearer <token>`

### 4.3 请求体

最小请求体：

```json
{
  "text": "我想咨询旗顺路和市发改委，还有工商认定。"
}
```

扩展请求体：

```json
{
  "text": "我想咨询旗顺路和市发改委，还有工商认定。",
  "trafficKey": "call-001",
  "enablePinyinChannel": true,
  "enablePinyinAutoReplace": true
}
```

### 4.4 输入字段说明

#### `text`

- 含义：ASR 原始文本
- 类型：`string`
- 是否必填：是

#### `trafficKey`

- 含义：灰度分流键
- 类型：`string`
- 是否必填：否
- 说明：
  - 若当前存在启用中的灰度策略，服务会基于该值决定请求落到 `stable` 或 `canary`

#### `enablePinyinChannel`

- 含义：是否启用拼音增强召回
- 类型：`boolean`
- 是否必填：否
- 默认值：`true`

#### `enablePinyinAutoReplace`

- 含义：是否允许拼音通道自动替换
- 类型：`boolean`
- 是否必填：否
- 默认值：`true`

### 4.5 成功响应

```json
{
  "correctedText": "我想咨询祁顺路和上海市发展和改革委员会，还有工伤认定。"
}
```

### 4.6 输出字段说明

#### `correctedText`

- 含义：通过本系统修正/增强后的文本
- 类型：`string`

## 5. HTTP 候选纠错接口

### 5.1 路径

```text
POST /api/runtime/correct_cand
```

### 5.2 请求体

与 `POST /api/runtime/correct` 保持一致：

```json
{
  "text": "我想咨询旗顺路和市发改委，还有工商认定。",
  "trafficKey": "call-001",
  "enablePinyinChannel": true,
  "enablePinyinAutoReplace": true
}
```

### 5.3 成功响应

```json
{
  "correctedTexts": [
    "我想咨询祁顺路和上海市发展和改革委员会，还有工伤认定。",
    "我想咨询岐顺路和上海市发展和改革委员会，还有工伤认定。"
  ]
}
```

### 5.4 输出字段说明

#### `correctedTexts`

- 含义：主结果加候选整句结果集合
- 类型：`string[]`
- 约束：
  - `correctedTexts[0]` 永远是当前主结果
  - 后续最多再返回 3 条候选整句
  - 与主结果重复的候选不会返回

## 6. WebSocket 正式纠错接口

### 5.1 路径

```text
GET /ws/runtime/correct
```

### 5.2 连接地址

本机：

```text
ws://127.0.0.1:8787/ws/runtime/correct
```

远程：

```text
ws://<host>:8787/ws/runtime/correct
```

如果前置 HTTPS 反向代理，则可使用：

```text
wss://<host>/ws/runtime/correct
```

### 5.3 握手请求头

如果启用了 token，需在握手时带：

```http
Authorization: Bearer <token>
```

### 5.4 输入消息

最小消息：

```json
{
  "text": "我想咨询旗顺路和市发改委，还有工商认定。"
}
```

扩展消息：

```json
{
  "text": "我想咨询旗顺路和市发改委，还有工商认定。",
  "trafficKey": "call-001",
  "enablePinyinChannel": true,
  "enablePinyinAutoReplace": true
}
```

### 5.5 输出消息

```json
{
  "correctedText": "我想咨询祁顺路和上海市发展和改革委员会，还有工伤认定。"
}
```

### 5.6 WebSocket 当前治理能力

当前服务已提供：

- 最大连接数限制
- 空闲超时限制
- 最大消息大小限制

这些能力用于避免长连接无限占用资源。

## 8. 错误响应与错误消息

### 6.1 HTTP 错误响应格式

```json
{
  "error": "错误码: 错误描述"
}
```

### 6.2 WebSocket 错误消息格式

```json
{
  "error": "错误码: 错误描述"
}
```

### 6.3 常见错误

#### 缺少 `text`

HTTP / WebSocket 可能返回：

```json
{
  "error": "missing_text: text is required"
}
```

#### 缺少运行时 token

HTTP / WebSocket 可能返回：

```json
{
  "error": "runtime_token_required: runtime token required"
}
```

#### WebSocket 升级请求不合法

可能返回：

```text
HTTP 400 invalid websocket upgrade request
```

#### WebSocket 连接数超限

可能返回：

```text
HTTP 503 websocket connection limit reached
```

#### WebSocket 消息过大

连接可能被关闭，close code 为：

- `1009`

#### WebSocket 空闲超时

连接可能被关闭，close code 为：

- `1001`

## 7. 调用示例

### 7.1 HTTP 示例

#### 无鉴权

```bash
curl -X POST "http://127.0.0.1:8787/api/runtime/correct" \
  -H "Content-Type: application/json" \
  -d '{"text":"我想咨询旗顺路和市发改委，还有工商认定。"}'
```

#### 带鉴权

```bash
curl -X POST "http://127.0.0.1:8787/api/runtime/correct" \
  -H "Authorization: Bearer your-token" \
  -H "Content-Type: application/json" \
  -d '{"text":"我想咨询旗顺路和市发改委，还有工商认定。"}'
```

## 9. 对外开放建议

建议对外只开放：

- `POST /api/runtime/correct`
- `GET /ws/runtime/correct`

不建议对外暴露：

- `/api/simulate`
- `/api/admin/*`
- `/api/runtime/stats`

## 10. 多命中冲突时的选择规则

当输入文本在同一位置命中多个词典候选时，系统不会把多个结果同时返回给正式对外接口，而是会做一次确定性的冲突消解，只保留一个最终结果。

处理顺序如下：

### 10.1 先做规则评估

命中结果在进入最终替换前，会先经过规则层判断：

- 长度规则
- 上下文规则
- 正向/阻断正则
- `candidateOnly`
- 边界策略

规则评估后，命中结果可能被：

- 保留为 `replace`
- 降级为 `candidate`
- 直接标记为 `block`

### 10.2 再做冲突消解

对仍可参与最终替换的命中，系统按以下优先级排序并保留：

1. 起始位置更靠前优先
2. 命中文本长度更长优先
3. 置信度更高优先
4. 字面命中优先于拼音命中
5. 若以上都相同，则按 `canonical` 做稳定兜底排序

### 10.3 正式接口的输出行为

对于：

- `POST /api/runtime/correct`
- `GET /ws/runtime/correct`

正式接口只返回一个最终结果：

- `correctedText`

不会返回：

- 多候选并列结果
- 候选排序列表
- 被淘汰候选

如果需要查看调试明细，应使用内部调试接口 `/api/simulate`。

## 11. 当前边界

当前正式纠错接口已经可用于 MVP 对外调用，但仍有以下边界：

- `200 并发/秒` 目标虽然已有真实主机压测达标结果，但仍建议在目标环境再做一次验收
- WebSocket 当前仅具备基础连接治理，尚未提供更细粒度的 caller quota / blacklist / client identity
- 更复杂的调用方身份治理仍待后续接入
