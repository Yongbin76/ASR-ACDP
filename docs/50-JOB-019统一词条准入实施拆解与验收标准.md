## 2. JOB-019 范围定义

### 2.1 本批目标

建立一套统一词条准入能力，并同步收口：

- 导入中心
- 词条中心
- 审核中心相关说明

使得“什么词可以进入字典、什么词必须被拦住、为什么被拦住”具备统一口径。

### 2.2 本批不做

本批不包含：

- NLP / 模型判词
- 类别体系迁移
- 审核流重构
- 规则模型重构
- 历史主数据全量去重治理
- 数据库唯一约束强制上线

## 3. 交付原则

### 3.1 统一服务层，不做中心级散补丁

不得只在导入中心做规则、词条中心另做一套文案。

必须优先建立统一的服务层准入校验能力，然后由各中心消费。

### 3.2 先让入口统一，再考虑数据库硬化

本批优先级：

1. 应用层准入口径统一
2. 页面和接口返回一致
3. 审核说明可见
4. 最后才讨论数据库唯一约束

### 3.3 blocked 必须真阻断

`blocked` 不能只是红字提示。

必须体现在：

- 导入批次不可确认
- 词条中心不可保存

## 4.2 T02 统一准入校验器

目标：

- 新增统一词条准入校验器
- 输出标准化结果：`blocked / warning / ready`
- 输出结构化 `issues[]`

必须覆盖：

- 基础字段校验
- 字典准入拦截规则
- 业务属性轻量特征校验
- 冲突校验
- 拼音字段校验

建议主责任范围：

- `prototype/src/lib/`

完成标志：

- 任意词条入口都可以复用同一套校验函数

### 4.2.1 建议文件范围

- 新增：`prototype/src/lib/term-admission.js`
- 可能补充：`prototype/src/lib/platform-db.js`
- 新增测试：`prototype/tests/unit/term-admission.test.js`

### 4.2.2 建议实现步骤

1. 在 `prototype/src/lib/term-admission.js` 建立统一入口：
   - `evaluateTermAdmission(db, payload, options)`
   - `summarizeTermAdmission(issues)`
   - `createBlockedAdmissionError(summary)`
2. 统一输入结构，至少覆盖：
   - `categoryCode`
   - `canonicalText`
   - `aliases`
   - `priority`
   - `riskLevel`
   - `replaceMode`
   - `baseConfidence`
   - `pinyinRuntimeMode`
   - `pinyinProfile.customFullPinyinNoTone`
   - `pinyinProfile.alternativeReadings`
3. 在校验器内部建立稳定 `issue code` 集合，至少覆盖：
   - `canonical_required`
   - `category_required`
   - `single_character_blocked`
   - `invalid_risk_level`
   - `invalid_replace_mode`
   - `invalid_pinyin_runtime_mode`
   - `invalid_base_confidence`
   - `invalid_priority`
   - `canonical_hits_existing_alias`
   - `dictionary_phrase_blocked`
   - `dictionary_sentence_blocked`
   - `dictionary_noise_blocked`
   - `poi_road_shape_blocked`
   - `gov_term_shape_warning`
   - `gov_term_shape_blocked`
   - `invalid_custom_pinyin`
   - `invalid_alternative_reading`
   - `duplicate_alternative_reading`
   - `alternative_matches_primary`
4. 对“栏目词 / 说明词 / 流程词”建立显式名单常量，不做动态模型判断。
5. 对 `poi_road` / `gov_term` 建立独立轻量特征判断函数，避免散落在调用层。
6. 把 alias 冲突查询抽成独立 helper，返回结构化 `trace`。
7. 统一输出：
   - `level: blocked | warning | ready`
   - `issues: []`
   - `normalizedInput`
8. 先保证校验器只读数据库、不直接写库。

### 4.2.3 建议数据库读取 helper

若 `term-admission.js` 不直接写 SQL，建议在 `platform-db.js` 补以下只读 helper：

- `findTermByCategoryAndCanonical(db, categoryCode, canonicalText)`
- `findTermsByAliasText(db, aliasText, options)`
- `listAdmissionConflictTraces(db, payload, options)`

说明：

- 这些 helper 只负责查询
- 不应把准入规则逻辑写回 `platform-db.js`

### 4.2.4 T02 验收点

- `term-admission.js` 可独立单测
- 单字标准词在“无错误词 / 含任意单字错误词”时可直接打出 `blocked`
- 单字标准词在“错误词均为非单字”时可通过该条准入校验
- 栏目词 / 说明词名单命中可直接打出 `blocked`
- `poi_road` / `gov_term` 轻量特征可稳定产出 `blocked / warning`
- alias 冲突可以返回结构化 trace
- 不依赖页面层文案拼接

## 4.3 T03 导入中心预览接入

目标：

覆盖点：

- 导入错误导出 / 说明文案

完成标志：

- 预览页不再只靠一条 `errorMessage`
- blocked 行会阻断确认导入

### 4.3.1 建议文件范围

- `prototype/src/lib/import-jobs.js`
- `prototype/src/lib/platform-db.js`
- `prototype/src/lib/console-service.js`

- `prototype/tests/unit/import-jobs.test.js`
- `prototype/tests/unit/console-api.test.js`
- `prototype/tests/unit/console-read.test.js`

### 4.3.2 建议实现步骤

1. 在 `parseStructuredTerms()` 中调用统一准入校验器。
2. 保持统一映射关系：
   - `ready` -> 导入行 `status=ready`
   - `warning` -> 导入行 `status=warning`
   - `blocked` -> 导入行 `status=error`
3. 在 `import_job_rows` 中持久化结构化 issue 数据。
   建议新增字段：
   - `issues_json TEXT NOT NULL DEFAULT '[]'`
4. `createImportJob()` 写入每行 `issues_json`。
5. `listImportJobRows` / `getConsoleImportJobDetail()` / 错误导出接口返回结构化 `issues[]`。
6. `confirmImportJob()` 改成严格确认条件：
   - 只允许 `ready` 和 `warning` 行存在时确认
   - 若批次内仍有 `error/blocked` 行，返回 409 阻断
7. `confirmImportJob()` 在真正写库前，建议对将入库行再跑一次准入校验，防止预览到确认之间数据漂移。
8. `/console/import/jobs/:jobId` 页面增加：
   - blocked / warning 数量摘要
   - 行级 issue 展示
   - 确认按钮禁用态与说明文案
9. 错误报表下载中补 issue code / issue message / trace 摘要列。

### 4.3.3 建议接口变化

建议导入批次详情与行列表新增返回字段：

- `issues`
- `admissionLevel`
- `blockedRowCount`
- `warningRowCount`

若暂不扩散字段命名，至少应做到：

- 每行可读到完整 `issues[]`
- 批次详情能判断“是否允许确认”

### 4.3.4 T03 验收点

- 预览页能明确区分 ready / warning / blocked
- blocked 行存在时，后端拒绝确认导入
- 导入错误下载能带出 issue 信息
- warning 行仍允许导入，但能明确说明风险

## 6. 关单依据

`JOB-019` 只有在以下条件同时满足时才能关闭：

1. 已存在统一词条准入校验器，且导入中心与词条中心共同复用
2. 单字标准词在“无错误词 / 含任意单字错误词”时阻断，在“错误词均为非单字”时允许通过该条准入校验
3. 栏目词 / 说明词 / 流程词首批名单默认拦截
4. `poi_road` / `gov_term` 轻量特征校验已生效
5. `canonicalText` 命中其他词条 alias 时直接阻断，且可溯源
6. `customFullPinyinNoTone` 与 `alternativeReadings` 的校验和行为在导入与词条编辑中保持一致
7. 导入批次中若存在 blocked 行，则不可确认导入
8. 词条中心新建 / 编辑对 blocked 内容不可保存
9. 审核中心至少能读到准入风险摘要
10. 文档和回归测试已同步更新
11. `docs/50-JOB-019统一词条准入实施拆解与验收标准.md` 中 `T02`~`T07` 所列文件范围与验收点已逐项对照完成或显式删减并记录原因

## 7. 明确不作为关单前置条件的事项

以下事项不作为 `JOB-019` 必须完成后才能关单的前置条件：

- 给 `terms` 增加数据库级联合唯一约束
- 历史重复词条一次性清洗完成
- 引入 NLP / 模型判词
- 重做审核流状态机

说明：

- 这些事项重要，但会显著扩大批次范围
- 若在实现中发现它们已成为新的独立工作面，应新开后续 job，而不是无限拖长 `JOB-019`

## 8. 发现阻塞时的拆分原则

若 `JOB-019` 实施中出现以下情况，应拆出新 job：

1. 为支持准入 trace，必须大幅改造导入批次表结构
2. 为支持准入统一，必须先做历史主数据去重
3. 为避免重复写入，必须同步完成数据库唯一约束迁移和历史修复
4. 审核中心需要引入新的审批决策分支，而不仅是读摘要

拆分原则：

- `JOB-019` 保持“统一词条准入与跨中心收口”主线
- 额外的数据清洗、DDL 硬化、审核流升级，按新 job 挂单
