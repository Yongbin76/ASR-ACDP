## 1. 文档目的

本文档用于把 [131-v1.0词典建设域与验证回流域重构讨论稿](./131-v1.0词典建设域与验证回流域重构讨论稿.md) 细化到可直接交给其他 Codex 编码的阶段。

当前只定义：

- 目标对象

- 配置模型

- 编码顺序
- 验收标准

当前不直接进入代码修改。

## 2. 本批目标

把现有“主数据”重构为两个清晰域：

- `词典建设`
- `验证与回流`

并同时满足以下要求：

- `taskId` 成为唯一审核原子对象
- `importJobId` 仅作为可选上下文，不新增“审核批次”
- 手工创建与 CSV 导入共用同一套审核逻辑
- `业务属性` 与 `来源类型` 改为统一配置驱动
- `来源类型` 通过 `scope` 拆开 `dictionary / validation`
- 手工创建保持“保存草稿 -> 提交审核”两步
- 首批直接改页面路由与 API 路由口径

## 6. 统一审核逻辑

本批必须固定以下原则：

- 手工创建与 CSV 导入，不得走两套审核系统
- 所有审核动作最终只作用于 `taskId`
- 页面差异只能是查询视图差异，不能是逻辑差异
- 手工创建提交审核后才生成 `taskId`

统一定义：

- 单条审核：处理 1 个 `taskId`
- 批量审核：处理一组显式勾选的 `taskId`
- 导入审核：处理某个 `importJobId` 下的一组 `taskId`

## 7. 冲突处理规则

## 7.1 冲突识别发生在录入阶段

需要识别：

- `businessAttributeCode` 非法或已停用
- `sourceTypeCode` 非法、已停用或 `scope` 不匹配
- 标准词与现有记录冲突
- alias 与现有 canonical / alias 冲突
- 统一准入规则阻断

建议固定为以下 4 类冲突：

- `exact_match_existing`
- `alias_conflict`
- `attribute_invalid`
- `source_type_invalid`

输出等级：

- `ready`
- `warning`
- `blocked`

说明：

- `blocked` 不生成 `taskId`
- `warning` 可以生成 `taskId`，但必须带冲突摘要
- `exact_match_existing` 不再按“新建词条”处理，而是统一转为“更新现有记录”的任务语义
- 手工创建的 `保存草稿` 也跑同一套冲突识别，但只要不是 `blocked` 就允许保存

## 7.2 冲突展示发生在审核阶段

审核详情必须展示：

- 录入来源
- `importJobId`
- `businessAttributeCode`
- `sourceTypeCode`
- 变更前快照
- 变更后快照
- 冲突摘要

## 8. 页面职责与必备能力

## 8.1 `词典记录`

必备能力：

- 列表查询
- 保存草稿
- 手工创建并提交审核
- 编辑
- 查看当前状态
- 查看关联 `taskId`
- 单条提交审核

不承接：

- CSV 预览
- 样本导入
- 批量审核主入口

补充要求：

- 如果识别到 `exact_match_existing`，页面必须显式提示“当前操作将转为更新现有词条”

## 8.2 `批量导入`

必备能力：

- 上传文件
- 预览
- 行级状态查看
- 冲突查看
- 确认导入
- 查看本批生成的 `taskId`
- 跳转到 `词典审核?importJobId=...`

不承接：

- 手工创建
- 样本导入

## 8.3 `词典审核`

必备能力：

- 单条审核
- 批量勾选审核
- 按 `importJobId` 审核
- 查看冲突摘要
- 查看来源上下文

建议固定 3 个视图：

- `全部待审`
- `手工录入待审`
- `导入批次待审`

## 8.4 `基础配置`

必备能力：

- 业务属性配置维护
- 来源类型配置维护
- 启停
- 顺序调整
- 统一帮助说明

RBAC 要求：

- `JOB-035` 必须同步用户、角色、权限配置
- 首批不强制新增角色类型
- 首批优先复用现有权限编码：
  - `term.read / term.write / term.review.submit / term.review.decide`
  - `review.read / review.decide`
  - `validation.read / validation.write`
  - `system.governance.read / system.governance.manage`
- 页面、路由、帮助改名后，`pageFeature / pageAccess / route permission` 必须同步更新

## 12. 验收标准

只有同时满足以下条件，才算进入可关单状态：

- `词典建设` 与 `验证与回流` 的对象边界不再混淆
- `taskId` 成为唯一审核原子对象
- 手工创建与 CSV 导入共用同一套审核逻辑
- `业务属性` 与 `来源类型` 都改为配置驱动
- `来源类型` 已按 `scope` 切开 `dictionary / validation`
- 冲突在录入阶段识别，在审核阶段展示
- 手工创建保持两步且提交审核后才生成 `taskId`
- 页面与 API 路由首批直接切换到目标口径
- 用户、角色、权限配置已和新页面、新路由、新能力点同步
- 页面、接口、帮助文档口径一致
