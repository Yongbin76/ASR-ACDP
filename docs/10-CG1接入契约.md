# CG1 接入契约

## 1. 目标

本文件定义 ACDP与 CG1 热词规则纠错子组件之间的接入契约，重点明确：

- 输入输出对象
- 运行时调用顺序
- 快照加载接口
- 命中与替换结果格式
- 拼音通道在 CG1 中的位置

## 2. CG1 内部位置

ACDP提供的运行时 bundle 位于 CG1 以下阶段：

1. 时间片排序缓冲
2. 文本规范化
3. 字面 AC 匹配
4. 拼音候选召回
5. 规则打分与冲突消解
6. 替换结果输出
7. 传给 LLM 层

说明：

- 姓名上下文锚定和逐字核对重建器可以复用平台词典能力，但不强依赖 AC 主通道
- 数字和字母仍以 AIBOC 主处理为主

## 3. 输入对象

建议 CG1 在热词层使用统一输入对象：

```json
{
  "callId": "1731562234-276976",
  "role": "1",
  "segId": "63",
  "reqId": "20241114134006804",
  "bg": 47580,
  "ed": 48077,
  "speed": 4.43,
  "rawText": "我想咨询婚姻检查所和工商问题",
  "contextWindow": {
    "leftText": "前文窗口",
    "rightText": ""
  },
  "runtimeOptions": {
    "enablePinyinChannel": true,
    "grayVersion": "2026.03.26-rc1"
  }
}
```

## 4. 输出对象

热词层建议输出统一结构，供 LLM 层、质检和前端使用。

```json
{
  "callId": "1731562234-276976",
  "segId": "63",
  "rawText": "我想咨询婚姻检查所和工商问题",
  "correctedText": "我想咨询婚姻介绍所和工伤问题",
  "dictVersion": "2026.03.26-rc1",
  "normalizerVersion": "v1",
  "spans": [
    {
      "start": 4,
      "end": 8,
      "orig": "婚姻检查所",
      "canonical": "婚姻介绍所",
      "category": "proper_noun",
      "channel": "literal",
      "confidence": 0.95,
      "action": "replace",
      "termId": "TERM_PROPER_0001"
    }
  ],
  "candidates": [],
  "blocked": []
}
```

## 5. Span 字段约定

每个 span 建议包含：

| 字段 | 说明 |
|------|------|
| `start` | 起始下标 |
| `end` | 结束下标 |
| `orig` | 原文命中 |
| `canonical` | 标准词 |
| `category` | 类别 |
| `channel` | `literal` / `pinyin_exact` / `pinyin_fuzzy` |
| `confidence` | 最终置信度 |
| `action` | `replace` / `candidate` / `block` |
| `termId` | 词条 ID |
| `variantId` | 变体 ID |
| `reasons` | 规则打分原因 |

## 6. CG1 对运行时 bundle 的调用方式

建议对 CG1 暴露一个本地库接口，而不是网络调用。

接口示例：

```text
match(input: HotwordRequest) -> HotwordResult
getCurrentVersion() -> RuntimeVersionInfo
reload(version?: string) -> ReloadResult
health() -> RuntimeHealth
```

原因：

- 热词层时延预算很紧
- 不应增加跨服务网络调用
- 快照加载完成后本地内存查询最稳

## 7. 规范化与偏移映射

CG1 调用方必须接受两个事实：

- 运行时匹配是在规范化视图上进行的
- 返回的 span 必须已经映射回原文下标

否则：

- 替换结果会错位
- 前端标注和质检回放会出现下标不一致

## 8. 拼音通道接入原则

### 8.1 默认行为

拼音通道默认行为：

- 字面 AC 未命中时才重点参与
- 优先产生候选
- 高风险短词不自动替换

### 8.2 可放开的场景

满足以下条件时，拼音通道结果可提升为替换：

- 类别为地名、政务专有词、机构名
- 长度大于等于 3
- 上下文正向规则命中
- 无阻断规则
- 历史正确率达标

## 9. 灰度接入

CG1 应支持按请求粒度决定使用哪个 bundle。

推荐策略：

- 若命中灰度策略，使用灰度版本 bundle
- 否则使用稳定版本 bundle

这样可在同一进程内同时维护：

- `stableBundle`
- `canaryBundle`

## 10. 多命中冲突消解契约

当同一位置出现多个候选命中时，热词层不会向 CG1 返回多个并列的最终替换结果，而是只保留一个最终决策。

处理顺序建议与当前原型保持一致：

1. 先执行规则评估
2. 再对剩余可替换命中做冲突消解

冲突消解优先级：

- 起始位置更靠前优先
- 命中长度更长优先
- 置信度更高优先
- 字面命中优先于拼音命中
- 若仍相同，则按稳定排序规则兜底

这意味着：

- 最终 `replace` 结果是单结果决策
- `candidate` 与 `blocked` 可作为辅助观察信息
- 若需要查看被淘汰候选，应走内部调试或观测链路，而不是把多个最终替换结果回传给主流程

## 11. 错误处理契约

### 10.1 匹配失败

如果热词层内部异常：

- 记录结构化日志
- 返回原文直通
- 不阻塞后续 LLM 层

### 10.2 拼音通道失败

如果仅拼音索引加载或查询异常：

- 降级为仅字面 AC
- 记录专项告警

### 10.3 bundle 加载失败

如果新 bundle 加载失败：

- 保持当前 bundle
- 不切换
- 报告失败原因

## 12. 监控字段

CG1 建议输出以下热词层监控字段：

- `dictVersion`
- `literalHitCount`
- `pinyinCandidateCount`
- `replaceCount`
- `candidateCount`
- `blockCount`
- `processingLatencyMs`

## 13. 测试契约

CG1 接入平台后，至少需要以下测试集：

- 单词地名纠错
- 长短词冲突
- 中文上下文阻断
- 拼音候选输出
- 拼音高风险短词不替换
- 热更新后版本切换
- 灰度版本和稳定版本共存

## 14. 建议的首版接入范围

首版接入建议：

- 先只接字面 AC + 规则层
- 拼音通道先开候选模式
- 统计和标注字段一次性接全

等样本反馈稳定后，再扩大拼音自动替换范围。
