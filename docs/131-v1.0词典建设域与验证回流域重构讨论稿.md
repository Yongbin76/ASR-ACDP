# v1.0 词典建设域与验证回流域重构讨论稿

> 口径说明
>
> 本文档中保留 `主数据` 等旧称，仅用于描述为什么需要从旧口径收口到当前：
>
> - `词典建设`
> - `验证与回流`
>
> 这两个独立业务域。

## 1. 文档目的

本文档用于收口当前围绕“原主数据子系统重构”的讨论结论，并形成后续页面、接口、状态和帮助文档统一收口的逻辑基线。

本稿当前只做逻辑定稿，不直接进入编码。

## 2. 修订背景

现有系统中，“主数据”长期同时承载了以下几类不同性质的对象：

- 词典内容本体
- 批量导入过程对象
- 审核流程对象
- 业务属性配置对象
- 样本与回流数据

这导致了几个直接问题：

- 页面职责混乱
- 状态口径混乱
- “手工创建”和“批量导入”看起来像两套逻辑
- 冲突识别和审核动作的边界不清
- `来源类型` 在词典建设域和验证回流域中混用

因此，本稿明确：

- 先把“验证与回流”从原主数据域剥离
- 再把剩余部分收口为“词典建设域”

## 3. 已确认共识

当前已确认的讨论结论如下：

- 同意将现有“主数据”子系统拆分，将“验证与回流”剥离出来
- 同意在页面、帮助和后续文档中，正式使用 `词典建设` 作为替代名称
- 剥离后的“主数据”应承接词典内容的创建、导入和审核
- 词典内容创建包含两种来源：
  - 手工创建
  - 批量导入
- 不新增“审核批次”实体
- `taskId` 是唯一审核原子对象
- `importJobId` 只是可选上下文，不是新的审核对象
- 在 `词典建设` 下新增 `基础配置`
- `业务属性` 和 `来源类型` 都归到 `基础配置`
- `来源类型配置` 必须带 `scope`
- `scope` 在 `JOB-035` 首批编码范围内至少分为：
  - `dictionary`
  - `validation`
- `基础配置` 保持为一个二级页面，页面内用两个 Tab 或两个区块承接：
  - `业务属性配置`
  - `来源类型配置`
- 手工创建保持两步：
  - `保存草稿`
  - `提交审核`
- 手工创建提交审核后才生成 `taskId`
- `warning` 类型允许进入审核，并允许审核员直接通过或驳回
- `业务属性配置` 与 `来源类型配置` 当前只允许新增、编辑、启停，不允许删除已引用项
- `来源类型配置` 在 `JOB-035` 首批编码范围内字段冻结为：
  - `code`
  - `label`
  - `description`
  - `enabled`
  - `sortOrder`
  - `scope`
  - `allowedEntryModes`
- `词典审核` 固定 3 个工作视图：
  - `全部待审`
  - `手工录入待审`
  - `导入批次待审`
- `验证样本导入` 从原“批量导入”逻辑中彻底迁出，归入 `验证与回流`
- 首批直接改页面路由与 API 路由口径，不再把旧路由作为主入口长期保留

## 4. 域边界重构

## 4.1 词典建设域

词典建设域只负责：

- 词典内容录入
- 词典内容导入
- 词典内容审核
- 词典建设基础配置

词典建设域不再负责：

- 验证样本维护
- 外部回流样本接入
- 样本导入
- 发布前验证结果分析

## 4.2 验证与回流域

验证与回流域只负责：

- 验证样本维护
- 外部反馈与回流样本接入
- 样本导入
- 样本与词典命中结果查看
- 发布前/发布后验证回放结果查看

验证与回流域不再负责：

- 词典内容创建
- 词典内容审核
- 词典内容导入

## 4.3 两个域的根本区别

| 维度 | 词典建设域 | 验证与回流域 |
|---|---|---|
| 核心目标 | 建设词典内容 | 验证词典效果 |
| 主对象 | 词典记录 | 验证样本 |
| 是否进入 release | 是 | 否 |
| 是否生成 `taskId` | 是 | 否 |
| 是否使用 `importJobId` | 仅词典导入使用 | 仅样本导入使用 |
| 是否影响 runtime 正式词典 | 直接影响 | 间接影响 |

## 5. 词典建设域对象模型

词典建设域只保留以下对象。

| 对象 | 性质 | 是否主对象 | 核心标识 | 说明 |
|---|---|---|---|---|
| 词典记录 | 业务内容对象 | 是 | `termId` | 一条标准词、一组别名、一组规则 |
| 审核任务 | 审核流程对象 | 否 | `taskId` | 词典记录进入审核流后的唯一审核原子 |
| 导入批次 | 录入过程对象 | 否 | `importJobId` | 只用于承接 CSV 批量录入上下文 |
| 业务属性配置 | 基础配置对象 | 否 | `businessAttributeCode` | 词典分类和业务分层的配置源 |
| 来源类型配置 | 基础配置对象 | 否 | `sourceTypeCode` | 词典建设与验证回流共用的来源类型配置源 |

## 5.1 关于 `taskId`

本稿明确：

- `taskId` 是唯一审核原子对象
- 不管是手工创建还是 CSV 导入，只要进入审核流，最终都落到 `taskId`
- 审核动作只作用于 `taskId`
- 页面可以按来源和上下文做不同视图，但底层审核逻辑必须统一

## 5.2 关于 `importJobId`

本稿明确：

- `importJobId` 不是新的审核对象
- `importJobId` 只是批量导入来源下的一组 `taskId` 的分组上下文
- 手工创建时，`importJobId = null`
- CSV 导入时，`importJobId != null`

因此：

- 单条审核 = 处理一个 `taskId`
- 批量记录审核 = 处理一组被勾选的 `taskId`
- 导入批次审核 = 处理某个 `importJobId` 下的一组 `taskId`

三者底层都是同一套 `taskId` 审核逻辑。

## 6. 统一录入与审核逻辑

词典建设域应统一收口为一条主链路：

`录入来源 -> 标准化 -> 冲突识别 -> 生成审核任务 -> 按 taskId 审核 -> 审核通过后进入下一次 build 输入池`

## 6.1 手工创建

手工创建只是一种录入来源。

进入统一链路后，应至少具备：

- `termId`
- `taskId`
- `importJobId = null`
- `sourceMode = manual`

## 6.2 批量导入

CSV 导入也是一种录入来源。

进入统一链路后，应至少具备：

- `termId`
- `taskId`
- `importJobId != null`
- `sourceMode = import`

## 6.3 审核统一原则

审核系统不再区分“两套逻辑”，只允许存在：

- 同一套任务状态机
- 同一套通过/驳回副作用
- 同一套审计日志
- 不同来源上下文下的不同查询视图

## 7. 冲突识别与处理位置

“冲突”不能只在导入页出现，也不能只在审核页出现，必须分两层处理。

## 7.1 冲突识别发生在录入阶段

录入阶段负责识别：

- 业务属性非法或已停用
- 来源类型非法、已停用或 `scope` 不匹配
- 标准词与现有词典记录冲突
- alias 与现有 canonical / alias 冲突
- 统一准入规则阻断

建议固定为以下 4 类冲突：

- `exact_match_existing`
  - 含义：同一 `businessAttributeCode + canonicalText` 已存在
  - 处理：不再按“新建”处理，统一转为“针对现有记录发起更新任务”
- `alias_conflict`
  - 含义：新 alias 已被其他词典记录占用
  - 处理：记为 `warning`，进入审核时必须展示冲突对象
- `attribute_invalid`
  - 含义：`businessAttributeCode` 非法或已停用
  - 处理：记为 `blocked`
- `source_type_invalid`
  - 含义：`sourceTypeCode` 非法、已停用或 `scope` 不匹配
  - 处理：记为 `blocked`

录入阶段的输出结果只能是 3 类：

- `ready`
  可以进入审核流
- `warning`
  可以进入审核流，但必须保留冲突摘要
- `blocked`
  不进入审核流，必须先修正

说明：

- 对手工录入，这体现在保存前校验和提交审核前校验
- 对 CSV 导入，这体现在预览行状态和导入确认前校验
- 手工录入的“保存草稿”与“提交审核”应共用同一套冲突识别引擎
- 其中：
  - `blocked` 阻断保存草稿与提交审核
  - `warning` 允许保存草稿，也允许提交审核

## 7.2 冲突展示发生在审核阶段

审核阶段不重新定义“冲突规则”，只消费录入阶段生成的冲突摘要。

审核任务详情必须直接展示：

- 来源类型
- 业务属性
- `importJobId`
- 变更前快照
- 变更后快照
- 冲突摘要

这样审核员才能知道自己在审什么，而不是只看到一条抽象任务。

## 8. 页面职责建议

## 8.1 词典建设域页面结构

建议 `词典建设` 下固定为 4 个二级入口：

- `词典记录`
- `批量导入`
- `词典审核`
- `基础配置`

## 8.2 词典记录

职责：

- 查看词典记录
- 手工创建
- 编辑词典内容
- 查看当前审核状态
- 查看当前关联 `taskId`

补充规则：

- `保存草稿` 不生成 `taskId`
- `提交审核` 才生成 `taskId`
- 若识别到 `exact_match_existing`，不再落“新建词条”，而是切换为“更新现有词条”的任务语义

不承接：

- CSV 导入预览
- 批量审核动作
- 验证样本维护

## 8.3 批量导入

职责：

- 上传 CSV
- 生成导入预览
- 查看行级状态和冲突
- 确认导入
- 查看本批生成的 `taskId`
- 进入“本批待审任务”

不承接：

- 手工录入
- 词典审核决策主入口
- 样本导入

## 8.4 词典审核

职责：

- 承接所有词典审核动作
- 只按 `taskId` 处理
- 按来源上下文提供不同视图

建议固定 3 个工作视图：

- `全部待审`
- `手工录入待审`
- `导入批次待审`

视图区别只在查询条件：

- `全部待审`：`status = pending`
- `手工录入待审`：`status = pending AND importJobId is null`
- `导入批次待审`：`status = pending AND importJobId = ?`

## 8.5 基础配置

`基础配置` 只保留一个二级页面，不增加左侧三级导航。

页面内建议使用两个 Tab 或两个配置区块：

- `业务属性配置`
- `来源类型配置`

并固定以下限制：

- 已被引用的配置项不允许物理删除
- 只允许启用 / 停用
- 所有历史记录、任务、导入批次都必须能继续解释原始配置值

## 9. 基础配置模型建议

## 9.1 业务属性配置

建议至少包含：

- `code`
- `label`
- `description`
- `enabled`
- `sortOrder`

用途：

- 决定词典记录属于哪类业务属性
- 驱动新建词条、导入预览、审核详情、筛选条件

## 9.2 来源类型配置

建议至少包含：

- `code`
- `label`
- `description`
- `enabled`
- `sortOrder`
- `scope`
- `allowedEntryModes`

其中：

- `scope` 在 `JOB-035` 首批编码范围内固定为：
  - `dictionary`
  - `validation`
- `allowedEntryModes` 在 `JOB-035` 首批编码范围内至少支持：
  - `manual`
  - `import`

## 9.3 来源类型配置的消费规则

建议固定为：

- `词典建设` 只消费 `scope = dictionary`
- `验证与回流` 只消费 `scope = validation`
- `词典记录` 手工新建页只展示：
  - `scope = dictionary`
  - 且允许 `manual`
- `批量导入` 页只展示：
  - `scope = dictionary`
  - 且允许 `import`
- `样本导入` 页只展示：
  - `scope = validation`
  - 且允许 `import`

这样可以避免把 `cg3 / online_feedback / validation_import` 这类验证来源混到词典建设里。

## 10. 状态机拆分建议

为避免继续混乱，建议严格拆分以下 4 套状态。

## 10.1 词典记录状态

- `draft`
- `pending_review`
- `approved`
- `disabled`

## 10.2 导入批次状态

- `preview_ready`
- `imported`
- `cancelled`

## 10.3 导入行状态

- `ready`
- `warning`
- `error`
- `imported`

## 10.4 审核任务状态

- `pending`
- `approved`
- `rejected`

## 11. 本稿相对旧方案的修正

本稿明确修正以下旧口径：

- 不再把“样本与回流”视为主数据的一部分
- 不再把“手工创建”和“批量导入”拆成两套审核逻辑
- 不新增“审核批次”对象
- 不再允许 `来源类型` 继续作为散落硬编码存在
- 不再允许词典建设和验证回流共用同一套无边界来源类型列表

## 12. 当前已冻结的首批编码边界

以下内容当前已经冻结为 `JOB-035` 首批编码范围边界：

- `来源类型` 在 `JOB-035` 首批编码范围内不扩展更多 `scope`
- `业务属性` 在 `JOB-035` 首批编码范围内不引入更细粒度显示分组
- `验证结果` 在 `JOB-035` 首批编码范围内只作为“验证执行结果视图”存在，不单独成页
- `外部回流` 在 `JOB-035` 首批编码范围内只作为“外部回流样本视图”存在，不单独成页

如后续需要扩展以上内容，统一按新批次重开，不混入 `JOB-035`。

- 词典建设域、验证与回流域、基础配置页的统一口径来源
- 后续若进入编码拆解时的上层设计依据
