# v1.0 测试目标与 Codex 主导测试总方案

> 口径说明
>
> 本文档中若出现 `主数据域 / 内容审核域` 等旧测试口径，当前统一按以下映射理解：
>
> - `主数据域` -> `词典建设 + 验证与回流`
> - `内容审核域` -> `词典审核`

## 1. 文档目的

本文档用于把 ACDP `v1.0` 的测试目标、测试边界、执行方式和资源需求固定下来，作为后续：

- 测试主线挂单
- 测试脚本补齐
- 测试数据准备
- 真实环境验证
- 缺陷回流与修正

的统一基线。

本文档解决 5 个问题：

1. `v1.0` 测试到底要证明什么
2. 为什么测试必须以 Codex 为主
3. 测试分几层、每层覆盖什么
4. 需要哪些资源、工具和数据
5. 什么时候可以给出 go / no-go 结论

## 2. 测试总目标

`v1.0` 测试的目标定义为：

**用 Codex 主导，把 ACDP 从“功能能跑”验证到“可发布、可回滚、可运维、可追责”的正式发布候选系统。**

这个目标拆成 6 项：

### 2.1 正确性目标

- 主数据录入、内容审核、版本发布、运行治理、系统配置 5 个域按设计工作
- 状态流转正确
- 权限与治理策略生效
- 页面与帮助文档口径一致

### 2.2 一致性目标

- 单一真源、派生文档、代码实现和页面展示保持一致
- 同一对象在不同页面不再互相矛盾
- 主数据、审核、发布、运行治理之间边界明确

### 2.3 稳定性目标

- 回归可重复执行
- 关键链路具备 smoke、workflow、unit 多层兜底
- 历史信息不会污染当前判断
- 前端页面不能因为脚本解析错误或启动期异常长期停留在“加载中”

### 2.4 可运维目标

- admin / runtime 可独立启动与观测
- 节点可备案、可注册、可收敛、可验证
- 故障能够被归类、定位和留档

### 2.5 可发布目标

- 至少完成一次真实部署验证
- 至少完成一次真实吞吐与并发验证
- 至少完成一次发布与回滚演练
- 具备完整的发布证据包与 go / no-go 清单

### 2.6 可交接目标

- 其他 Codex 或测试执行人能按文档直接继续推进
- 测试证据、失败归因和修正建议可直接交接

## 3. 为什么以 Codex 为主

当前阶段人员紧张，测试如果继续以人工手工点页面为主，会遇到 4 个问题：

1. 覆盖范围不稳定
2. 回归不可重复
3. 失败归因难沉淀
4. 发布证据不成体系

因此建议采用：

**Codex 主导测试设计、测试实现、测试执行、失败归因和报告输出；人工只负责授权资源、提供外部条件和做最终发布决策。**

## 4. 测试执行原则

### 4.1 Codex-first 原则

- 能自动化的全部优先自动化
- 能脚本化的全部优先脚本化
- 能沉淀成模板的全部优先模板化

### 4.2 分层验证原则

测试至少分为：

- L0 静态一致性
- L1 单元测试
- L2 服务与接口测试
- L3 工作流测试
- L3.5 浏览器真实渲染完成度检查
- L4 部署与运行验证
- L5 性能与容量验证
- L6 发布演练与 go / no-go

### 4.3 证据优先原则

每轮测试都必须产出：

- 执行命令
- 执行结果
- 失败原因
- 风险判断
- 证据路径

### 4.4 缺陷回流原则

测试不是只给结论，还必须形成：

- 缺陷归类
- 修复建议
- 回归建议
- 是否阻塞 `v1.0`

### 4.5 外部条件显式化原则

凡是当前环境做不了的验证，都必须提前写清：

- 依赖什么资源
- 由谁提供
- 如何执行
- 产出什么证据

## 5. 测试分层方案

## 5.1 L0 静态一致性

目标：

- 确认真源、文档、帮助、配置和脚本入口没有漂移

重点：

- `source_of_truth.json`
- `pm:sync / pm:check`
- 帮助文档索引与 `slug`
- 导航名称和帮助目录映射
- 模板与示例文件一致性

现有基础：

- `npm run pm:check`
- `prototype/tests/unit/project-management-docs.test.js`
- `prototype/tests/unit/template-assets-docs.test.js`

## 5.2 L1 单元测试

目标：

- 验证核心域逻辑与状态机

重点：

- 词条准入
- 导入解析
- release 生命周期
- 节点注册治理
- 权限与治理策略
- 帮助配置解析

现有基础：

- `npm run test:unit`

## 5.3 L2 服务与接口测试

目标：

- 验证 admin / console / runtime 的接口 contract

重点：

- console 管理接口
- runtime 对外接口
- 节点注册与心跳
- 发布审核与发布动作
- 帮助 API

现有基础：

- `prototype/tests/unit/console-api.test.js`
- `prototype/tests/unit/server-surfaces.test.js`
- `prototype/tests/unit/runtime-node-registry.test.js`
- `prototype/tests/unit/runtime-control.test.js`
- `prototype/tests/unit/console-help-api.test.js`
- `prototype/tests/unit/api-contract-snapshots.test.js`
- `prototype/src/cli/check-api-contract-snapshots.js`

当前补充说明：

- `contract snapshot` 已落地，不再只是待讨论项
- 当前已固定 console/runtime 关键接口的最小 fixture：
  - `prototype/tests/fixtures/api_contract_snapshots/console_core.json`
  - `prototype/tests/fixtures/api_contract_snapshots/runtime_public.json`

## 5.4 L3 工作流测试

目标：

- 验证跨域主流程闭环

重点：

- 词条新增 -> 审核 -> 构建 -> 审核 -> 发布
- 导入 -> 审核 -> 构建 -> 灰度 / 发布
- 节点备案 -> runtime 注册 -> 收敛 -> 验证
- `file / admin_http_signed / minio` 三种运行快照下发模式的合同与主链路回归
- 发布后风险查看 -> 回滚

现有基础：

- `npm run test:console`
- `prototype/tests/unit/console-workflows.test.js`

## 5.5 L3.5 浏览器真实渲染完成度检查

目标：

- 验证前端脚本真正完成执行，页面不会长期停留在“正在加载”

重点：

- 顶部“当前用户 / 当前角色”完成填充
- 首页工作台卡片完成渲染
- `.loading-card` 消失
- 前端脚本不存在语法错误
- 页面启动阶段异常可被捕获

现状说明：

- 当前已补前端脚本静态语法检查
- 但真实浏览器渲染完成度与首页加载耗时仍需继续补强，不应误判为已完全覆盖
## 5.6 L4 部署与运行验证

目标：

- 验证控制面和数据面在真实或准真实形态下可运行

重点：

- `1 admin + 2/3 runtime`
- 节点备案与注册
- 版本下发与收敛
- `admin_http_signed` 在无共享目录、无 MinIO 场景下的版本安装
- 运行治理页面与实际状态一致

现有基础：

- `npm run verify:multi-runtime`

- `prototype/src/cli/verify-runtime-control.js`

## 6. Codex 主导的执行模式

建议采用“测试工厂”模式：

### 6.1 Codex 负责

- 设计测试方案
- 编写测试脚本
- 执行本地可执行测试
- 归因失败
- 输出测试报告
- 生成缺陷修正建议
- 生成证据包与 go / no-go 草稿

### 6.2 人工负责

- 提供外部环境
- 提供凭据与授权
- 在真实环境执行当前会话环境无法直接执行的命令
- 进行最终发布决策

### 6.3 输出要求

每轮测试都至少输出：

- 测试范围
- 执行命令
- 通过项
- 失败项
- blocker
- 修复建议
- 证据路径

## 7. 资源与工具获取建议

如果后续授权 Codex 获取资源与工具，建议优先级如下：

### 7.1 P0

- 覆盖率工具
- API contract snapshot 工具
- 更稳定的 console e2e 驱动
- 性能测试结果格式化工具

### 7.2 P1

- JSON schema 校验工具
- Markdown 链接 / slug 校验工具
- 证据包自动汇总工具
- 发布前检查脚本

### 7.3 P2

- 视觉回归工具
- 更完整的 K8S 验证脚本
- 日志聚合与报告生成脚本

说明：

- 不建议为了测试再引入一套比业务更复杂的工具链
- `v1.0` 的工具目标是自动化、可重复、可证据化

## 8. 数据与环境准备原则

测试前必须准备：

- 基础测试账号与角色矩阵
- 主数据测试样本
- 导入测试文件
- validation case 样本
- release 构建与发布演练数据
- runtime 节点备案与密钥
- 真实部署与压测环境说明

详细清单见：

- [92-v1.0测试数据与资源准备清单](./92-v1.0测试数据与资源准备清单.md)

## 9. 测试通过标准

建议 `v1.0` 测试 exit criteria 至少包括：

1. `pm:check` 通过
2. `test:unit` 通过
3. `test:console` 通过
4. `smoke:console` 通过
5. 关键 runtime 接口 contract 通过
6. 多 runtime 验证通过
7. 发布演练通过
8. 回滚演练通过
9. 真实吞吐验证有结果
10. 帮助文档与页面、接口口径一致
11. 证据包与 go / no-go 材料齐全
