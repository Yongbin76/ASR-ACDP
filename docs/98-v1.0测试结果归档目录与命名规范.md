# v1.0 测试结果归档目录与命名规范

## 1. 文档目的

本文档用于固定 `JOB-090` 后续进入实际测试执行时的：

- 归档目录
- 批次命名
- 报告文件名
- 证据归档结构

避免后续测试 Codex 每轮测试都重新决定目录和命名方式。

## 2. 总目录约定

统一归档到：

- `prototype/workspace/test_runs/`

说明：

- `host_verification/` 继续保留给宿主验证、部署验证和发布准备证据
- `test_runs/` 用于 `JOB-090` 主线的日常测试批次归档

## 3. 批次目录命名

建议格式：

```text
prototype/workspace/test_runs/
  2026-04-09T15-00-00Z_job090_l0_l3_baseline/
  2026-04-10T09-30-00Z_job090_release_domain/
  2026-04-10T19-00-00Z_job090_runtime_domain/
```

命名建议：

- `时间戳`
- `job090`
- `批次主题`

## 4. 每个测试批次的固定结构

```text
prototype/workspace/test_runs/<batch>/
  commands/
  raw/
  reports/
  defects/
  screenshots/
  summary.json
  report.md
```

### 4.1 `commands/`

保存：

- 每条执行命令
- 环境变量说明
- 执行顺序

### 4.2 `raw/`

保存：

- 原始 stdout / stderr
- 原始 JSON
- 原始 HTTP 响应

### 4.3 `reports/`

保存：

- 格式化后的测试报告
- 分层结果摘要

### 4.4 `defects/`

保存：

- 缺陷报告
- 缺陷回流材料
- 修复后复测记录

### 4.5 `screenshots/`

保存：

- 需要保留的页面截图
- 人工核验截图

## 5. 固定文件

### 5.1 `summary.json`

建议字段：

```json
{
  "batchId": "2026-04-09T15-00-00Z_job090_l0_l3_baseline",
  "jobId": "JOB-090",
  "layers": ["L0", "L1", "L2", "L3"],
  "domains": ["master_data", "release", "runtime", "system", "help"],
  "result": "PASS",
  "defectCount": 0,
  "blockerCount": 0,
  "reportFile": "report.md"
}
```

### 5.2 `report.md`

使用：

- [96-v1.0测试报告模板与缺陷回流模板](./96-v1.0测试报告模板与缺陷回流模板.md)

### 5.3 `defects/index.json`

保存本批所有缺陷索引，建议字段：

- `issueId`
- `severity`
- `domain`
- `layer`
- `status`
- `reportPath`

## 6. 与发布准备证据目录的关系

- `JOB-090` 日常测试批次归档到 `test_runs/`
- `JOB-030` 发布前最终证据继续归档到：
  - `prototype/workspace/host_verification/<timestamp>_v1_release_readiness/`

不要混用。

## 7. 当前用途

本文档当前作为：

- `JOB-090` 的结果归档目录基线
- 后续测试 Codex 与开发 Codex 共享证据路径时的命名约束
