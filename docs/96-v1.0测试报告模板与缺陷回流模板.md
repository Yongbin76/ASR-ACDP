# v1.0 测试报告模板与缺陷回流模板

## 1. 文档目的

本文档用于统一：

- 测试报告怎么写
- 缺陷怎么回流给开发 Codex
- 修正后怎么回传给测试 Codex

建议每轮测试输出使用以下结构：

```md

## 基础信息

- 批次：
- 时间：
- 执行人：
- 测试层：
- 覆盖域：

## 执行清单

- 命令 1：
- 命令 2：

## 结果摘要

- 通过：
- 失败：
- 跳过：
- blocker：

## 失败明细

### ISSUE-001

- severity：
- domain：
- layer：
- symptom：
- expected：
- actual：
- repro：
- suspectedArea：
- recommendedFix：
- requiredRegression：
- blockingRelease：

## 证据路径

- 日志：
- 报告：
- 截图：
- 证据包：

## 结论

- 是否继续：
- 是否阻塞：
- 下一步建议：
```

## 3. 缺陷回流模板

建议测试 Codex 回流给开发 Codex 时，固定包含：

### 3.1 元信息

- `issueId`
- `severity`
- `domain`
- `layer`

### 3.2 问题描述

- 问题摘要
- 复现步骤
- 预期结果
- 实际结果

### 3.3 修正建议

- 疑似影响文件
- 疑似影响接口 / 页面
- 建议最小修正方向

### 3.4 回归建议

- 需要重跑的 unit
- 需要重跑的 workflow
- 是否需要补新测试

## 4. 开发 Codex 回传模板

开发 Codex 完成修正后，建议按以下结构回传：

```md

- 修改点 1：

- 文件：

## 风险说明

- 剩余风险：
- 是否需要测试 Codex 复测：
```

## 5. 缺陷分级模板

### blocker

- 直接阻塞发布
- 主流程不可用
- 数据损坏
- 部署 / 收敛 / 回滚失败

### high

- 主流程可走但结果错误
- 状态表达严重误导
- 权限 / 治理策略失效

### medium

- 功能可用但表达明显不一致
- 测试证据不完整
- 帮助与实现不一致

### low

- 不阻塞当前批次
- 可进入后续维护
