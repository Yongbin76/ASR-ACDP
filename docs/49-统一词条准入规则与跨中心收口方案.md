# 统一词条准入规则与跨中心收口方案

> 历史说明
>
> 本文档对应 `JOB-019` 收口阶段，继续保留为统一准入方案基线。
>
> 文中若出现旧目录或旧中心名称，以 `JOB-035` 后的现行页面与接口口径为准。

## 1. 文档目的

本文档用于明确下一轮 ACDP“统一词条准入”实现基线：

- 不再只在导入中心局部补校验
- 把导入中心、词条中心、审核中心相关口径统一到同一套词条准入规则
- 为后续实现“更激进的字典准入”提供可解释、可回溯、可维护的规则基线

本文档是 `JOB-019` 的设计基线文档。

## 2. 当前问题

### 2.1 当前存在双轨准入

当前系统中，词条进入主数据的两个主要入口是：

1. 导入中心
2. 词条中心手工新建 / 编辑

但两者当前并没有使用同一套准入规则。

现状：

- 导入中心只做了较轻的预览校验：
  - `categoryCode` 必填
  - `canonicalText` 必填
  - 是否命中同类同名已有词条
- 词条中心新建 / 编辑则基本直接调用 `createTerm()` / `updateTerm()`，没有同等强度的“适不适合进字典”判断

这会导致：

- 导入中心拦住的内容，词条中心可能手工放进来
- 词条中心允许保存的内容，导入中心可能只给模糊 warning
- 后续审核员与发布链路收到的是“准入口径不一致”的结果

### 2.2 当前“适不适合入字典”尚未前移

当前实现更多是在词条落库后，再依赖：

- 规则治理
- 拼音治理
- 审核流
- 发布前 gate

来控制风险。

但对于明显不应该进入字典的内容，这个时机太晚。

### 2.3 当前缺少结构化可溯源 issue 模型

现有导入行主要依赖：

- `status`
- `decision`
- `error_code`
- `error_message`

来表达问题。

这对“多问题并存”“跨词条冲突追溯”“页面统一展示”都不够。

### 2.4 当前数据库没有 `(category_code, canonical_text)` 唯一约束

当前 `terms` 表没有针对：

- `category_code`
- `canonical_text`

的联合唯一约束。

因此若只做局部 UI 提示，而不统一收口到服务层，仍有机会写入重复主数据。

## 3. 总体策略

## 3.1 建立统一词条准入层

下一轮应新增一层统一的“词条准入校验器”，并复用于：

1. 导入中心预览
2. 导入中心确认前校验
3. 词条中心新建
4. 词条中心基础信息编辑
5. 词条中心拼音编辑
6. 审核中心只读风险说明

说明：

- 它不是新的业务模块
- 它是所有词条入口共享的一层服务能力

## 3.2 先做规则化准入，不先做 NLP 判词

本轮“更激进的字典准入”采用规则化方案先落地，而不是先引入模型判词。

这里“规则化”指的是：

- 用显式、可解释的准入规则判断是否可入字典
- 每条命中的规则都能明确告诉用户“为什么被拦”“依据是什么”

不采用首轮 NLP/模型判词的原因：

1. 当前项目没有现成的标注集和分类基线
2. 导入中心和词条中心都需要明确原因回显，不能只给一个黑盒结论
3. 首轮目标是统一准入口径，而不是追求高复杂度语义理解
4. 当前数据模型和页面交互更适合先承接“规则命中”而不是“模型置信度”

因此本轮策略是：

- 先用规则把明显不适合进字典的内容前移拦截
- 之后如确有必要，再讨论 NLP/模型辅助分流，但不作为首轮依赖

## 3.3 准入结果采用三级模型

统一词条准入结果采用以下三级：

- `blocked`
- `warning`
- `ready`

解释：

- `blocked`
  当前内容不允许进入正式词条写入流程；导入批次中该行不可确认，词条中心不可保存
- `warning`
  允许继续，但必须把风险和影响展示清楚
- `ready`
  当前未发现阻断性问题

## 4. 统一 issue 模型

## 4.1 issue 结构

建议统一 issue 结构如下：

| 字段 | 说明 |
|---|---|
| `code` | 规则编码，稳定可回溯 |
| `level` | `blocked` / `warning` |
| `field` | 关联字段，如 `canonicalText` / `aliases` / `customFullPinyinNoTone` |
| `message` | 面向用户的中文说明 |
| `evidence` | 命中依据，如实际文本、命中关键词、非法值 |
| `trace` | 冲突对象或来源链路，可为空 |
| `suggestion` | 修复建议 |

## 4.2 trace 结构

对需要追溯的问题，建议返回：

| 字段 | 说明 |
|---|---|
| `termId` | 关联词条 ID |
| `categoryCode` | 关联词条类别 |
| `canonicalText` | 关联词条标准词 |
| `aliasText` | 冲突 alias 文本 |
| `importJobId` | 若来源于导入链路，返回关联导入批次 |
| `sourceFileName` | 来源文件名 |
| `sourceRowNo` | 来源行号 |
| `sourceType` | 来源类型 |

说明：

- 当前库里已有 `term_sources` / `alias_sources` 可作为 trace 主要来源
- 第一阶段不需要新增新的来源主表

## 5. 规则分层

## 5.1 基础字段硬校验

以下问题直接记为 `blocked`：

- `canonicalText` 为空
- `categoryCode` 行内为空且批次默认值也为空
- `riskLevel` 不在允许值内
- `replaceMode` 不在允许值内
- `pinyinRuntimeMode` 不在允许值内
- `baseConfidence` 非数字或不在 `0~1`
- `priority` 非整数或超出允许范围
- `customFullPinyinNoTone` 不是合法无声调拼音
- `alternativeReadings` 内存在非法读音、空值、重复值

## 5.2 更激进的字典准入

### 5.2.1 直接拦截类型

以下内容默认直接 `blocked`：

- 单字标准词在以下任一场景下直接阻断：
  - 未填写错误词 / 别名
  - 错误词 / 别名中存在任意单字
- 明显句子或短句
- 栏目词 / 说明词 / 流程词
- 明显脏数据文本

### 5.2.2 首批栏目词 / 说明词 / 流程词拦截名单

首批建议直接拦截以下高频非词条型内容：

- `办理材料`
- `申请材料`
- `材料清单`
- `办理流程`
- `申请流程`
- `办理指南`
- `办事指南`
- `办理条件`
- `受理条件`
- `申请条件`
- `申请对象`
- `办理地点`
- `联系电话`
- `咨询电话`
- `办公时间`
- `常见问题`
- `注意事项`
- `附件下载`

说明：

- 第一阶段采用显式名单
- 名单可后续增补，但不应先做成黑盒语义分类

### 5.2.3 明显句子 / 脏数据特征

首批建议直接拦截以下形态：

- 含明显句读符号：`，。！？；：`
- 含 URL / 邮箱 / 手机号样式
- 含页码 / 序号 / 注释残留
- 含明显栏目名、备注名、说明名
- 存在连续异常符号或 OCR 残留噪声

## 5.3 业务属性轻量特征校验

第一阶段只对以下两类业务属性加轻量特征校验：

1. `poi_road`
2. `gov_term`

### 5.3.1 路名 `poi_road`

首批建议采用较强约束。

#### 直接通过特征

标准词以典型道路后缀结尾，例如：

- `路`
- `街`
- `巷`
- `弄`
- `道`
- `大道`
- `公路`
- `支路`
- `支弄`
- `胡同`
- `里`

#### 直接拦截特征

以下情况直接 `blocked`：

- 命中栏目词 / 说明词 / 流程词名单
- 明显是机构词、栏目词、句子
- 完全不具备路名特征，且整体不呈现地名道路名形态

### 5.3.2 政府部门 `gov_term`

首批建议采用“强拦截 + 弱 warning”组合。

#### 直接通过特征

标准词含明显机构后缀，例如：

- `委`
- `局`
- `厅`
- `办`
- `中心`
- `大队`
- `总队`
- `法院`
- `检察院`
- `政府`
- `党委`
- `人大`
- `政协`
- `派出所`
- `服务中心`

#### 直接拦截特征

以下情况直接 `blocked`：

- 命中栏目词 / 说明词 / 流程词名单
- 明显是地址、说明语、句子
- 明显不是机构名

#### warning 特征

以下情况给 `warning`：

- 看起来像机构简称或机构片段，但信息不足
- 例如：`发展改革`、`市场监管`

## 5.4 冲突规则

### 5.4.1 已有同类同名标准词

当 `categoryCode + canonicalText` 已在 `terms` 中存在时：

- 结果记为 `warning`
- 处理决策保持 `merge_existing`
- 页面上需要明确说明这是“合并已有词条”，不是新建词条

### 5.4.2 `canonicalText` 命中其他词条 alias

当当前 `canonicalText` 命中其他词条的 `alias_text` 时：

- 结果直接记为 `blocked`
- 需要返回可溯源原因
- trace 中应尽量带回：
  - 命中的 `termId`
  - 对方 `categoryCode`
  - 对方 `canonicalText`
  - 命中的 `aliasText`
  - 来源链路

这是首轮必须落地的强规则。

### 5.4.3 alias 冲突

建议首轮同时定义以下规则：

- alias 命中其他词条 `canonicalText`
  - 建议至少 `warning`
  - 若风险较高，可升 `blocked`
- alias 命中其他词条 `aliasText`
  - 建议 `warning`
- 同批次内部 canonical / alias 互撞
  - 建议 `warning`

说明：

- 这部分首轮可以先分层落地
- 但必须在设计中先明确，不应继续只处理“同类同名标准词”

## 5.5 拼音规则

### 5.5.1 主读音覆写允许批量导入

首轮明确允许：

- 在结构化词条导入中直接填写 `customFullPinyinNoTone`
- 以此覆写标准词主读音

### 5.5.2 主读音与备用读音的边界

- `customFullPinyinNoTone`
  - 用于覆写标准词主读音
- `alternativeReadings`
  - 用于补充备用可识别读音
  - 不替换主读音

### 5.5.3 不填写主读音时的行为

若 `customFullPinyinNoTone` 为空：

- 系统仍按 `canonicalText` 自动生成默认主拼音画像
- 但 `customFullPinyinNoTone` 字段本身保持为空

## 6. 跨中心处理流程调整

## 6.1 导入中心

### 6.1.1 预览阶段

导入预览应先执行统一准入校验。

每一行输出：

- `status`
- `decision`
- `issues[]`
- 需要时输出 `trace`

### 6.1.2 blocked 行处理

若某一行存在 `blocked` issue：

- 该行状态应进入不可导入状态
- 该行不能在“确认导入”阶段继续入库

### 6.1.3 批次确认条件

建议首轮采用更严格策略：

- 只要批次内存在 `blocked` 行，就不允许整批确认导入
- 用户必须修文件后重新上传

### 6.1.4 warning 行处理

`warning` 行可以继续导入，但必须明确说明：

- 为什么是 warning
- 会产生什么影响
- 是否会合并已有词条
- 是否会改动主读音

## 6.2 词条中心

### 6.2.1 新建词条

词条中心新建词条必须走同一套准入校验。

结果要求：

- `blocked`：不允许保存
- `warning`：可按后续实现决定是否允许保存，但必须显式展示
- `ready`：正常保存

### 6.2.2 基础信息编辑

当以下字段发生变更时，必须重新执行统一准入校验：

- `categoryCode`
- `canonicalText`
- `aliases`
- `priority`
- `riskLevel`
- `replaceMode`
- `baseConfidence`
- `sourceType`
- `pinyinRuntimeMode`

### 6.2.3 拼音编辑

词条中心拼音编辑页应沿用同一套拼音字段校验：

- `customFullPinyinNoTone`
- `alternativeReadings`

### 6.2.4 规则编辑

`term_rules` 编辑不属于本轮统一准入的直接判定范围。

说明：

- 本轮重点是“词条能不能进入主数据”
- 规则治理仍作为词条进入后的一层治理能力

## 6.3 审核中心

审核中心不需要在首轮承担准入判断本身，但建议同步展示：

- 该词条 / 导入批次的准入摘要
- 已命中的 warning 项
- 人工主读音覆写说明

这样审核员看到的是“为何此词条值得注意”，而不是只看最终快照。

## 7. 数据与接口建议

## 7.1 import row 建议持久化多 issue

为了支持导入批次详情、错误下载和后续复盘，建议导入链路支持持久化每行 `issues[]`。

推荐方向：

- 在 `import_job_rows` 增加结构化 issue 存储字段

若首轮控范围，也至少应做到：

- API 返回结构化 `issues[]`
- 页面不再只依赖一条 `error_message`

## 7.2 不把本轮目标扩大为新主数据模型迁移

本轮不建议新增新的“准入结果主表”或“词条语义分类表”。

原因：

- 目标是统一词条准入口径
- 不是重做词典主数据模型

## 7.3 数据库唯一约束作为后续硬化项

本轮必须承认一个现实：

- 当前没有 `(category_code, canonical_text)` 唯一约束

建议：

- 首轮先完成应用层统一准入
- 历史脏数据复核完成后，再评估是否追加数据库级唯一约束

说明：

- 数据库级唯一约束很重要
- 但不应在未清点历史重复数据前，直接塞进同一批次导致实现范围失控

## 8. 本轮明确不做的事

以下内容不属于 `JOB-019` 首轮必做范围：

- 引入 NLP / LLM / 分类模型做词条判词
- 新增类别体系迁移，把“常用词”从 `proper_noun` 中拆出
- 重做审核流状态机
- 重做词条规则模型
- 直接把数据库唯一约束硬上到生产口径

## 9. 收口目标

`JOB-019` 收口后，应至少达到：

1. 导入中心与词条中心使用同一套词条准入口径
2. 单字词、栏目词、脏数据、显著不合适词条可在入口前移拦截
3. `canonicalText` 命中其他词条 alias 时可直接阻断且可追溯
4. `poi_road` / `gov_term` 已有轻量特征校验
5. 主读音覆写与备用读音边界在导入与词条编辑中保持一致
6. 审核中心至少能看到准入风险摘要，而不是完全失联
