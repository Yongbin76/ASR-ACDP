本文档用于说明如何把当前已经讨论完成的 `v1.0` 重构主线，正式同步进单一真源。

本文件最初用于说明如何把 `v1.0` 主线同步进单一真源。

- `JOB-024 ~ JOB-030` 已完成真源挂单
- 本文档继续保留，作为为什么这样挂单的说明依据

## 2. 当前建议

建议不要再沿用单个“大杂烩式”重构 job。

建议把 `v1.0` 重构主线拆成：

- `R1` 运行治理域重构
- `R2` 版本发布域重构
- `R3` 主数据域重构
- `R4` 系统配置域重构
- `R5` 帮助域与接口文档体系
- `R6` 工作台重构
- `R7` 真实环境验证与发布准备

这样做的原因：

1. 领域边界清楚
2. 便于逐批推进
3. 便于收尾和回归
4. 不会再把页面 patch、运行治理、发布链路、帮助体系混成一个 job

## 3. 建议的真源挂单方式

已采用：新增一组 `v1.0` 主线 job，而不是继续挂在旧的 `JOB-023` 下面。

### 3.1 建议结构

建议在真源中新增一个新的 stage group，例如：

- `Release-v1.0`

建议 job 结构：

- `V1-R1` 运行治理域重构
- `V1-R2` 版本发布域重构
- `V1-R3` 主数据域重构
- `V1-R4` 系统配置域重构
- `V1-R5` 帮助域与接口文档体系
- `V1-R6` 工作台重构
- `V1-R7` 真实环境验证与发布准备

说明：

- 当前已采用 `JOB-024` ~ `JOB-030` 连续编号

### 3.2 建议状态

首次同步时建议：

- `V1-R1`
  - `in_progress`
- `V1-R2` ~ `V1-R6`
  - `pending`
- `V1-R7`
  - `blocked`

原因：

- 当前应只让 `R1` 正式开工
- `R7` 受真实环境条件影响，应在真源中直接标记为外部依赖阻塞

## 4. 为什么先只细化 R1

当前 `R1 ~ R7` 都已经有 checklist 基线，并且已按域开始细化到文件级。

建议节奏：

1. 先把 `R1 ~ R7` 作为主线挂进真源
2. 只把 `R1` 先细化到：
   - 文件范围
   - 接口范围
   - 页面范围
   - 回归项
3. `R1` 进入编码
4. `R1` 稳定后再细化 `R2`
5. 依次推进

这样能避免：

- 一次性把 7 个主线都细化到文件级，文档体量失控
- 还没编码就反复推翻后面的细节拆解

## 5. 建议的真源字段口径

每个 `v1.0` 主线 job 建议至少包含：

- `stageGroup`
- `title`
- `status`
- `priority`
- `owner`
- `updatedAt`
- `currentGoal`
- `nextAction`
- `externalDependencies`
- `closable`
- `primaryRisk`
- `detailMarkdown`

其中 `detailMarkdown` 至少包含：

- 当前负责人
- 最近更新时间
- 目标
- 本批范围
- 非目标项
- 当前已完成
- 下一步
- 依赖 / 风险
- 完成标准
- checklist

## 6. 建议的收尾策略

### 6.1 单批收尾

每个 `R` 批次完成后：

- 同步更新真源
- 执行 `pm:sync`
- 执行 `pm:check`
- 更新 handoff 文档

### 6.2 主线收尾

`v1.0` 主线不应在 `R1 ~ R6` 都完成前提前视为 ready。

至少要等：

- 运行治理域稳定
- 版本发布域稳定
- 主数据域稳定
- 系统配置域稳定
- 帮助域齐套
- 工作台重做完成

之后才进入 `R7`
