# v1.0 词典建设域接口与字段改造对照表

## 1. 文档目的

本文档用于把“词典建设域”后续编码会遇到的接口、字段和兼容迁移问题一次性对齐，减少后续 Codex 在实现时重复猜测。

## 2. 设计原则

- 不重建审核体系，继续以 `taskId` 为唯一审核原子对象
- 不新增“审核批次”实体
- `importJobId` 只作为导入来源上下文
- 页面路由与 API 路由首批直接切换到目标口径
- 字段口径统一收口，但允许保留一段兼容期
- 兼容期内优先“前端读新、后端兼容旧”

## 5.4 RBAC 与页面能力同步

`JOB-035` 不是单纯页面改名，必须同步以下权限配置层：

- `admin-auth.js`
  - 更新 `pageFeature`
  - 更新 `pageKey`
  - 更新 `pageAccess`
- `admin-surface.js`
  - 更新新路由对应的 `permissionForAdminRoute()` 映射
- `access_control.json`
  - 复核现有角色是否能覆盖新页面能力
  - 不要求首批新增角色类型，但必须保证新页面可被既有角色正确访问

建议首批权限复用口径：

| 页面/能力 | 建议权限 |
|---|---|
| 词典记录查看 | `term.read` |
| 词典记录写入 | `term.write` |
| 词典记录送审 | `term.review.submit` |
| 词典审核查看 | `review.read` |
| 词典审核决策 | `review.decide` |
| 验证样本查看/维护 | `validation.read / validation.write` |
| 基础配置查看 | `system.governance.read` |
| 基础配置维护 | `system.governance.manage` |

| `scope` | 可见页面 | 禁止出现位置 |
|---|---|---|

## 7. 兼容迁移策略

`JOB-035` 首批编码范围建议采用：

- 数据库暂不强制重命名列
- 服务层统一输出新口径字段
- 前端优先切到新口径
- 保留旧字段兼容，直到页面和文档全部迁完
- 页面与 API 路由直接切到新口径；旧路由如需保留，仅允许做短期跳转或别名

## 7.1 返回字段兼容建议

### 词典记录

建议在兼容期内同时返回：

```json
{
  "businessAttributeCode": "gov_term",
  "categoryCode": "gov_term",
  "sourceTypeCode": "manual",
  "sourceType": "manual"
}
```

说明：

- 前端新页面只消费 `businessAttributeCode / sourceTypeCode`
- 老页面暂时还可以读旧字段

### 导入批次

建议在兼容期内同时返回：

```json
{
  "importJobId": "import_job_001",
  "jobId": "import_job_001",
  "sourceTypeCode": "import_csv",
  "sourceType": "import_csv"
}
```

## 7.2 写入兼容建议

兼容期内，后端建议接受两种写法：

- `businessAttributeCode`
- `categoryCode`

以及：

- `sourceTypeCode`
- `sourceType`

但内部统一标准化为目标字段口径。

## 7.3 删除限制

- `业务属性配置`：已被引用时不得物理删除
- `来源类型配置`：已被引用时不得物理删除
- 第一阶段统一改为启停模式

## 8. 文件 ownership 建议

为了让多 Codex 并行编码时不互相打架，建议按以下 ownership 划分。

| 编码批次 | 负责人范围 | 建议文件集合 |
|---|---|---|
| A 后端 contract | 读模型、写模型、接口兼容 | `platform-db.js`、`console-service.js`、`admin-surface.js` |
| B 前端路由与页面 | 页面重命名、区块职责、字段口径 | `console/client/app.js`、`console/client/app.css` |
| C 基础配置 | 业务属性/来源类型配置模型与管理页 | `business-properties.js`、配置文件、`app.js` |
| D 帮助与文档 | 帮助页、接口文档、页面说明 | `docs/help_manuals/*`、`docs/12`、`docs/18` |
| E RBAC 同步 | 页面能力、路由权限、角色配置 | `admin-auth.js`、`admin-surface.js`、`access_control.json`、角色矩阵文档 |
| F 回归 | 单测、workflow、smoke | `prototype/tests/unit/*` |

## 9. 需要显式避免的逻辑漏洞

编码时必须避免以下问题：

- 手工创建和 CSV 导入重新走成两套审核流
- 在审核层重新发明“批次实体”
- `dictionary` 范围的来源类型跑到验证样本页面
- `validation` 范围的来源类型跑到词典建设页面
- 冲突只在导入页显示，审核员看不到
- 冲突只在审核页显示，录入人看不到
- 只改前端字段名，不改服务层输出，导致双口径长期并存
