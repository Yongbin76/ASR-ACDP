# v1.0 覆盖率门槛方案

## 1. 文档目的

本文档用于把 `JOB-090` 中剩余的“覆盖率门槛方案”收口为可执行基线。

目标不是把覆盖率变成唯一质量标准，而是：

- 给 `v1.0` 提供一个最小、稳定、可重复执行的覆盖率门槛
- 避免后续新增测试时完全没有覆盖率下限
- 让 Codex、开发和测试在回归时有统一口径

## 2. 当前策略

`v1.0` 阶段采用：

- Node 内置测试覆盖能力
- 基于 `test:unit` 的统一入口
- 使用**基线门槛**，而不是高压门槛

当前不做：

- 覆盖率平台化托管
- 按 PR 强制历史增量覆盖趋势图
- 按每个模块单独设复杂门槛

## 3. 门槛建议

建议当前采用全仓基线门槛：

- 行覆盖率：`35%`
- 函数覆盖率：`28%`
- 分支覆盖率：`25%`

理由：

1. 当前仓库已有较大体量历史代码
2. `v1.0` 目标是先建立下限，不是一次性追高
3. 与现有 `unit + workflow + smoke + contract snapshot` 组合后，已经能形成基础质量门槛

当前本地实测基线：

- 行覆盖率：`77.27%`
- 函数覆盖率：`82.34%`
- 分支覆盖率：`58.62%`

当前结论：

- 行覆盖率已达标
- 分支覆盖率已达标
- 函数覆盖率已达标
- `npm run test:unit:coverage` 已恢复为有效入口

补充说明：

- 本轮修正点不是单纯补测试，而是修复了 `test:unit:coverage` 脚本缺少 `--test` 的执行错误
- 修复后，覆盖率统计已按完整单测集重新计算

## 4. 执行入口

新增统一入口：

```bash
cd /Codex/ACDP
npm run test:unit:coverage
```

当前脚本基于：

- `node --experimental-test-coverage`
- `--test-coverage-lines`
- `--test-coverage-functions`
- `--test-coverage-branches`

## 5. 使用建议

### 5.1 何时执行

建议在以下场景执行：

1. 测试主线批次收口前
2. 大批量新增测试后
3. 准备进入 `JOB-030` 前

### 5.2 何时不作为阻塞

当前阶段不建议把它作为：

- `JOB-006`
- `JOB-009`
- `JOB-030`

的单独硬阻塞项。

更准确地说：

- 覆盖率门槛是 `JOB-090` 的增强质量项
- 真正阻塞 `v1.0` 的仍然是：
  - 真实部署
  - 真实吞吐
  - 发布前证据收口

## 6. 当前用途

本文档当前用于：

- 把 `JOB-090` 的覆盖率门槛从待选项变成已落地方案
- 作为后续回归与收口时的统一执行依据
