## 1. 文档目的

本文档用于把 ACDP 后续工作的目标从“继续修页面”升级为“为正式 `v1.0` 发布做系统级重构准备”。

本文件回答 4 个问题：

1. `v1.0` 到底要交付什么
2. 当前系统为什么不能直接作为 `v1.0`
3. 重构的目标结构是什么
4. 哪些内容必须在 `v1.0` 前完成，哪些可以延后

## 2. 当前系统现状判断

当前 ACDP 已经具备较完整的原型功能，但距离正式 `v1.0` 仍有明显差距。

主要问题不是“缺几个页面”，而是以下 4 类结构性问题：

### 2.1 领域边界不清

- 词条中心、导入中心、审核中心、发布中心、运行节点中心之间职责交叉
- 主数据、审核、发布、运行治理没有被清晰拆开
- 页面之间会对同一对象给出不同解释

### 2.2 状态机混乱

- 主数据状态、审核状态、release 状态、运行节点状态没有被明确分层
- “当前状态 / 历史状态 / 目标状态 / 风险状态”经常被混读
- `gate`、`review`、`published`、`post-publish risk` 没有彻底收口

### 2.3 后端聚合口径不统一

- 不同页面对同一对象依赖不同来源的数据
- 有些页面读主表，有些页面读聚合结果，有些页面读运行时状态
- 导致前后逻辑矛盾、不同页面结果冲突

### 2.4 前端信息架构建立在不稳定口径上

- 导航与页面职责是按阶段性 patch 长出来的
- 页面之间缺少统一的域划分
- 文案在弥补模型问题，而不是模型本身清晰

## 3. v1.0 发布目标

`v1.0` 的目标不应定义为“页面更顺了”，而应定义为：

**ACDP 可以被真实业务角色持续使用，并能够完成主数据录入 -> 内容审核 -> 版本构建/灰度/正式发布 -> runtime 节点收敛 -> 风险回看 的最小闭环。**

具体拆成 8 个目标：

### 3.1 角色可用

至少支持以下角色连续工作：

- 编辑
- 审核
- 发布
- 运维
- 管理员

且不同角色在不同页面看到的同一对象应保持同一语义。

### 3.2 主流程闭环

至少形成以下闭环：

1. 词条录入 / 批量导入
2. 内容审核
3. release 构建
4. release 审核
5. 灰度 / 正式发布 / 回滚
6. runtime 节点收敛观察
7. 发布后风险回看

### 3.3 状态一致

以下 4 套状态机必须各自独立、语义清楚：

- 主数据状态机
- 内容审核状态机
- release 状态机
- runtime 节点状态机

### 3.4 运行可落地

至少支持：

- `1 admin + 2/3 runtime`
- 节点可备案
- 节点可注册
- 版本可下发
- 收敛可观测

### 3.5 治理可控

- 用户 / 角色 / 权限可管理
- 发布治理策略可配置
- 审批人与发布执行人的分离规则不再硬编码

### 3.6 发布可背书

在正式发布前，必须具备：

- 完整回归
- 在线帮助手册
- 操作手册
- 部署手册
- 回滚手册
- runtime 对外接口帮助
- admin / console 对外接口帮助
- go/no-go 标准

### 3.7 在线文档可交付

`v1.0` 必须具备完整的在线帮助手册体系，而不是零散说明文。

至少要覆盖：

- 页面级帮助
- 流程级帮助
- 运维部署帮助
- 故障排查帮助
- runtime 对外接口帮助
- admin / console 对外接口帮助

### 3.8 用户主文案全面中文化

`v1.0` 阶段，整个后台页面的用户主文案必须全面中文化。

要求：

- 一级导航、二级目录、页面标题、区块标题、按钮、提示、状态说明统一优先中文
- 技术码只在技术详情、辅助信息或调试视图中暴露
- 不再允许把英文枚举和内部 code 直接作为主阅读文案

## 4. v1.0 范围

### 4.1 必须进入 v1.0 的内容

1. 领域边界和状态机收口
2. 主数据管理、内容审核、版本发布、运行治理、系统管理 5 个域的职责划分
3. release 生命周期收口
4. runtime 节点治理修通
5. 多 runtime 基线可落地
6. 关键角色权限与治理策略配置化
7. 文档、测试和发布流程收口
8. 在线帮助手册与接口帮助收口
9. 用户主文案中文化收口

### 4.2 明确延期到 v1.1 的内容

以下内容建议不作为 `v1.0` 硬门槛：

- 多 admin 高可用
- 多环境 deployment matrix
- 复杂审批引擎
- 灰度算法重写
- 大规模视觉精修
- 全量数据库化 IAM 平台

### 4.3 对当前 blocked job 的建议

对 `v1.0` 的影响建议如下：

- `JOB-006`
  建议作为 `v1.0` 硬门槛
- `JOB-009`
  建议作为 `v1.0` 硬门槛
- `JOB-007`
  建议默认延期到 `v1.1`
  除非真实外部 validation feed 是 `v1.0` 的承诺能力

## 5. 重构后的目标结构

建议把 ACDP 重构成以下 6 个域。

### 5.1 主数据域

对象：

- 词条
- 别名
- 拼音画像
- 规则
- 业务属性
- 导入批次

职责：

- 维护可治理的主数据
- 不直接承担 release 语义

### 5.2 内容审核域

对象：

- 词条审核
- 拼音候选审核

职责：

- 判断内容是否可进入可发布输入池
- 不处理 release 审核

### 5.3 版本发布域

对象：

- build
- release review
- canary
- publish
- rollback
- release risk

职责：

- 统一表达版本生命周期
- 把升级前置条件和发布后风险拆开

### 5.4 运行治理域

对象：

- 节点备案
- 节点注册
- 节点状态
- 版本收敛
- 运行验证
- 运行统计

职责：

- 统一表达 runtime 节点是否存在、是否在线、是否对齐、是否异常

### 5.5 系统管理域

对象：

- 用户
- 角色
- 权限
- 治理策略
- 系统配置

职责：

- 不承担业务主流程
- 只负责系统治理和配置管理

### 5.6 帮助手册域

对象：

- 页面帮助
- 业务流程帮助
- 运维帮助
- 排障帮助
- runtime 对外接口帮助
- admin / console 对外接口帮助

职责：

- 提供可被业务、联调、运维直接使用的在线帮助体系
- 不再依赖仓库散落文档和临时说明

## 6. 导航命名与配置驱动原则

### 6.1 导航命名需要再收一轮

后续不建议继续直接沿用当前一批“中心 / 管理 / 工具”混搭命名。

建议原则：

- 一级目录名称尽量简洁
- 二级目录名称直接表达对象或动作
- 少用“中心”作为泛化后缀
- 同一个对象在导航、页面标题、帮助文档中保持同名

建议一级目录候选：

- 工作台
- 主数据
- 内容审核
- 版本发布
- 运行治理
- 系统配置
- 帮助

说明：

- `总览` 可收口为 `工作台`
- `主数据管理` 可收口为 `主数据`
- `系统管理` 可收口为 `系统配置`
- `运行治理` 单独成域，不再与系统配置混读

### 6.2 导航不再硬编码

导航名称、顺序和归口后续建议统一由配置驱动。

建议新增统一导航配置文件，例如：

- `prototype/config/console_navigation.json`

配置建议至少包含：

- 一级目录 key
- 一级目录 label
- 一级目录排序
- 二级目录 key
- 二级目录 label
- 路由 path
- `pageKey`
- `feature`

### 6.3 导航、页面和帮助必须同源

后续应满足：

- 导航名称来自统一配置
- 页面标题与 breadcrumb 使用同一命名源
- 帮助页目录与导航目录保持一致
- 不允许一处叫“业务属性”，另一处又叫“类别编码”

## 7. 必须先收口的状态机

### 6.1 主数据状态机

建议口径：

- `draft`
- `pending_review`
- `approved`
- `disabled`

### 6.2 内容审核状态机

建议口径：

- `pending`
- `approved`
- `rejected`

### 6.3 release 状态机

建议口径：

- `built`
- `canary`
- `published`

说明：

- `built` 是候选快照
- `gate` 只决定是否能从 `built/canary` 继续升级
- `published` 后的问题统一叫“发布后风险”

### 6.4 runtime 节点状态机

建议至少拆成：

- 备案状态
- 注册状态
- 在线状态
- 版本收敛状态
- 异常状态

不能继续只用一个 `status` 字段覆盖全部语义。

## 8. v1.0 go / no-go 标准

只有满足以下条件，才建议正式发布：

1. 主数据、内容审核、版本发布、运行治理、系统管理边界清晰
2. 关键角色可以连续完成自己的工作流
3. 运行节点中心能正确展示全部已备案节点及其注册/在线/收敛状态
4. release 构建、审核、灰度、发布、回滚都真实可走通
5. `JOB-006` 完成
6. `JOB-009` 完成
7. `smoke:console`、`test:console`、`test:unit` 全绿
8. 部署、操作、回滚文档齐全
9. 在线帮助手册可覆盖页面、流程、运维与接口说明
10. 整个后台用户主文案中文化完成

## 9. 当前结论

当前不建议继续围绕现有页面做零碎 patch。

正确方向是：

1. 先定义 `v1.0` 目标与边界
2. 先收口领域边界与状态机
3. 再统一后端聚合 contract
4. 最后再收前端页面壳层

本文件只定义目标与结构，不直接进入编码拆解。
