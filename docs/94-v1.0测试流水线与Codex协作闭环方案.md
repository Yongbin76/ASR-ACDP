# v1.0 测试流水线与 Codex 协作闭环方案

## 1. 文档目的

本文档用于把 `v1.0` 测试阶段中：

- 测试 Codex
- 开发 Codex
- 人工测试 / 运维 / 发布负责人

之间的协作方式固定下来，确保后续能形成：

**测试执行 -> 测试报告 -> 缺陷回流 -> 开发修正 -> 回归复测 -> 证据归档**

的稳定闭环。

## 2. 角色定义

## 2.1 测试 Codex

职责：

- 按测试主线执行 L0 ~ L6 各层验证
- 产出测试报告
- 归类缺陷
- 给出阻塞等级
- 给出最小修复建议
- 指明回归范围
- 组织证据包

不负责：

- 未经确认直接改业务代码
- 代替人工做真实环境授权决策

## 2.2 开发 Codex

职责：

- 消化测试报告
- 修正代码 / 文档 / 配置
- 补回归
- 回写真源与实施文档
- 交回给测试 Codex 复测

不负责：

- 重写测试目标
- 跳过 blocker 直接宣布通过

## 2.3 人工测试 / 运维 / 发布负责人

职责：

- 提供真实环境
- 提供凭据 / 访问条件
- 在需要时执行外部命令
- 审核 go / no-go
- 做最终发布决策

## 3. 流水线阶段

建议按以下顺序推进：

1. `L0` 静态一致性
2. `L1` 单元测试
3. `L2` 服务与接口测试
4. `L3` 工作流测试
5. `L4` 部署与运行验证
6. `L5` 性能与容量验证
7. `L6` 发布演练与 go / no-go

规则：

- 低层失败时，不应直接跳高层
- 真实环境验证前，应先完成本地可完成的所有层

## 4. 测试 Codex 输出合同

测试 Codex 每一轮必须产出：

### 4.1 测试范围

- 本轮覆盖的层
- 本轮覆盖的域
- 本轮执行的脚本 / 页面 / 接口

### 4.2 执行清单

- 命令
- 输入数据
- 环境说明

### 4.3 结果摘要

- 通过项
- 失败项
- 跳过项
- blocker

### 4.4 缺陷列表

每个缺陷至少包含：

- `issueId`
- `severity`
- `domain`
- `layer`
- `summary`
- `symptom`
- `expected`
- `actual`
- `reproSteps`
- `suspectedArea`
- `recommendedFix`
- `requiredRegression`
- `blockingRelease`

### 4.5 证据路径

- 日志
- JSON 结果
- 截图
- 报告
- 证据包目录

## 5. 开发 Codex 输出合同

开发 Codex 收到测试报告后，每轮必须回写：

### 5.1 修正摘要

- 修了什么
- 没修什么
- 为什么没修

### 5.2 改动范围

- 文件路径
- 接口路径
- 页面路径
- 配置文件路径

### 5.3 风险说明

- 影响面
- 是否需要扩回归
- 是否引入新行为

### 5.4 回归清单

- 已执行命令
- 结果
- 仍未通过项

## 6. 缺陷闭环规则

建议固定为 6 步：

1. 测试 Codex 执行测试
2. 测试 Codex 产出报告
3. 开发 Codex 接手修正
4. 开发 Codex 回写修正结果
5. 测试 Codex 针对影响面复测
6. 通过后归档证据并更新状态

## 7. 缺陷分级建议

### P0 / blocker

- 直接阻塞发布
- 主流程断裂
- 状态模型错乱
- 数据污染
- 真实环境无法部署 / 无法收敛 / 无法回滚

### P1 / high

- 主流程可走但结果错误
- 帮助与实现严重不一致
- 权限 / 治理策略失效
- 关键页面信息误导

### P2 / medium

- 主要功能可用，但表达或体验有明显问题
- 测试证据不完整
- 文档缺口影响交接

### P3 / low

- 不阻塞当前阶段
- 可收敛到后续批次

## 8. 目录与文件约定

建议新增或固定：

- `reports/tests/<batch>/`
- `prototype/workspace/host_verification/<timestamp>_.../`
- `docs/` 下的测试 runbook、模板和缺陷归类说明

## 9. 真源与状态更新规则

建议：

- 测试 Codex 不直接把开发 job 改成 `done`
- 开发 Codex 修正完后，先更新自己的 job 状态
- 测试 Codex 复测通过后，再更新测试主线状态

对于 `JOB-090`：

- 当前保持 `pending`
- 等进入测试执行窗口时改为 `in_progress`
