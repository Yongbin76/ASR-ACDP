# v1.0 测试执行顺序与域级 owner 建议

> 口径说明
>
> 本文档中若出现 `主数据域 / 内容审核域` 等旧测试口径，当前统一按以下映射理解：
>
> - `主数据域` -> `词典建设 + 验证与回流`
> - `内容审核域` -> `词典审核`

## 1. 文档目的

本文档用于把 `JOB-090` 后续实际执行时的：

- 测试顺序
- 域级优先级
- 建议 owner

固定下来，减少不同 Codex 并行推进时的冲突和重复。

## 2. 总顺序

建议固定为：

1. `L0` 静态一致性
2. `L1` 单元测试
3. `L2` 服务与接口测试
4. `L3` 工作流测试
5. `L4` 部署与运行验证
6. `L5` 性能与容量验证
7. `L6` 发布演练与 go / no-go

## 3. 域级顺序

建议按以下域顺序推进：

1. 运行治理域
2. 版本发布域
3. 主数据域
4. 系统配置域
5. 帮助域
6. 工作台

原因：

- 运行治理和发布域最容易造成整体认知错乱
- 主数据域依赖前两者稳定
- 帮助域要建立在前面页面语义稳定基础上
- 工作台应最后验证，因为它依赖前面所有域的稳定摘要

## 4. 建议 owner 角色

### 4.1 测试 Codex

负责：

- 测试设计
- 测试执行
- 报告产出
- 缺陷归因

### 4.2 开发 Codex

负责：

- 缺陷修正
- 回归补齐
- 文档与真源同步

### 4.3 人工测试 / 运维

负责：

- 真实环境执行
- 凭据和资源提供
- 发布窗口确认

## 5. 每轮建议的最小批次

### Batch-01

- `L0`
- `L1`
- `L2`

目标：

- 确认基础口径、单元测试和接口 contract 没漂

### Batch-02

- `L3`

目标：

- 确认主流程闭环

### Batch-03

- `L4`

目标：

- 确认 admin / runtime / 多 runtime 可运行
- 确认 `file / admin_http_signed / minio` 下发模式至少在目标环境满足选定方案

### Batch-04

- `L5`

目标：

- 确认性能与容量边界

### Batch-05

- `L6`

目标：

- 形成发布前最终结论

## 6. 当前用途

本文档当前作为：

- `JOB-090` 的执行顺序与 owner 基线
- 后续测试 Codex 和开发 Codex 的推进顺序约束
