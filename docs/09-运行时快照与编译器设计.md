# 运行时快照与编译器设计

## 1. 目标

本文件定义：

- 词典控制面数据如何编译为运行时快照
- 运行时快照包含哪些文件与结构
- AC 自动机和拼音索引如何构建
- 运行时如何热加载和切换

设计原则：

- 运行时只读快照，不查控制面数据库
- 编译期尽量做重活，运行时尽量只做查表和状态机扫描
- 快照必须不可变、可校验、可回滚

## 2. 编译器职责

编译器负责把管理态数据转换成运行态产物。

具体职责：

- 拉取待发布词条、变体、规则和拼音画像
- 执行规范化和校验
- 构建字面 AC 自动机
- 构建拼音精确索引和拼音模糊索引
- 编译上下文规则和优先级配置
- 生成 manifest 和校验哈希
- 产出版本快照

## 3. 编译流程

建议编译流程如下：

1. 加载词条数据
2. 规范化文本
3. 执行发布前校验
4. 按类别分桶
5. 生成字面变体索引
6. 生成拼音画像和拼音索引
7. 编译规则结构
8. 构建多自动机
9. 生成 manifest
10. 生成签名与校验哈希
11. 产出快照并写入制品仓库

## 4. 规范化策略

编译器必须和运行时共享同一套规范化规则。

至少包括：

- Unicode 兼容规范化
- 全半角统一
- 控制字符清理
- 多空白折叠
- 可配置的简繁映射
- 标点归一

注意：

运行时不应自行发明另一套规范化逻辑，否则离线编译和在线匹配会失配。

## 5. 快照文件结构

建议每个发布版本产出一个目录。

```text
snapshot/
  manifest.json
  categories.json
  rules.bin
  terms.min.jsonl
  literal/
    poi.ac.bin
    gov_term.ac.bin
    proper_noun.ac.bin
  pinyin/
    pinyin_exact.idx.bin
    pinyin_fuzzy.idx.bin
    pinyin_meta.json
  checksum.sha256
```

## 6. Manifest 结构

`manifest.json` 建议包含：

```json
{
  "version": "2026.03.26-rc1",
  "builtAt": "2026-03-26T12:00:00Z",
  "schemaVersion": 1,
  "normalizerVersion": "v1",
  "artifactHash": "sha256:xxxx",
  "categories": ["poi", "gov_term", "proper_noun"],
  "features": {
    "pinyinChannel": true,
    "fuzzyPinyin": true
  },
  "files": {
    "rules": "rules.bin",
    "literalPoi": "literal/poi.ac.bin"
  }
}
```

## 7. 字面 AC 快照

### 7.1 为什么按类别拆分

原因：

- 不同类别优先级不同
- 可单独启停和灰度
- 可分别统计效果
- 便于局部回滚和排障

### 7.2 节点负载

AC 节点不建议存完整词条对象，建议只存紧凑引用：

- `variant_ref_id`
- `term_id`
- `category_id`
- `payload_idx`

完整规则和元数据放在独立 payload 表中，运行时再引用。

### 7.3 运行时输出

字面 AC 返回：

- `start`
- `end`
- `variantRefId`
- `termId`
- `categoryCode`
- `baseConfidence`

## 8. 拼音索引快照

### 8.1 精确拼音索引

键建议采用：

- `full_pinyin_no_tone`
- `initials`

值建议为：

- `term_id[]`
- `risk_level`
- `runtime_mode`

### 8.2 模糊拼音索引

键建议为模糊归一后的拼音签名。

示例：

- `gong shang`
- `gong san`
- `gongsang`

模糊索引默认只用于候选召回。

### 8.3 多音字策略

对于多音字：

- 标准读音进精确索引
- 备用读音进候选索引
- 不能无限展开到所有可能读音

## 9. 规则快照

`rules.bin` 或 `rules.json.zst` 中建议保存：

- `term_id -> boundary_policy`
- `term_id -> candidate_only`
- `term_id -> min_len/max_len`
- `term_id -> allow contexts`
- `term_id -> block contexts`
- `term_id -> priority`
- `term_id -> replace_mode`

运行时加载后映射为内存字典。

## 10. 编译输出中的最小词条元数据

`terms.min.jsonl` 建议至少保存：

- `termId`
- `canonicalText`
- `categoryCode`
- `priority`
- `replaceMode`
- `baseConfidence`
- `historicalPrecision`
- `pinyinRuntimeMode`

## 11. 运行时加载流程

建议流程：

1. 读取 manifest
2. 验证 checksum
3. 加载 categories 和 rules
4. 分类别加载字面 AC
5. 加载拼音索引
6. 组装为新的 runtime bundle
7. 预热
8. 原子切换

## 12. 运行时 Bundle 结构

建议运行时内存对象如下：

```text
RuntimeBundle
  - version
  - normalizer
  - literalAutomataByCategory
  - pinyinExactIndex
  - pinyinFuzzyIndex
  - ruleMap
  - termMetaMap
  - grayPolicy
```

## 13. 热切换机制

### 13.1 双缓冲

运行时维护：

- `activeBundle`
- `buildingBundle`

构建成功后：

- 原子交换 `activeBundle`
- 老 bundle 引用计数归零后回收

### 13.2 失败保护

如果新 bundle：

- checksum 不通过
- 文件缺失
- 构建异常
- 预热失败

则直接放弃，不影响当前活跃版本。

## 14. 编译器内部模块

建议拆分：

### 14.1 `SourceLoader`

负责从控制面拉取发布数据。

### 14.2 `NormalizerCompiler`

编译规范化配置。

### 14.3 `LiteralAutomatonCompiler`

负责构建字面 AC。

### 14.4 `PinyinCompiler`

负责生成：

- 词级拼音
- 精确拼音索引
- 模糊拼音签名

### 14.5 `RuleCompiler`

把 JSON 规则压缩为运行时轻量结构。

### 14.6 `ManifestBuilder`

生成 manifest、checksum、版本信息。

## 15. 编译器输出校验

编译完成后建议自动执行：

- 样本文本冒烟验证
- 冲突词验证
- 高风险短词验证
- 拼音冲突统计
- 快照完整性校验

## 16. 性能建议

### 16.1 编译侧

- 允许秒级或分钟级构建
- 构建过程可以占用较多 CPU
- 支持并行按类别编译

### 16.2 运行侧

目标是：

- 字面 AC 查询为 O(n)
- 拼音索引只在必要分支触发
- 规则判断为常数级查表或短窗口匹配

## 17. 快照兼容性

建议通过 `schemaVersion` 管理快照格式。

当快照结构变化时：

- 新旧运行时可通过版本判断兼容性
- 不兼容时禁止加载

## 18. 推荐实现顺序

建议先实现：

1. manifest
2. 字面 AC 快照
3. 规则快照
4. 双缓冲加载器
5. 拼音精确索引
6. 拼音模糊索引
