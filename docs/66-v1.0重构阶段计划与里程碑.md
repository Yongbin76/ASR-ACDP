# v1.0 重构阶段计划与里程碑

> 历史说明
>
> 本文档用于保留 `v1.0` 重构阶段计划讨论。
>
> 若文中阶段名称与当前实际关单顺序不一致，以 `docs/38` 当前 job 状态为准。

## 1. 文档目的

本文档用于把 `v1.0` 重构目标进一步落到阶段计划和时间里程碑上，避免“目标很大，但没有节奏”。

当前日期基线：

- `2026-04-08`

## 2. 建议发布日期

建议把 `v1.0` 的目标时间定为：

- `2026-05-08`：Release Candidate
- `2026-05-12`：正式发布

前提条件：

- `JOB-006` 的真实集群条件在 `2026-04-22` 前准备完成
- `JOB-009` 的真实宿主机压测条件在 `2026-04-22` 前准备完成
- `JOB-007` 默认不作为 `v1.0` 硬门槛

## 3. 阶段拆解

### 阶段 0：目标冻结与文档收口

时间：

- `2026-04-08` ~ `2026-04-10`

目标：

- 冻结 `v1.0` 范围
- 冻结 `v1.0` 不做项
- 确认 `JOB-006 / JOB-009 / JOB-007` 的版本归属

交付：

- `v1.0` 目标文档
- 时间计划文档
- 重构讨论文档
- 在线帮助手册与接口帮助框架文档

### 阶段 1：领域边界与状态机收口

时间：

- `2026-04-11` ~ `2026-04-17`

目标：

- 定义 5 个域
- 定义 4 套状态机
- 统一后端聚合 contract

必须产出：

- 主数据域读模型
- 内容审核域读模型
- 版本发布域读模型
- 运行治理域读模型
- 系统管理域读模型
- 帮助手册域结构

### 阶段 2：运行治理与发布链路重构

时间：

- `2026-04-18` ~ `2026-04-24`

目标：

- 先修 runtime 节点治理
- 再修发布生命周期链路

必须完成：

- 运行节点中心以备案节点为主视图
- 节点注册 / 在线 / 收敛 / 异常口径统一
- release build / review / canary / publish / rollback / post-publish risk 收口

### 阶段 3：主数据域与系统管理域重构

时间：

- `2026-04-25` ~ `2026-05-01`

目标：

- 收口词条中心 / 导入中心 / 业务属性
- 收口用户 / 角色 / 权限 / 治理策略

必须完成：

- 主数据录入链路统一
- 内容审核和主数据关系统一
- 配置驱动入口与页面职责统一
- 页面级帮助与流程级帮助骨架

### 阶段 4：RC 验证

时间：

- `2026-05-02` ~ `2026-05-08`

目标：

- 做 `RC` 候选验证

必须完成：

- 全量回归
- 真实部署验证
- 真实并发验证
- 文档收口
- 帮助手册可直接覆盖主要页面、流程和接口
- go / no-go 判断

### 阶段 5：正式发布

时间：

- `2026-05-12`

目标：

- 发布 `v1.0`

必须完成：

- 发布记录
- 发布说明
- 回滚预案确认

## 4. 关键依赖

### 4.1 必须在阶段 2 前准备完成

- `JOB-006` 所需真实 kubeconfig / registry / target-cluster
- `JOB-009` 所需真实宿主机 / 压测窗口

### 4.2 对外部依赖的处理建议

如果到 `2026-04-22` 仍未准备完成：

- `v1.0` 计划应整体顺延
- 不建议在缺乏真实部署和真实压测证据的情况下硬发正式版

## 5. 建议的阶段验收口径

### 阶段 0 通过条件

- 目标、范围、时间和非目标项已明确

### 阶段 1 通过条件

- 领域边界和状态机文档完成
- 后端统一读模型设计完成

### 阶段 2 通过条件

- 运行治理和发布链路不再出现跨页面冲突

### 阶段 3 通过条件

- 主数据和系统管理域页面职责清晰

### 阶段 4 通过条件

- 所有测试通过
- 真实部署与真实压测证据具备
- 在线帮助手册和接口帮助齐备

### 阶段 5 通过条件

- 正式版本可发布并可回滚

## 6. 当前建议

当前应先停在“阶段 0”。

也就是：

- 先把文档梳理清楚
- 先完成目标讨论
- 先把“中文化”和“在线文档”为硬门槛写清楚
- 再决定是否正式进入编码阶段
