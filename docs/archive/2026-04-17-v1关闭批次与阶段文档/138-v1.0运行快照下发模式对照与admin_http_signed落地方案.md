# v1.0 运行快照下发模式对照与 `admin_http_signed` 落地方案

## 1. 目标

本文档用于把 ACDP 当前 runtime 安装 release 时的快照下发模式讲清楚，并回答以下问题：

- 当前已落地的下发方式有哪些
- `file://` 和 `MinIO` 之外，是否可以直接通过 admin 的 HTTP 下发
- 如果引入 `admin_http_signed`，边界、接口、配置和收尾标准分别是什么
- 三种模式分别适合什么阶段

本文档当前既用于说明三种下发模式，也作为后续测试和运维的选型基线。

注意：

- 当前代码已落地三种模式：
  - `file`
  - `admin_http_signed`
  - `minio`

如需直接进入编码实施，参见：

- [139-JOB-036-admin_http_signed运行快照下发实施拆解与收尾标准](./139-JOB-036-admin_http_signed运行快照下发实施拆解与收尾标准.md)

## 2. 当前现状确认

当前 release 构建和 runtime 安装链路的事实是：

1. admin 在本地生成 release 目录和 `snapshot.json` / `manifest.json`
2. runtime control 下发 `desiredVersion` 和 `artifactMetadata`
3. runtime 按 `artifactMetadata.files[].artifactUrl` 逐个下载文件
4. runtime 校验 checksum，验证 `snapshot.json`，再做本地原子切换

当前 runtime 下载端已经支持两种协议：

- `file://`
- `http(s)://`

当前已新增 admin 侧下载路由：

```text
GET /api/runtime-artifacts/releases/:releaseId/:fileName
```

并已支持：

- 签名校验
- 过期校验
- 节点备案校验
- 受控文件名白名单

## 3. 三种模式定义

### 3.1 `file`

定义：

- admin 和 runtime 通过共享文件系统访问同一份 release 文件
- runtime control 中的 `artifactUrl` 直接是 `file://...`

特点：

- 最简单
- 不需要额外服务
- 只适合同机或共享挂载目录场景

### 3.2 `admin_http_signed`

定义：

- release 文件仍保存在 admin 本地工作区
- admin 暴露一组“短时效签名下载 URL”
- runtime control 返回的 `artifactUrl` 指向 admin 下载接口
- runtime 继续按现有 `http(s)` 下载逻辑取回 `snapshot.json` / `manifest.json`

特点：

- 不需要 MinIO
- 不要求共享文件系统
- 适合 `1 admin + 1~3 runtime` 的小规模拆分部署

### 3.3 `minio`

定义：

- admin 先把 release 制品同步到 MinIO / S3 兼容对象存储
- runtime control 下发对象地址或预签名地址
- runtime 再从对象存储拉取文件

特点：

- 解耦最好
- 扩展性最好
- 但依赖额外组件，配置、运维和排障都更重

## 4. 三种模式对照

| 模式 | 额外组件 | 是否支持跨机 | 落地复杂度 | 运维复杂度 | 推荐阶段 |
|---|---|---|---|---|---|
| `file` | 无 | 否，或依赖共享挂载 | 最低 | 最低 | 本地联调、同机部署 |
| `admin_http_signed` | 无 | 是 | 低 | 中 | `v1.0` 小规模拆分部署 |
| `minio` | MinIO / 对象存储 | 是 | 中 | 高 | 正式生产、较大规模部署 |

## 5. 建议结论

建议按阶段使用，不要试图让一种模式覆盖所有场景：

1. 本地开发、同机验证：
   - 优先用 `file`
2. 当前 `v1.0` 小规模拆分部署：
   - 优先使用 `admin_http_signed`
3. 后续正式生产、跨可用区、更多 runtime：
   - 再切到 `minio`

原因很直接：

- `file` 最省事，但不能自然支持跨机
- `minio` 最标准，但对当前阶段偏重
- `admin_http_signed` 正好补上“跨机但不想先引入对象存储”的中间档

## 6. 为什么不建议做“裸 admin HTTP 直链”

不建议把 release 文件直接公开成固定 URL，让 runtime 无鉴权下载。

原因：

- runtime 当前下载 artifact 时不会额外附加自定义鉴权 header
- 如果做成固定 URL，泄漏面太大
- 节点备案、版本控制和过期控制都无法收住

因此建议的不是“裸 admin HTTP”，而是：

- `admin_http_signed`

也就是：

- admin 为具体节点、具体 release、具体文件生成短时效签名 URL
- runtime 只拿到可直接 GET 的 URL
- admin 在下载接口校验签名、节点、文件名和过期时间

## 7. `admin_http_signed` 推荐落地口径

### 7.1 设计原则

必须同时满足：

- 不新增外部对象存储依赖
- 不要求 runtime 挂共享目录
- 不改 runtime 现有“按 URL 下载并安装”的主流程
- 下载 URL 必须有时效
- 下载 URL 必须可绑定备案节点
- admin 重启不影响已经安装完成的 runtime 对外服务

### 7.2 推荐接口

admin 新增只读下载接口：

```text
GET /api/runtime-artifacts/releases/:releaseId/:fileName?nodeId=...&expires=...&signature=...
```

建议只允许下载以下文件：

- `snapshot.json`
- `manifest.json`
- `package.tar.gz`

其他任意路径一律拒绝，避免变成通用文件服务。

### 7.3 推荐签名参数

建议签名原文至少包含：

- `releaseId`
- `fileName`
- `nodeId`
- `expires`

可选再绑定：

- `configVersion`
- `desiredVersion`

这样可以避免旧链接长期复用，也能减少“节点 A 的链接被节点 B 直接复用”的风险。

### 7.4 推荐配置结构

不建议把 `admin_http_signed` 混进 `artifactStore.provider`。

原因：

- `MinIO` 是“存储方式”
- `admin_http_signed` 是“下发方式”
- 两者不是同一维度

建议新增独立配置段，例如继续放在 [artifact_store.config.json](../prototype/config/artifact_store.config.json) 中：

```json
{
  "runtimeDelivery": {
    "mode": "admin_http_signed",
    "adminArtifactBaseUrl": "http://127.0.0.1:8788",
    "adminArtifactBaseUrlEnv": "ACDP_RUNTIME_ARTIFACT_BASE_URL",
    "signedUrlSecret": "runtime-artifact-local-secret",
    "signedUrlSecretEnv": "ACDP_RUNTIME_ARTIFACT_SIGNED_URL_SECRET",
    "signedUrlExpiresSeconds": 300,
    "bindNodeId": true,
    "bindConfigVersion": true
  }
}
```

推荐模式值：

- `file`
- `admin_http_signed`
- `minio`

这样可以把“制品放哪”和“runtime 从哪拿”分开管理。

### 7.5 推荐 control 下发逻辑

建议按 `runtimeDelivery.mode` 分支：

1. `file`
   - `artifactUrl = file://...`
2. `admin_http_signed`
   - `artifactUrl = admin 生成的短时效签名 URL`
3. `minio`
   - `artifactUrl = MinIO 对象地址或预签名地址`

runtime 安装主流程继续保持不变：

- 仍按 `artifactMetadata.files[].artifactUrl` 下载
- 仍校验 checksum
- 仍本地原子切换

## 9. 收尾标准

当前以以下条件作为落地完成依据：

- `file / admin_http_signed / minio` 三种模式的配置边界明确
- `admin_http_signed` 在无 MinIO、无共享目录场景下可完成版本安装
- 至少完成一轮 `1 admin + 2 runtime` 验证
- 已补单元测试和集成验证
- 帮助文档、部署文档、真源文档同步完成
- 页面和运维说明中明确“当前启用的是哪一种模式”

## 10. 当前结论

对你现在这个阶段，最实用的结论是：

- 同机联调继续优先用 `file`
- 如果要跨机、又不想先把 MinIO 运维链路做重，建议下一步实现 `admin_http_signed`
- `MinIO` 继续保留为后续正式生产形态，而不是当前唯一必须形态
