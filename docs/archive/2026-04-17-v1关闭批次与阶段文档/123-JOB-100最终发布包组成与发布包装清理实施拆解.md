# JOB-100 最终发布包组成与发布包装清理实施拆解

## 1. 目标

在 `v1.0` 正式发布前，把：

- 运行制品包
- 部署交付包
- 发布证据包

的边界彻底分清，并完成正式镜像、部署资产和证据包的打包收口。

## 2. 当前问题

当前主要问题包括：

1. 镜像打包边界偏宽
2. 开发过程文档与正式交付物边界不清
3. 测试脚本与开发脚本是否应进入正式镜像尚未收口
4. 最终发布证据包虽然有模板，但还没有与正式打包动作串成一个收尾批次

### 3.5 真实环境最小交付口径

真实环境推荐最小交付物为：

1. `acdp-admin` 镜像
2. `acdp-runtime` 镜像
3. `release_bundle_templates/deployment_bundle/` 生成出的部署交付包
4. `release_bundle_templates/release_notes_bundle/` 生成出的发布说明包

不建议作为真实环境最小交付物的内容：

1. 整个项目目录
2. `prototype/workspace*`
3. `prototype/tests`
4. `project_management`
5. `docs/archive`
6. `SESSION_HANDOFF.md`
7. `NEXT_STEPS.md`
8. `acdp-prototype` 一体镜像

说明：

1. `acdp-prototype` 保留给本地一体联调和演示
2. 真实环境默认只交付 admin/runtime 双镜像

### 3.6 真实环境推荐部署口径

推荐口径如下：

1. `1 admin + N runtime`
2. `admin` 单实例
3. `runtime` 多实例
4. 快照下发优先使用 `admin_http_signed`
5. 节点先备案，再部署 runtime

### 3.7 最小部署顺序

1. 构建并推送 `acdp-admin` / `acdp-runtime`
2. 生成 release bundle
3. 部署 admin
4. 验证 admin 的 `/console`、管理接口和 runtime artifact 下载基地址
5. 在 admin 中完成人工节点备案
6. 按备案信息部署 runtime
7. 验证 runtime register / heartbeat / desiredVersion / apply-result

### 3.8 最小验证命令

admin：

```bash
curl -i http://<admin-host>:8788/admin
curl -i http://<admin-host>:8788/console
curl -H 'x-role: dict_admin' -H 'x-operator: admin_verify' \
  http://<admin-host>:8788/api/console/overview
```

runtime：

```bash
curl http://<runtime-host>:8787/health
curl http://<runtime-host>:8787/api/runtime/current
```

## 5. 收尾标准

`JOB-100` 只有在以下条件同时满足时才可关闭：

1. 最终发布包三类交付物边界明确
2. 正式镜像不再混入无关过程文档和运行产物
3. 发布证据包结构固定
4. 发布手册、回滚手册、go / no-go 模板与最终打包边界一致
5. 打包相关回归通过
6. 真实环境最小交付口径明确
