# JOB-099 零数据底座初始化与系统清库实施说明

## 1. 目标

本文档用于固定 ACDP 从“已有演示/测试/导入/发布残留数据”恢复到“零数据底座”的执行口径。

这里的`零数据底座`指：

- 主库表结构仍存在
- 系统可以正常启动
- 但库内没有词条、样本、导入批次、审核任务、版本和节点数据
- 启动 admin 后也不会自动回灌默认词条、默认样本或初始 release

## 2. 与标准初始化态的区别

### 2.1 标准初始化态

执行：

```bash
npm run setup:prototype
```

结果通常为：

- 有 seed 词条
- 有默认 validation case
- 有 1 个 built release

### 2.2 零数据底座

结果应为：

- `terms = 0`
- `validation_cases = 0`
- `releases = 0`
- 启动 `npm run start:admin` 后仍保持为 `0`

## 2.3 与 JOB-098 的关系

`JOB-098` 与 `JOB-099` 的关系如下：

1. `JOB-098` 只负责清理 `prototype/workspace-*`
2. `JOB-098` 的目标是磁盘瘦身，不是主系统清库
3. `JOB-099` 定义“零数据底座”全流程
4. `JOB-099` 的第一步可以直接复用 `JOB-098` 的执行结果

因此：

1. 只完成 `JOB-098`，不等于完成零数据底座
2. 若要做完整零数据化，应执行 `JOB-099` 的第二步和第三步

## 2.4 当前状态说明

需要明确：

1. `JOB-099` 记录的是一次已经完成的零数据化执行
2. 它本质上是“可重复执行的重置 runbook”
3. 它不保证系统后续永远保持零数据状态

也就是说，只要后续再次发生以下任一动作：

1. 重新录入词条
2. 导入批次
3. 创建或发布 release
4. 启动 runtime 实例并重新生成本地状态
5. 执行 `prepare:data / bootstrap:db / setup:prototype`

当前系统就可能重新离开零数据态。

因此：

1. `JOB-099 done` 代表 runbook 已定义且曾执行成功
2. 不代表当前时刻数据库计数一定仍为 `0`

## 3. 必须同时处理的自动回灌入口

要把系统真正切到零数据底座，不能只删 `platform.db`。

还必须同时处理这两个自动回灌源：

1. `prototype/workspace/catalog/seed_terms.json`
2. `prototype/config/release_validation_cases.json`

原因：

- `prototype/src/server.js` 在空库启动时会读取 `seed_terms.json` 并自动导入词条
- `prototype/src/lib/platform-db.js` 在 `validation_cases` 为空时会读取 `release_validation_cases.json` 并自动灌入默认样本

所以零数据底座要求：

- `seed_terms.json` 必须是 `[]`
- `release_validation_cases.json` 必须是 `[]`

## 4. 执行范围

本任务清理的对象包括：

- 第一步历史测试/验证工作区：
  - `prototype/workspace-*`
- 主工作区数据库：
  - `prototype/workspace/platform.db`
  - `prototype/workspace/platform.db-wal`
  - `prototype/workspace/platform.db-shm`
- 主工作区运行态目录：
  - `prototype/workspace/import_jobs`
  - `prototype/workspace/releases`
  - `prototype/workspace/runtime_artifacts`
  - `prototype/workspace/runtime_state`
  - `prototype/workspace/validation_feeds`
  - `prototype/workspace/host_verification`
  - `prototype/workspace/test_runs`
  - `prototype/workspace/release_bundle_preview`
- runtime 实例工作区：
  - `prototype/runtime_instances/*/workspace`

默认保留：

- `prototype/workspace/backups`
- 历史备份库
- 镜像构建日志

## 5. 推荐执行步骤

1. 如尚未执行 `JOB-098`，先清理 `prototype/workspace-*`
2. 备份当前主库 `platform.db`
3. 备份 `prototype/workspace/catalog/seed_terms.json`
4. 备份 `prototype/config/release_validation_cases.json`
5. 确认当前 service 管理脚本下没有运行中的 `prototype/admin/runtime`
6. 清空 runtime 实例工作区
7. 清空主工作区数据库及运行态目录
8. 将以下文件置空为 `[]`
   - `prototype/workspace/catalog/seed_terms.json`
   - `prototype/config/release_validation_cases.json`
9. 只重新打开数据库，让 SQLite 建空表结构
10. 回读关键表计数，确认全部为 `0`
11. 再启动 `npm run start:admin`，确认系统不会自动回灌数据

## 6. 完成标准

以下条件必须同时满足，`JOB-099` 才可视为完成：

- `prototype/workspace-*` 已清理，或已有 `JOB-098` 结果可复用
- 已保留 1 份 `platform.db` 备份
- 已保留 1 份 `seed_terms.json` 备份
- 已保留 1 份 `release_validation_cases.json` 备份
- `terms / aliases / import_jobs / review_tasks / releases / validation_cases / runtime_nodes / runtime_node_registry` 全为 `0`
- `seed_terms.json` 为 `[]`
- `release_validation_cases.json` 为 `[]`
- 主工作区无历史 `release / import job / validation feed / host verification / test run` 残留
- runtime 实例工作区已清空
- `npm run service:status` 返回 `stopped`
- 后续单独执行 `npm run start:admin` 后，数据库计数仍保持为 `0`

## 7. 验证命令

```bash
cd /Codex/ACDP

node - <<'EOF'
const { DatabaseSync } = require('node:sqlite');
const db = new DatabaseSync('/Codex/ACDP/prototype/workspace/platform.db');
const tables = ['terms','aliases','import_jobs','review_tasks','releases','validation_cases','runtime_nodes','runtime_node_registry'];
const out = {};
for (const t of tables) out[t] = db.prepare(`SELECT COUNT(*) AS c FROM ${t}`).get().c;
console.log(JSON.stringify(out, null, 2));
db.close();
EOF
```


## 8. 注意事项

- 进入零数据底座后，不应再执行：

```bash
npm run prepare:data
npm run bootstrap:db
npm run setup:prototype
```

因为这些命令会重新生成标准初始化数据。

- 如果后续重新启动 runtime 实例：

```bash
npm run start:runtime:instance -- ...
```

则对应实例的 `workspace` 中会重新生成本地状态、快照和统计文件，这不表示 `JOB-099` 文档错误，而表示系统已经离开了“零数据底座”状态。

- 如果需要回到“标准初始化态”，应明确执行：

```bash
npm run setup:prototype
```

或手动按：

```bash
npm run prepare:data
npm run bootstrap:db
npm run build:snapshot
```

## 9. 备份与恢复

执行 `JOB-099` 前，标准备份口径应保留以下 3 份备份：

- 1 份 `prototype/workspace/platform.db` 备份
- 1 份 `prototype/workspace/catalog/seed_terms.json` 备份
- 1 份 `prototype/config/release_validation_cases.json` 备份

本机历史执行中，曾额外保留过两份大体量主库备份：

- `prototype/workspace/backups/platform.db.init-*.bak`
- `prototype/workspace/backups/platform.db.reset-*.bak`

这属于当时的历史执行结果，不再作为 `JOB-099` 的标准动作模板。

如果当前已确认这些历史备份文件不再具备回滚价值，可以单独清理；
只要 `JOB-099` 的备份策略、执行步骤和恢复口径仍然保留在文档中，`JOB-099` 本身的 runbook 完整性不受影响。

如需恢复旧库或恢复默认种子，应优先从这些备份回滚，而不是手工重新拼装数据。
