## 1. 文档目的

本文件用于定义 ACDP 下一阶段后台管理系统的重构方向，重点解决：

- 当前后台以技术模块组织，用户不知道如何使用
- `terms`、`aliases`、审核、发布、验证之间缺少统一业务闭环
- 初始化导库链路是工程脚本，不是用户可操作流程
- 人工维护链路分散，缺少统一管理归口

本方案只定义新后台的目标形态、信息架构、数据链路和实施计划。

本方案不直接替换当前原型后台。

## 2. 核心约束

### 2.1 双轨并行

新后台重构必须遵循以下硬约束：

1. 不影响当前已经存在的 `/admin` 后台页面与现有能力
2. 当前 `/admin` 和 `/test-client` 继续保留，作为 MVP 演示版本对外提供
3. 新后台必须独立建设，不以“直接改造当前单页原型”为前提
4. 新后台上线前，现有 MVP 能力必须保持可运行、可演示、可回退

### 2.2 用户优先

新后台必须从“用户任务”组织，而不是从“技术模块”组织。

用户看到的应该是：

- 去哪里导入词条
- 去哪里维护词条和错误词
- 去哪里审核
- 去哪里发布
- 去哪里看验证失败与回流

而不是：

- SQLite 表
- 独立 API
- 零散 sheet/tab

### 2.3 领域统一

新后台必须把 `terms`、`aliases`、`term_rules`、`pinyin_profiles` 统一归口到“词条管理”之下。

用户操作对象应该是“词条”，而不是分别管理若干技术表。

## 3. 当前问题总结

基于当前原型，主要问题如下：

### 3.1 信息架构问题

- 页面按“运行时/词条规则/拼音治理/发布治理/校验样本/审计”分 sheet
- 这是技术实现视角，不是用户任务视角
- 用户不知道“我想新增一个词”应走哪条完整路径

### 3.2 数据归口问题

- `terms` 是标准词主表
- `aliases` 是错误词/别名表
- 但在用户侧没有被收敛为“一个词条对象”
- 用户难以理解“种子词典”和“错误词库”的关系

### 3.3 导入链路问题

- 初始化导库依赖 `prepare-data` 和 `bootstrap-db` 脚本
- 这是工程链路，不是后台产品流程
- 用户无法在后台完成“上传原始文件 -> 清洗 -> 预览 -> 导入 -> 审核”的一条龙闭环

### 3.4 人工维护问题

- 现有人工维护可用，但入口分散
- 缺少词条详情页和统一编辑页
- 审核、拼音、规则、发布状态没有在一个页面集中呈现

### 3.5 发布与验证问题

- `validation_cases` 与词典主数据关系不直观
- release gate 虽已实现，但用户不容易理解验证失败意味着什么、应回到哪里修

## 4. 重构目标

新后台的目标不是“把现有页面做得更漂亮”，而是构建一个真正可使用的词典运营控制面。

目标包括：

1. 统一词条管理归口
2. 打通批量导入与人工维护两条主链路
3. 形成“导入/维护 -> 审核 -> 发布 -> 验证 -> 回流”的完整闭环
4. 让用户不需要理解底层表结构，也能完成平台操作
5. 保留现有 MVP 页面用于演示和兜底

## 5. 新后台的核心业务对象

从用户视角，新后台只暴露以下核心对象：

### 5.1 词条

词条是最核心对象，统一承载：

- 标准词
- 类别
- 错误词/别名
- 规则
- 拼音画像
- 来源
- 状态
- 审核状态
- 发布状态
- 验证表现

后台中不再把 `terms` 和 `aliases` 作为两个并列主入口暴露给用户。

### 5.2 导入批次

用于承接批量导入过程中的中间态。

一个导入批次至少应包含：

- 导入类型
- 原始文件
- 清洗结果
- 预览结果
- 错误记录
- 确认导入状态
- 最终写入结果

### 5.3 审核任务

统一承接：

- 词条审核
- 拼音候选审核
- 发布审核
- 批量导入确认任务

### 5.4 发布版本

统一承接：

- 构建
- 审核
- 灰度
- 全量
- 回滚
- 验证结果

### 5.5 验证样本

单独作为“样本与回流中心”的对象，承接：

- 业务验证样本
- 外部反馈样本
- 发布前回放验证
- 验证失败的修复闭环

## 6. 新后台信息架构

建议新后台一级导航调整为以下 6 个模块。

### 6.1 总览

面向运营、审核、发布人员展示：

- 当前词条规模
- 错误词规模
- 待审核数量
- 当前发布版本
- 最近一次验证结果
- 最近一次灰度/回滚状态

### 6.2 词条中心

作为 `terms + aliases + rules + pinyin` 的统一管理归口。

包括：

- 词条列表
- 词条详情
- 新建词条
- 编辑词条
- 错误词/别名管理
- 规则管理
- 拼音治理
- 来源展示
- 审核/发布/验证关联展示

### 6.3 导入中心

把当前初始化导库链路产品化。

包括：

- 原始文件上传
- 数据类型识别
- 清洗预览
- 去重/冲突提示
- 错误行反馈
- 导入确认
- 导入批次记录
- 导入历史与重放

### 6.4 审核中心

统一展示所有待处理任务。

包括：

- 词条审核
- 拼音候选审核
- 发布审核
- 导入确认任务

### 6.5 发布中心

统一承接：

- release 构建
- release 审核
- release 灰度
- release 全量
- release 回滚
- gate blocker 查看
- 验证结果查看

### 6.6 样本与回流中心

统一管理：

- `validation_cases`
- feed 来源样本
- 外部反馈样本
- 业务回放样本
- 验证失败样本
- 问题回流定位

## 7. 两条主链路的重构方案

## 7.1 批量导入链路

目标：把“原始数据文件 -> 清洗 -> 导库”做成可操作的一条龙闭环。

### 7.1.1 用户视角流程

1. 进入导入中心
2. 选择导入类型
3. 上传原始文件
4. 系统自动清洗和标准化
5. 用户预览将生成的词条和错误词
6. 查看重复、冲突、异常记录
7. 确认导入
8. 数据写入词条中心
9. 进入审核与发布流程

### 7.1.2 需要补齐的能力

- 文件上传入口
- 导入模板定义与模板下载入口
- 每类导入文件的格式约束、字段说明和示例
- 导入批次模型
- 原始文件存档
- 清洗预览能力
- 行级错误回显
- 导入确认能力
- 导入结果可追踪

### 7.1.3 技术改造建议

建议新增 staging / import 层，不直接让原始文件一步写入正式词条表。

建议新增对象：

- `import_jobs`
- `import_job_files`
- `import_job_rows`
- `import_job_results`

建议处理流程：

1. 原始文件进入导入批次
2. 清洗为 staging 记录
3. 用户确认后写入 `terms/aliases`
4. 自动生成后续审核任务

补充要求：

- 所有上传型导入都必须先定义清楚文件格式
- 每种导入类型都必须提供模板下载
- 模板中必须包含字段说明、示例值和必填/选填标记
- 用户在上传前就能看到格式要求，而不是等到导入失败后再被动修复

## 7.2 人工维护链路

目标：让人工维护从“零散操作”变成直观、连贯的词条维护流程。

### 7.2.1 用户视角流程

1. 进入词条中心
2. 新建或搜索词条
3. 填写标准词
4. 添加错误词/别名
5. 编辑规则
6. 编辑拼音画像
7. 提交审核
8. 审核通过后进入发布

### 7.2.2 页面建议

词条详情页建议统一包含以下区域：

- 基本信息
- 错误词/别名
- 规则
- 拼音
- 来源信息
- 审核历史
- 发布历史
- 验证表现

也就是说，一个词条应在一个主页面内完成绝大多数操作。

## 8. 新后台与现有 MVP 的关系

## 8.1 保留现有 MVP

现有：

- `GET /admin`
- `GET /test-client`

继续保留为：

- MVP 演示版本
- 开发联调版本
- 能力兜底版本

## 8.2 新后台独立建设

建议新后台采用独立入口，避免与现有原型共用前端代码组织。

建议使用独立命名，例如：

- 新后台页面入口：`/console` 或 `/ops`
- 现有 MVP 继续保留 `/admin`

说明：

- 是否最终采用 `/console` 还是 `/ops` 可后续确定
- 关键点是必须与当前 `/admin` 彻底分开

## 8.3 技术隔离建议

建议从一开始就做以下隔离：

1. 新后台前端代码独立目录
2. 新后台页面路由独立
3. 新后台后端聚合接口独立命名空间
4. 现有 `/admin` 的 API 不做破坏式变更
5. 新后台优先新增聚合接口，不强迫复用原型页面接口形态

## 9. 新后台的 API 与服务改造方向

新后台不应只是一层前端壳，还应引入面向用户任务的聚合接口。

建议新增三类接口层：

### 9.1 词条中心聚合接口

对用户暴露“词条”视角，而不是底层多表视角。

例如：

- 词条详情聚合
- 词条来源聚合
- 词条审核/发布/验证状态聚合

### 9.2 导入中心接口

新增：

- 文件上传
- 预清洗预览
- 导入批次查询
- 批次确认导入
- 错误记录查看
- 批次回放

### 9.3 发布验证中心接口

新增：

- 发布准备视图
- gate blocker 聚合视图
- 验证失败定位视图
- 回流修复建议视图

## 10. 数据链路统一原则

词典主数据链路和样本验证链路必须明确分离，但页面中要能关联展示。

### 10.1 词典主数据链路

目标表：

- `terms`
- `aliases`
- `term_rules`
- `pinyin_profiles`

来源：

- 批量导入
- 人工维护

### 10.2 验证样本链路

目标表：

- `validation_cases`

来源：

- QA 回放
- 在线反馈
- feed 文件
- 人工补录

### 10.3 统一展示原则

虽然两条链路不同，但新后台中应做到：

- 在词条详情中可看到关联验证样本
- 在验证样本详情中可回跳到对应词条
- 在发布中心可看到验证失败对应的词条修复入口

## 11. 分阶段实施计划

建议分 4 个阶段实施。

### 阶段 1：方案与信息架构阶段

目标：

- 明确新后台范围
- 确定信息架构
- 形成页面清单和用户流程

产出：

- 本文档
- 新后台模块定义
- 页面规划
- 交互草图

### 阶段 2：词条中心阶段

目标：

- 完成 `terms + aliases + rules + pinyin` 的统一词条管理归口

产出：

- 新后台词条列表
- 新后台词条详情
- 新后台词条编辑
- 审核状态与发布状态整合展示

### 阶段 3：导入中心阶段

目标：

- 完成“原始文件上传 -> 清洗预览 -> 导入确认 -> 入库”的一条龙流程

产出：

- 文件上传
- 模板下载
- 格式说明
- 导入批次
- 清洗预览
- 行级错误反馈
- 确认导入
- 导入历史

### 阶段 4：发布与验证闭环阶段

目标：

- 完成发布、验证、回流的完整运营闭环

产出：

- 发布中心
- gate blocker 视图
- 验证样本中心
- 回流定位与跳转
