# 发布中心状态分层与页面收口方案

## 1. 文档目的

本文档用于把当前发布中心中容易混淆的“状态”拆成 3 套明确口径，并作为后续以下页面收口的统一依据：

- `/console/releases`
- `/console/releases/:id`
- `/console/reviews` 中的 release publish review 卡片
- `/console/reviews/:id` 中的 release publish review 详情

重点回答：

- 发布版本到底有哪些正式状态
- 发布审批状态与版本状态有什么区别
- 灰度状态应该如何展示，避免继续和版本状态混在一起
- 发布列表页、详情页、动作区应该如何调整

本文档描述的是后续实现方案，不代表当前页面已经全部具备这些能力。

## 2. 当前问题

当前发布中心里，用户容易把以下 3 套状态混成一个“总状态”：

1. 发布版本状态
2. 发布审批状态
3. 灰度/流量状态

这会带来几类典型误解：

- 版本是 `built`，但审批已经 `approved`，用户却以为“版本还没准备好”
- 版本是 `published`，页面上又出现“灰度中”语义，用户无法判断这是版本属性还是流量属性
- 审批按钮、灰度按钮、正式发布按钮容易因为状态混杂而显得逻辑不稳定

## 3. 三层状态模型

### 3.1 第一层：版本状态

这层对应 `releases.status` 的真实数据库状态。

当前正式枚举只有 3 个：

- `built`
- `canary`
- `published`

语义：

- `built`
  - 已构建完成
  - 当前不是正式线上版本
  - 当前不是灰度版本
- `canary`
  - 当前灰度目标版本
  - 仍然是一个 release 版本状态，不等于审批已完成
- `published`
  - 当前正式稳定版本

要求：

- 不新增第四种 release 状态来承载审核或灰度说明
- `rolled_back` 不作为 release status 存在
- 回滚后的目标版本仍然表现为 `published`

### 3.2 第二层：审批状态

这层不是 release 自身状态，而是 release 对应发布审核任务的聚合结果。

建议统一使用：

- `not_submitted`
- `pending`
- `partially_approved`
- `approved`
- `rejected`

语义：

- `not_submitted`
  - 还没有提交发布审核
- `pending`
  - 已提交，但还没有通过门槛
- `partially_approved`
  - 已有部分审批通过，但未达到 requiredApprovals
- `approved`
  - 已满足发布审批门槛
- `rejected`
  - 最新审核任务被拒绝

要求：

- 审批状态必须单独展示
- 不再借用 release status badge 表达审批含义

### 3.3 第三层：流量状态

这层不是 release status，也不是 review status，而是灰度策略与流量入口的当前运行态。

建议页面展示统一成：

- `当前无灰度`
- `灰度生效中`
- `灰度已关闭`

同时补充：

- 灰度目标版本
- 灰度百分比
- scopeType

语义：

- 用户看到的是“当前线上有没有在做灰度”
- 而不是“该 release 是不是 canary”

### 3.4 三层状态之间的关系

示意如下：

```text
版本状态：
built -> canary -> published

审批状态：
not_submitted -> pending -> partially_approved -> approved / rejected

流量状态：
无灰度 -> 灰度生效中 -> 灰度关闭
```

要求：

- 三层状态可同时存在
- 页面必须分别展示，不允许继续混成一个 badge

## 4. 页面收口原则

### 4.1 Release 列表页

列表页只展示最关键的分层摘要：

- 版本号
- 版本状态
- 审批状态
- 创建时间
- 次级摘要
  - 若是当前正式版本，标注“当前正式”
  - 若是当前灰度版本，标注“当前灰度”

建议：

- 不在列表页直接混入长说明
- 不在一个 badge 上同时表达“已发布 + 已审批 + 灰度中”

### 4.2 Release 详情页

详情页建议拆成 3 张主卡：

1. `版本状态`
   - 当前 release status
   - 是否当前正式版本
   - 是否当前灰度版本

2. `发布审批`
   - 当前审批状态
   - 已通过人数 / 要求人数
   - 最新任务 ID
   - 最近审批记录

3. `流量状态`
   - 当前是否存在启用中的灰度策略
   - 当前灰度是否指向本版本
   - 百分比
   - scopeType

### 4.3 审核中心中的 release 审核任务

审核中心里的 release publish review 不应再继续沿用“当前状态：已构建，已通过 1”这类混合描述。

建议：

- 审核列表卡片中，把 release target summary 也拆成：
  - 版本状态
  - 审批状态
  - 流量状态
- 审核详情页中，如果目标是 release publish review，增加独立的 release 状态摘要区块
- 审核中心看到的 release 状态语义，必须与发布中心保持一致

### 4.4 按钮与动作

动作判断也要分层，不再只看 release status。

建议：

- 当版本状态为 `built`
  - 可提交审核
  - 可设为灰度
  - 若审批满足，可正式发布

- 当版本状态为 `canary`
  - 可提交审核
  - 可正式发布
  - 可调整/关闭灰度

- 当版本状态为 `published`
  - 不显示“设为灰度”
  - 不显示“正式发布当前版本”
  - 若业务允许，可在其他版本上显示“回滚为正式版本”

### 4.5 顶部主标题区

顶部只保留一条主标签作为“版本状态”。

审批和流量说明下沉为独立摘要，不再混在标题主 badge 中。

## 5. 接口口径建议

### 5.1 保持不变

本批建议保持以下底层口径不变：

- `releases.status` 仍只有：
  - `built`
  - `canary`
  - `published`

- 不新增数据库字段
- 不改 release build / canary / publish / rollback 的底层状态流转

### 5.2 建议新增或明确的聚合返回

建议在 console 聚合返回中补齐 3 组字段：

- `release.status`
- `release.approval`
- `release.traffic`

其中：

- `approval`
  - `status`
  - `requiredApprovals`
  - `approvedCount`
  - `taskId`

- `traffic`
  - `grayEnabled`
  - `isCurrentCanary`
  - `isCurrentPublished`
  - `scopeType`
  - `percentage`
  - `releaseId`

要求：

- release 列表页、release 详情页、审核中心 release 任务卡片、审核中心 release 任务详情，都使用同一套聚合语义
- 不让前端自己拼 release / review / gray policy 三张表

## 6. 非目标范围

本批不包含：

- 改 release 数据库 schema
- 改审批流模型
- 改灰度策略模型
- 引入多 canary、多环境版本矩阵
- 重做发布 gate / rollout / validation 模块

## 7. 收口依据

后续编码收口应以以下结果为准：

1. 发布中心和审核中心里与 release 相关的展示，都明确区分：
   - 版本状态
   - 审批状态
   - 流量状态
2. release status 仍只有 `built/canary/published`
3. 审批状态不再借用 release badge 表达
4. 灰度状态不再借用 release status 表达
5. 发布/灰度/审核动作的可见性和禁用态符合分层语义
