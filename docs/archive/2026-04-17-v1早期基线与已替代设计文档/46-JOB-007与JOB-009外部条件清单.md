# JOB-007 与 JOB-009 外部条件清单

## 1. 目的

本文件用于明确当前 ACDP 在本地环境之外，继续推进 `JOB-007` 与 `JOB-009` 所必需的外部条件。

结论先写清楚：

- 当前本地可做的准备、契约、脚本、mock 验证都已完成
- 后续再推进这两个 job，不再缺“代码能力”或“本地验证入口”
- 真正缺的是：
  - 外部系统接入条件
  - 真实宿主机压测条件

## 2. JOB-007 外部条件清单

`JOB-007` 当前目标：

- 以 `cg3` 为首个真实 validation feed 源，完成真实 `http_pull_json` 接入、ack 联调与证据留档

### 2.1 必须提供的外部信息

1. `cg3` 真实 feed 拉取地址
   - 例如：
     - `https://<cg3-host>/...`
2. `cg3` 真实 ack 地址
   - 若对端需要导入成功确认，必须提供
3. 认证方式
   - 明确使用：
     - `bearer`
     - `header`
     - `query`
   - 并提供对应值或注入方式
4. 网络访问条件
   - 是否需要：
     - VPN
     - 堡垒机
     - 白名单
     - 特定 DNS / 代理
     - 自签 CA / 额外证书
5. 真实 payload 契约确认
   - 集合字段是否为：
     - `records[]`
     - `items[]`
   - 唯一批次字段是否为：
     - `deliveryId`
     - `batchId`
     - `cursor`
     - `requestId`
   - 下一个 cursor 字段路径是否为：
     - `meta.nextCursor`
     - 或其他路径
6. ack 语义确认
   - ack body 对端接受哪些字段
   - 什么状态码表示成功
   - ack 失败是否允许重试
7. 测试批次或测试环境说明
   - 至少要有一批可安全用于联调的 `cg3` 数据
   - 最好能提供一批可触发 ack 失败 / replay 的测试数据或测试开关

### 2.2 建议提供的环境变量

建议按当前实现直接提供：

- `ACDP_VALIDATION_FEED_CG3_ENDPOINT`
- `ACDP_VALIDATION_FEED_CG3_TOKEN`
- `ACDP_VALIDATION_FEED_CG3_ACK_ENDPOINT`
- `ACDP_VALIDATION_FEED_CG3_INITIAL_CURSOR`

如果不是 bearer token，则同步明确：

- `authType`
- `authHeaderName` 或 `authQueryKey`
- 对应认证值的注入方式

### 2.3 到位后应执行的命令

先做预检：

```bash
cd /Codex/ACDP
npm run check:validation-feeds -- --source-type cg3 --require-remote-configured --require-ack-configured
```

再做真实导入：

```bash
cd /Codex/ACDP
npm run import:validation-feeds -- --source-type cg3
```

若需要 replay 验证：

```bash
cd /Codex/ACDP
npm run import:validation-feeds -- --source-type cg3 --replay-errors
```

若要先复用本地基线确认流程：

```bash
cd /Codex/ACDP
npm run verify:validation-feeds
```

### 2.4 需要留档的证据

至少留以下内容：

1. 预检输出
2. 实际导入命令与输出
3. `feed-sources` 状态前后对比
4. receipt 文件或其摘要
5. ack 成功响应摘要
6. 若发生失败：
   - error envelope
   - replay 后恢复结果
7. 最终报告路径或人工记录说明

### 2.5 当前阻塞结论

`JOB-007` 当前不是代码阻塞，而是以下外部条件阻塞：

- 真实 `cg3` endpoint 未提供
- 真实 auth 注入条件未提供
- 真实 ack 语义未确认
- 真实网络访问条件未提供

## 3. JOB-009 外部条件清单

`JOB-009` 当前目标：

- 在真实宿主机拿到 runtime 并发和吞吐数据

### 3.1 必须提供的外部信息

1. 可执行压测的真实宿主机
   - 不能是当前 sandbox
   - 需要允许本地端口监听和真实并发请求
2. 宿主机运行环境信息
   - CPU
   - 内存
   - 操作系统
   - Node.js 版本
3. runtime 服务访问方式
   - 本机新启动
   - 或已有运行中的服务地址
   - 若复用已有服务，可直接通过 `--base-url http://<host>:<port>` 指向现成 runtime
4. 若 runtime 开启 Bearer Token：
   - 提供 `ACDP_RUNTIME_TOKEN`
5. 宿主机网络与防火墙条件
   - 确保测试请求不会被系统层限流或拦截

### 3.2 到位后应执行的命令

先做基础并发验证：

```bash
cd /Codex/ACDP
npm run test:concurrency -- --users 1 --iterations 20
npm run test:concurrency -- --users 3 --iterations 20
npm run test:concurrency -- --users 5 --iterations 20
```

若宿主机上已经有运行中的 runtime，可直接复用现成服务：

```bash
cd /Codex/ACDP
npm run test:concurrency -- --base-url http://127.0.0.1:8787 --users 5 --iterations 20
```

再做目标吞吐验证：

```bash
cd /Codex/ACDP
npm run test:concurrency -- --users 5 --iterations 50 --target-rps 200
```

当前 `test:concurrency` 每次执行后都会自动生成一份报告：

- `prototype/workspace/host_verification/*_runtime_concurrency_verify/summary.json`

报告内已包含：

- 宿主机信息
- runtime stats 前后快照
- 峰值并发
- 吞吐与延迟摘要
- 是否达到目标 `target-rps`（仅显式传 `--target-rps` 时）

### 3.3 需要留档的证据

至少留以下内容：

1. 宿主机规格
2. runtime 启动方式
3. 测试命令行
4. 每轮结果：
   - 请求总数
   - 错误数
   - p50 / p95 / max 延迟
   - 峰值并发
   - `summary.json` 报告路径
5. 是否达到 `--target-rps 200`
6. 若未达到：
   - 主要瓶颈说明

### 3.4 当前阻塞结论

`JOB-009` 当前不是脚本缺失，而是以下外部条件阻塞：

- 当前 sandbox 不能替代真实宿主机
- 本地环境不能产出可信吞吐结论
- 需要真实宿主机执行与留档

## 4. 当前建议动作

如果现在要继续推进，建议顺序如下：

1. 先向外部获取 `JOB-007` 的 `cg3` 联调条件
2. 若 `JOB-007` 条件短期内仍拿不到，再准备 `JOB-009` 的真实宿主机压测窗口
3. 在外部条件未到位前，不再继续扩本地 connector 设计或压测脚本设计
