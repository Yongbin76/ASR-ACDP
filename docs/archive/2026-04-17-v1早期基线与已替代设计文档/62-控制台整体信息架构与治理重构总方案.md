## 1. 文档目的

本文档用于定义 ACDP 下一轮控制台整体重构方向，重点解决当前系统中以下混乱点：

- 主数据录入、内容审核、版本发布、系统治理的边界不清
- 一级导航仍以当前阶段性页面堆叠为主，用户任务视角不足
- `词条中心` / `导入中心` / `审核中心` / `发布中心` / `运行节点` / `节点备案` 之间存在职责重叠
- 用户、角色、权限和发布治理规则混在一起，且实现仍然偏硬编码
- 页面中文化、术语一致性、配置驱动能力不够彻底

本方案用于承接下一轮系统级重构，不等同于局部页面优化。

## 2. 本轮重构结论

基于当前系统现状，本轮建议做以下总体调整。

### 2.1 主数据与 release 彻底解耦

词条中心和导入中心本质上都是主数据录入渠道，不应再让用户误解成“我在往某个 release 里加内容”。

统一口径：

- 词条中心、导入中心写入的是主数据
- release 是从主数据中构建出来的不可变快照
- 主数据审核通过后，进入下一次 build 的输入集合

### 2.2 审核中心与发布中心重新划边界

统一口径：

- 审核中心只处理内容审核：
  - 词条审核
  - 拼音候选审核
- 发布中心处理版本生命周期：
  - build
  - release review
  - canary
  - published
  - rollback
  - gate
  - 发布后风险

`release publish review` 不再作为审核中心主视图的一部分，而应纳入发布中心内部流程。

### 2.3 节点治理归口到系统管理

统一口径：

- 节点备案
- 运行节点
- 运行验证工具
- 用户
- 角色
- 权限
- 治理策略

都属于“系统管理”，不再分散挂在主业务导航中。

### 2.4 中文化与术语统一作为硬约束

本轮不是“顺手中文化”，而是把中文化和术语统一提升为硬约束：

- 面向用户的主文案统一中文
- 技术码只在次级信息或技术详情中暴露
- 同一业务概念只保留一种用户口径

例如：

- `词条中心-类别编码`
- `导入中心-业务属性`

当前本质是同一个概念，后续统一改为：

- `业务属性`

底层如需兼容历史字段，可继续使用 `categoryCode`，但不再作为主用户文案。

### 2.5 配置驱动，不再硬编码

本轮必须遵循：

- 不把业务属性、角色、权限、治理策略直接硬编码在页面和服务逻辑中
- 采用“配置种子 + 可持久化配置文件 + 管理界面”的方式

说明：

- 配置文件负责默认值、当前运行态和初始化种子
- 管理界面负责后续维护
- 当前原型阶段不要求把用户、角色、权限、治理策略额外落到数据库表中

因此，“通过配置文件实现”在工程上应理解为：

- 配置驱动
- 不在代码散落硬编码
- 管理动作写回统一配置文件，而不是继续把规则写死在代码里

## 3. 目标信息架构

建议控制台一级导航重构为以下 6 个域。

### 3.1 总览

统一承接：

- 当前词条规模
- 待处理事项
- 当前版本状态
- 当前运行与验证摘要

### 3.2 主数据管理

二级目录建议：

- 词条中心
- 导入中心
- 业务属性配置

目标：

- 把人工录入和批量录入统一归到“主数据输入”
- 让“业务属性”成为一个显式、可配置、可扩展的概念

### 3.3 内容审核

二级目录建议：

- 词条审核
- 拼音审核

目标：

- 审核中心只处理“内容是否可进入主数据发布流程”
- 不再把 release publish review 混在这里

### 3.4 版本发布

二级目录建议：

- 版本列表
- 版本详情
- 发布审核
- 灰度与正式发布
- 发布后风险

目标：

- 把 release 生命周期做成一条清晰链路

### 3.5 样本与验证

二级目录建议：

- 样本与回流中心
- 发布验证结果

目标：

- 把 validation 和发布验证结果统一归口

### 3.6 系统管理

二级目录建议：

- 节点备案
- 运行节点
- 运行验证工具
- 用户管理
- 角色管理
- 权限管理
- 治理策略
- 系统配置

## 4. 统一业务对象与边界

### 4.1 主数据对象

主数据对象包括：

- 词条
- 别名/错误词
- 规则
- 拼音画像
- 业务属性

这些对象由：

- 词条中心
- 导入中心

进行维护。

### 4.2 审核对象

审核对象应拆成两类：

1. 内容审核对象
   - 词条
   - 拼音候选
2. 版本审核对象
   - release publish review

但版本审核对象的操作入口应归到发布中心，而不是继续挂在内容审核中心。

### 4.3 发布对象

发布对象是 release。

release 的正式状态仍建议保持：

- `built`
- `canary`
- `published`

不新增额外 release 状态值。

### 4.4 系统治理对象

系统治理对象包括：

- runtime node registry
- runtime nodes
- users
- roles
- permissions
- governance policies
- 配置项

## 5. 业务属性统一与配置化

### 5.1 统一口径

后续统一用户口径：

- `业务属性`

不再前台主视图混用：

- `类别编码`
- `业务属性`

### 5.2 建议数据模型

建议引入可配置业务属性模型，例如：

- `business_property_code`
- `business_property_name`
- `description`
- `enabled`
- `sort_order`
- `legacy_category_code`

说明：

- 词条中心与导入中心统一读这套配置
- 底层仍可兼容历史 `categoryCode`

### 5.3 建议配置来源

建议采用：

- 配置种子文件
  - 例如：`prototype/config/business_properties.json`
- 数据库表
  - 例如：`business_properties`
- 系统管理页面
  - 用于维护新增、禁用、排序

## 6. 审核中心重构原则

### 6.1 内容审核的目的与下一步必须显式提示

审核按钮前必须明确告诉用户：

- 当前这一步在干什么
- 点“通过”后会发生什么
- 下一步应该去哪

### 6.2 词条审核

通过的目的：

- 把词条推进到可参与下次 build 的状态

通过后的下一步：

- 进入发布中心构建新版本

驳回后的下一步：

- 回词条中心修改

### 6.3 拼音候选审核

通过的目的：

- 把候选读音写回词条拼音画像

通过后的下一步：

- 如需进入 runtime，进入发布中心构建新版本

驳回后的下一步：

- 回词条详情继续治理

### 6.4 release publish review

后续不再作为内容审核中心主对象，而是迁入发布中心。

## 7. 发布中心重构原则

### 7.1 版本生命周期

统一口径：

1. `built`
   - 候选快照已构建
   - 用于检查 gate、审批和版本内容
2. `canary`
   - 当前灰度目标版本
3. `published`
   - 当前正式稳定版本

### 7.2 gate 只决定版本升级

建议统一成：

- `built -> canary` 必须无 gate 阻断
- `built -> published` 必须无 gate 阻断
- `canary -> published` 必须无 gate 阻断

但：

- `build` 本身不应被 gate 卡住

### 7.3 已发布后的问题不再继续叫 gate 阻断

一旦版本已 `published`，后续若因主数据变化、样本变化或运行态变化再次出现风险，不再继续叫：

- `门禁阻断`

建议统一改叫：

- `发布后风险`
- `运行中风险告警`
- `发布后漂移`

### 7.4 回滚策略

回滚不建议直接复用“当前 gate 重算结果”作为唯一准入条件。

后续建议：

- 回滚优先基于历史已发布快照的稳定性判断
- 不让“当前主数据重算 gate”把回滚也卡死

## 8. 系统管理重构原则

### 8.1 节点治理

节点治理统一归到系统管理：

- 节点备案
- 运行节点
- 运行验证工具

### 8.2 用户、角色、权限

当前用户、角色、权限仍是代码硬编码，后续应改为：

- 用户表
- 角色表
- 权限表
- 用户-角色关系
- 角色-权限关系

### 8.3 治理策略独立于 RBAC

不能再把“有没有权限”和“某类动作是否允许同人操作”混在一起。

建议单独建立治理策略，例如：

- 是否要求发布审核人与执行人分离
- 是否要求双人审批
- 是否强制二次确认
- 哪些环境允许单人直发

### 8.4 对“同人审批/发布限制”的建议

本轮不建议简单删除当前限制。

建议改为：

- 可配置治理策略

即：

- `dev/test` 可关闭
- `prod` 默认开启

## 9. 中文化、术语一致性与编码规范

### 9.1 中文化要求

要求：

- 页面主文案统一中文
- 面向用户的字段名称统一中文
- 技术码只放在次级信息或技术详情

### 9.2 术语统一要求

要求：

- 同一个概念只保留一种主文案
- 术语清单必须在跨中心保持一致

重点统一对象：

- 业务属性
- 内容审核
- 发布审核
- 发布后风险
- 节点备案
- 运行节点

### 9.3 代码规范要求

后续实施必须遵循：

- 函数必须有注释
- 注释必须写清：
  - 作用
  - 输入
  - 输出

### 9.4 配置驱动要求

后续实施必须遵循：

- 不写散落硬编码
- 页面枚举、业务属性、角色权限、治理策略都应配置驱动
- 优先走配置种子 + 数据库存储 + 管理界面

## 10. 本轮非目标范围

本轮不包含：

- 替换 `/admin`
- 多 admin 高可用
- 多环境 release deployment matrix 的完整实现
- 复杂审批编排引擎
- 灰度算法重写

## 11. 后续实施建议

建议以后续独立批次推进，而不是继续做零碎页面 patch。

建议拆分方向：

1. 信息架构与导航重组
2. 主数据管理重组
3. 内容审核重组
4. 发布生命周期重组
5. 系统管理与 RBAC / 治理策略重组
6. 中文化、术语统一、配置化收口
