# 新后台目录结构与模块拆分建议

## 1. 文档目的

本文件定义新后台首期编码的推荐目录结构与模块拆分方式，目标是让开发直接落目录，不再讨论“代码放哪里”。

## 2. 基本原则

### 2.1 与现有 MVP 隔离

现有目录继续保留：

- `prototype/public`
- `prototype/src/server.js`
- `prototype/src/lib/*`

新后台代码建议独立，不直接塞进现有 MVP 静态页目录。

### 2.2 共享底层，独立上层

建议：

- 底层 DB / runtime / release 能力继续复用
- 新后台页面、聚合接口、导入批次模块独立组织

## 3. 推荐目录结构

建议新增：

```text
ACDP/
  console/
    client/
      app/
      pages/
      components/
      styles/
      assets/
    server/
      routes/
      services/
      presenters/
      validators/
```

同时在 `prototype/` 下继续保留当前 MVP。

## 4. 前端目录建议

### 4.1 `console/client/app`

放：

- 路由入口
- 应用壳
- 全局上下文

### 4.2 `console/client/pages`

按一级页面拆分：

- `overview/`
- `terms/`
- `imports/`
- `reviews/`
- `releases/`
- `validation/`

### 4.3 `console/client/components`

放可复用组件：

- 表格
- 筛选栏
- 状态标签
- 摘要卡
- 详情分区
- 上传区

### 4.4 `console/client/styles`

放：

- 设计 token
- 全局样式
- 组件样式基线

## 5. 后端目录建议

### 5.1 `console/server/routes`

按接口域拆分：

- `terms.js`
- `imports.js`
- `reviews.js`
- `releases.js`
- `validation-cases.js`
- `templates.js`

### 5.2 `console/server/services`

放聚合业务服务：

- `term-console-service.js`
- `import-job-service.js`
- `review-console-service.js`
- `release-console-service.js`
- `validation-console-service.js`

### 5.3 `console/server/presenters`

放 API 响应拼装器：

- `term-presenter.js`
- `release-presenter.js`
- `import-job-presenter.js`

### 5.4 `console/server/validators`

放请求校验：

- `term-validator.js`
- `import-job-validator.js`
- `template-validator.js`

## 6. 与现有 `prototype/src/lib` 的关系

建议直接复用：

- `platform-db.js`
- `release-gates.js`
- `snapshot-builder.js`
- `admin-auth.js`

建议新增但仍放在 `prototype/src/lib` 的底层模块：

- `import-jobs-db.js`
- `import-templates.js`
- `console-query.js`

原因：

- 这些属于底层数据与查询能力
- 可以同时被 MVP 和新后台复用

## 7. 首批编码推荐顺序

1. 新增底层 DDL 与 import job 基础能力
2. 新增 `/api/console/*` 读接口
3. 新增新后台独立页面入口
4. 做总览页与词条中心只读页
5. 做导入中心模板下载与批次只读页
6. 再补写接口

## 8. 编码前结论

如果按本文件执行，开发可以直接确定：

- 新后台代码独立目录
- 前后端模块边界
- 哪些能力复用现有底层
- 哪些能力新增为新后台专用层
