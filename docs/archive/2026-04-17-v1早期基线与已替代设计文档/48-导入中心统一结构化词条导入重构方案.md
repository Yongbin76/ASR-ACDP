# 导入中心统一结构化词条导入重构方案

> 历史说明
>
> 本文档对应统一结构化词条导入收口阶段，继续保留为历史方案依据。
>
> 若本文中的旧目录或旧页面名称与当前实现冲突，以 `JOB-035` 文档组和现行帮助文档为准。

## 1. 文档目的

本文档用于明确 ACDP 导入中心下一轮重构方向：

- 不再把“原始路名 / 政府部门 / 结构化词条”作为三套并列词条导入能力继续扩展
- 统一收敛为一套“结构化词条导入”主入口和一套统一字段合同
- 用业务属性选择承接路名、政府部门、常用词等差异
- 为后续 `/console/import`、`import-jobs.js`、模板资产和测试回归提供实现基线

本文档是 `JOB-018` 的实现基线文档。

说明：

- 本文档解决的是“导入入口与字段合同统一”问题
- 后续“统一词条准入规则与跨中心收口”由 `JOB-019` 承接，见 [49-统一词条准入规则与跨中心收口方案](./49-统一词条准入规则与跨中心收口方案.md)

## 2. 当前问题

### 2.1 当前数据库结构已经说明词条落库模型是统一的

当前词条系统真正的核心落库对象只有：

- `terms`
- `aliases`
- `term_rules`
- `pinyin_profiles`
- `term_sources`
- `alias_sources`

对应文档：

- [2026-04-02/04-数据库字典与字段说明](./2026-04-02/04-数据库字典与字段说明.md)

也就是说：

- 当前没有“路名专属表”
- 当前没有“政府部门专属表”
- 当前没有“结构化词条专属表”

三类词条最终都落到同一套主数据模型。

### 2.2 当前导入中心的复杂度来自“输入形态分裂”，不是“目标模型分裂”

当前词条导入相关模板包括：

- `raw_roads_text_v1`
- `gov_departments_csv_v1`
- `structured_terms_csv_v1`

但三者最终都在导入预览阶段被标准化为同一类词条载荷，再写入：

- `terms`
- `aliases`
- 来源追踪表

因此，当前复杂度的来源是：

1. 用户入口按源文件形态拆成三套
2. 模板说明按源字段拆成三套
3. 解析逻辑按 `templateCode` 分叉维护

### 2.3 政府部门模板中的附加字段不是词典核心字段

当前政府部门模板里的以下字段：

- `district`
- `address`
- `level`
- `category`
- `remark`

目前并没有进入词典主数据正式表结构。

因此它们不应继续作为“词条主导入合同”的关键字段保留，否则会形成：

- 模板看起来很复杂
- 数据库却没有对应正式承接
- 用户误以为这些字段在字典系统中会持续起作用

## 3. 重构结论

### 3.1 一级入口收敛

导入中心后续只保留 3 个一级能力：

1. `结构化词条导入`
2. `错误词补录`
3. `验证样本导入`

其中本轮重构只收敛第一项：

- “原始路名 / 政府部门 / 结构化词条”统一并入 `结构化词条导入`

### 3.2 业务属性取代模板分类

路名、政府部门、常用词等差异，不再通过并列模板承接，而改为通过“业务属性”选择承接。

本轮固定业务属性选项如下：

| UI 展示名 | 底层映射值 | 说明 |
|---|---|---|
| `路名` | `poi_road` | 对应道路类词条 |
| `政府部门` | `gov_term` | 对应政务/机构词条 |
| `常用词` | `proper_noun` | 第一阶段直接复用现有类别编码，不新开 category migration |

说明：

- 第一阶段 UI 上可以使用“业务属性”或“业务类型”作为展示名
- 底层仍继续使用当前 `categoryCode`
- 本轮不新增新的数据库类别编码
- 若未来确实需要把“常用词”从 `proper_noun` 中进一步拆分，再作为独立批次处理

### 3.3 统一字段合同

后续只保留一套词条导入主合同。

推荐模板编码：

- 新主模板：`structured_terms_csv_v2`

旧模板处理策略：

- `structured_terms_csv_v1`：保留兼容，后续标记为 legacy
- `raw_roads_text_v1`：不再作为主入口展示，转为 legacy
- `gov_departments_csv_v1`：不再作为主入口展示，转为 legacy

## 4. 新的统一导入合同

### 4.1 合同原则

统一合同只保留“字典系统真正会持续使用”的字段。

不再保留：

- 仅用于原始数据清洗、但不进入主数据模型的来源字段
- 政府部门原始资料型字段

### 4.2 文件级形态

主路径统一采用：

- `CSV`
- `UTF-8`

说明：

- `txt` 不再作为正式主入口模板
- 纯路名逐行文本如果继续保留，也仅作为 legacy 兼容导入，不再作为主产品形态

### 4.3 统一字段定义

推荐统一字段如下：

| 字段名 | 必填 | 说明 |
|---|---|---|
| `canonicalText` | 是 | 标准词文本 |
| `aliases` | 否 | 错误词/别名，多个值用 `|` 分隔 |
| `categoryCode` | 否 | 行级类别覆盖；未提供时使用批次级业务属性默认值 |
| `priority` | 否 | 优先级，默认 `80` |
| `riskLevel` | 否 | 风险等级，默认 `medium` |
| `replaceMode` | 否 | 替换模式，默认 `replace` |
| `baseConfidence` | 否 | 基础置信度，默认 `0.9` |
| `pinyinRuntimeMode` | 否 | 拼音运行模式，默认 `candidate` |
| `customFullPinyinNoTone` | 否 | 人工指定全拼 |
| `alternativeReadings` | 否 | 备用读音，多个值用 `|` 分隔 |
| `sourceType` | 否 | 来源类型 |
| `remark` | 否 | 备注，仅用于导入辅助说明 |

说明：

- `categoryCode` 仍是底层标准字段名，不新增 `businessType` 入库字段
- 页面上对用户展示“业务属性”，但导入标准合同仍使用 `categoryCode`
- `customFullPinyinNoTone` / `alternativeReadings` 属于词条拼音画像补充字段，后续实现时应写入 `pinyin_profiles`

### 4.4 批次级默认值

创建导入批次时，页面先让用户选择一次业务属性：

- `路名`
- `政府部门`
- `常用词`

系统将其作为批次级默认 `categoryCode`。

规则如下：

1. 若 CSV 行未提供 `categoryCode`
   则使用批次默认值
2. 若 CSV 行显式提供 `categoryCode`
   则以行级值覆盖批次默认值
3. 第一阶段页面默认不鼓励混合类别导入
4. 第一阶段允许高级用户用 `categoryCode` 做行级覆盖，但不是主推荐路径

## 5. 页面与交互重构方向

### 5.1 导入中心首页

导入中心首页后续只保留：

- `结构化词条导入`
- `错误词补录`
- `验证样本导入`
- 最近导入批次
- 待确认批次

不再平铺：

- 原始路名模板
- 政府部门模板
- 结构化词条模板

### 5.2 结构化词条导入页

导入页应包含：

1. 业务属性选择区
2. 统一模板下载区
3. 示例文件下载区
4. 字段说明区
5. 文件上传区
6. 预览与校验结果区

页面上明确说明：

- 路名 / 政府部门 / 常用词的差异由业务属性承接
- 不再分别下载三套词条导入模板

### 5.3 词条编辑页

词条创建和编辑时，也应使用与导入一致的业务属性概念：

- 页面展示名称使用“业务属性”
- 底层仍绑定现有 `categoryCode`

这样可以保证：

- 手工创建词条
- 批量导入词条

在分类口径上保持一致。

## 6. 后端实现方向

### 6.1 模板元数据

`prototype/config/import_templates/index.json` 后续应调整为：

- 主推模板只保留 `structured_terms_csv_v2`
- `raw_roads_text_v1`、`gov_departments_csv_v1` 标记为 legacy
- 不再在 `/console/import` 首页主展示 legacy 词条模板

### 6.2 解析链路

`import-jobs.js` 后续应把词条导入解析收敛为：

1. 主入口解析：`structured_terms_csv_v2`
2. 可选 legacy 兼容适配：
   - `raw_roads_text_v1 -> structured_terms_v2 normalized payload`
   - `gov_departments_csv_v1 -> structured_terms_v2 normalized payload`

也就是说：

- 后续维护重点是一个统一词条合同
- legacy 只负责把旧输入转换成新合同
- 不再让三套词条模板长期并列演进

### 6.3 导入确认链路

确认导入逻辑继续统一写入：

- `terms`
- `aliases`
- `term_sources`
- `alias_sources`

新增要求：

- 如果行内出现拼音画像字段，则在确认导入阶段同步写入 `pinyin_profiles`
- 如果行内未提供拼音画像字段，则沿用当前自动生成或后续人工治理逻辑

## 7. 兼容与迁移策略

### 7.1 历史批次

已有历史批次必须继续保留原始 `template_code`，不做数据迁移。

原因：

- `import_jobs`
- `import_job_rows`
- `import_job_results`

已经形成历史记录和来源追踪，不应重写。

### 7.2 旧模板

旧模板策略如下：

| 模板 | 后续策略 |
|---|---|
| `raw_roads_text_v1` | 保留 legacy 兼容，不再主展示 |
| `gov_departments_csv_v1` | 保留 legacy 兼容，不再主展示 |
| `structured_terms_csv_v1` | 保留兼容，逐步让位给 `structured_terms_csv_v2` |

### 7.3 文档与帮助

后续用户文档、模板说明和帮助文档应统一改写为：

- 一套词条主导入合同
- 一套业务属性选择说明
- 一套 legacy 兼容说明

不再维持三套词条导入的主文案体系。

## 8. 本轮明确不做的事

本轮方案不包含：

- 新增数据库表
- 新增新的正式 `categoryCode`
- 对历史导入批次做迁移重写
- 将“业务属性”做成无约束自由标签系统
- 把样本导入和错误词补录并入结构化词条导入

## 9. 实施验收口径

后续实现完成后，应满足以下标准：

1. `/console/import` 首页只保留一个词条主导入入口
2. 用户不再需要在“原始路名 / 政府部门 / 结构化词条”之间做模板级选择
3. 词条导入统一为一套字段合同
4. 路名 / 政府部门 / 常用词差异通过业务属性承接
5. 政府部门原始资料型字段不再出现在主合同中
6. 旧模板与旧批次仍可查看、解释和兼容处理
7. `import-jobs`、`console` 导入页和相关测试保持通过
