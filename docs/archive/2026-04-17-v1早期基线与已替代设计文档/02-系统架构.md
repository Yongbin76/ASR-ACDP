# 系统架构

## 1. 架构定位

ACDP由三个层面组成：

- 控制面：词典管理、审核、发布、权限、审计
- 编译发布面：词典校验、变体生成、拼音扩展、快照构建、版本分发
- 运行数据面：CG1 热词规则引擎加载快照并执行在线匹配

## 2. 总体结构

```text
运营/算法/质检
    ↓
管理后台 / OpenAPI
    ↓
词典中心 + 审核流 + 规则配置中心
    ↓
编译器（AC快照 + 拼音索引 + 规则快照）
    ↓
版本仓库 / 发布中心
    ↓
CG1 运行时加载器
    ↓
在线匹配引擎（字面AC + 拼音增强 + 规则打分）
    ↓
asr-enhanced 标注输出
    ↓
监控、统计、CG3 数据治理回流
```

## 3. 与实时链路的关系

### 3.1 在线链路

CG1 的在线处理顺序建议为：

1. 时间片排序缓冲
2. 文本规范化
3. 字面 AC 精确匹配
4. 拼音增强候选召回
5. 规则打分与冲突消解
6. 执行替换或保留候选
7. 传递给 LLM 层

### 3.2 离线链路

CG3 数据治理平台负责：

- 发现 AC 未覆盖的新错词
- 发现线上误替换样本
- 统计词条历史正确率
- 生成拼音混淆候选与待审队列

控制面和离线治理面共同驱动下一版本词典发布。

## 4. 平台核心组件

### 4.1 管理后台

负责：

- 词典录入
- 词条查询
- 版本对比
- 发布审批
- 灰度与回滚操作
- 统计看板

### 4.2 词典中心

负责存储：

- 标准词条
- 错误变体
- 上下文规则
- 风险等级
- 类别优先级
- 拼音画像

### 4.3 变体与拼音引擎

负责：

- 生成拼音、首字母、无声调形式
- 基于混淆表生成可审核变体候选
- 维护多音字、口音、近音规则

### 4.4 编译器

负责把控制面词典编译为运行时快照：

- 字面 AC 自动机快照
- 拼音索引快照
- 词条规则快照
- 版本元信息

### 4.5 发布中心

负责：

- 预检
- 构建
- 灰度
- 全量发布
- 回滚
- 热更新通知

### 4.6 运行时加载器

负责：

- 拉取已发布快照
- 后台构建新实例
- 双缓冲原子切换
- 快照缓存
- 版本探针与健康检查

## 5. 运行时匹配架构

### 5.1 双通道匹配

运行时建议采用两条通道：

- 通道 A：字面 AC 精确匹配
- 通道 B：拼音增强通道

两条通道合并后进入统一规则层。

### 5.2 匹配结果分级

在线输出建议分三类：

- `replace`：高置信度，直接替换
- `candidate`：保留候选，供 LLM 层参考
- `block`：命中但被规则阻断，不替换

### 5.3 为什么不直接用一个大而全的 AC

原因：

- 中文不同词典类别风险差异大
- 规则优先级不同
- 拼音增强的可靠性低于字面精确匹配
- 灰度发布时常需要按类别启停

因此运行时建议多自动机并行，而不是单自动机混装。

## 6. 部署架构

### 6.1 控制面部署

- 管理后台服务
- 词典服务 API
- 编译服务
- 发布服务
- 统计服务

### 6.2 运行时部署

运行时不强依赖后台服务存活：

- 发布完成后，CG1 加载不可变快照
- 在线匹配只依赖本地内存结构
- 控制面故障不应影响已加载版本的在线匹配

## 7. 热更新架构

建议采用：

1. 后台拉取新版本快照
2. 内存中构建新自动机实例
3. 完成后做预检
4. 原子切换到新实例
5. 老实例在无引用后释放

该策略保证在线请求不中断。

## 8. 关键设计约束

- 在线匹配不直接读取数据库
- 在线匹配不直接读取 Excel
- 编译阶段完成尽可能多的工作
- 词典发布必须版本化
- 拼音增强必须可单独开关
- 高风险短词必须支持黑名单和候选模式
