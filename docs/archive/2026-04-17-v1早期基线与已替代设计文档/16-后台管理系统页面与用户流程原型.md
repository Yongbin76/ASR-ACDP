## 1. 文档目的

## 3. 产品设计原则

### 3.1 任务导向

页面必须围绕用户任务组织，而不是围绕技术模块组织。

用户的核心任务只有：

1. 导入词条
2. 维护词条
3. 审核
4. 发布
5. 验证和回流

### 3.2 单对象聚合

用户操作对象是“词条”，而不是：

- `terms`
- `aliases`
- `term_rules`
- `pinyin_profiles`

这些技术对象应在页面中被聚合成一个词条详情页。

### 3.3 上传前置说明

所有导入型页面必须在上传前提供：

- 文件格式要求
- 字段说明
- 示例说明
- 模板下载

不能让用户先上传，再靠错误提示猜格式。

## 4. 新后台整体页面结构

建议一级导航如下：

1. 总览
2. 词条中心
3. 导入中心
4. 审核中心
5. 发布中心
6. 样本与回流中心

## 4.1 模块组织方式

新后台的功能模块组织方式明确如下：

- 不采用当前 MVP 那种“单页多标签 / 多 sheet 切换”的主结构
- 采用“左侧一级导航 + 独立页面 + 页面内分区”的主结构

也就是说：

- `总览`
- `词条中心`
- `导入中心`
- `审核中心`
- `发布中心`
- `样本与回流中心`

这些都应该是独立主页面，而不是一个页面上的 6 个 sheet。

## 4.2 页内标签的使用边界

虽然不采用主页面级 sheet 结构，但在以下场景中，可以使用轻量页内标签或锚点：

- 词条详情页内的“基本信息 / 错误词 / 规则 / 拼音 / 审核 / 发布 / 验证”
- 发布详情页内的“基础信息 / gate / validation / 变更词条”
- 导入批次详情页内的“摘要 / 预览 / 错误行 / 导入结果”

要求：

- 页内标签只用于一个对象详情页内部的分区切换
- 不能把整个后台再次做成“大页签系统”
- 一级模块必须始终稳定存在于左侧导航中

## 5. 页面原型与用户流程

## 5.1 总览

### 页面目标

让用户一进入系统，就知道当前平台处于什么状态。

### 页面内容

- 当前词条总数
- 当前错误词总数
- 待审核数量
- 当前 stable / canary 版本
- 最近一次发布结果
- 最近一次验证失败数量
- 最近一次导入批次数量

### 用户动作

- 进入词条中心
- 进入审核中心
- 进入发布中心
- 进入导入中心

## 5.2 词条中心

词条中心是新后台最核心页面。

### 5.2.1 词条列表页

页面内容：

- 搜索框
- 分类筛选
- 状态筛选
- 来源筛选
- 风险等级筛选
- 审核状态筛选
- 列表结果

每条词条展示：

- 标准词
- 类别
- 来源
- 状态
- 错误词数量
- 风险等级
- 最近更新时间
- 当前是否已发布

主要动作：

- 新建词条
- 查看详情
- 提交审核
- 停用

### 5.2.2 词条详情页

一个词条详情页建议包含以下 8 个区块：

1. 基本信息
2. 错误词/别名
3. 规则
4. 拼音治理
5. 来源信息
6. 审核历史
7. 发布历史
8. 验证表现

#### 基本信息

展示：

- 标准词
- 类别
- 风险等级
- 来源
- 状态
- 创建时间
- 更新时间

#### 错误词/别名

展示：

- 当前所有错误词/别名
- 每个别名的来源
- 是否启用
- 最近更新时间

支持动作：

- 新增错误词
- 编辑错误词
- 删除/停用错误词
- 批量导入错误词

#### 规则

展示和编辑：

- `candidateOnly`
- `minTextLen`
- `maxTextLen`
- `boundaryPolicy`
- `left/right context`
- `regex allow/block`

#### 拼音治理

展示和编辑：

- 拼音运行模式
- 多音字策略
- 自定义标准读音
- 备用读音
- 拼音候选
- 拼音冲突

#### 来源信息

展示：

- 来源类型
- 是否来自批量导入
- 导入批次
- 原始文件
- 行号或来源引用

#### 审核历史

展示：

- 提交人
- 审核人
- 审核结论
- 审核时间

#### 发布历史

展示：

- 被哪些 release 引用
- 首次进入 release 的时间
- 当前是否在 stable / canary

#### 验证表现

展示：

- 关联验证样本数
- 最近验证命中/失败情况
- 失败样本列表
- 跳转到样本详情

### 5.2.3 人工维护用户流程

推荐流程：

1. 进入词条中心
2. 点击“新建词条”
3. 输入标准词与分类
4. 增加错误词/别名
5. 配置规则
6. 配置拼音
7. 保存为草稿
8. 提交审核
9. 审核通过后进入发布

## 5.3 导入中心

导入中心是把当前初始化导库链路产品化的核心页面。

## 5.3.1 导入中心首页

首页应先展示三类入口卡片：

1. 原始路名导入
2. 政府部门导入
3. 结构化词条导入

每张卡片都必须展示：

- 适用场景
- 支持文件格式
- 模板下载
- 字段说明入口
- 最近导入记录入口

## 5.3.2 上传前置区

用户进入某个导入类型后，在上传区上方必须固定展示：

- 文件格式说明
- 必填字段说明
- 选填字段说明
- 示例数据
- 模板下载按钮

模板下载要求：

- 模板必须和导入类型一一对应
- 模板必须有表头/字段名
- 模板必须给出示例值
- 模板必须清楚区分必填项和选填项

建议模板形式：

- `.csv`
- `.xlsx`
- `.json`

具体采用哪种格式，由后续技术方案确定，但产品必须先提供明确下载入口。

## 5.3.3 导入批次流程

用户流程建议为：

1. 下载模板
2. 按模板整理文件
3. 上传文件
4. 系统预校验格式
5. 系统清洗并生成预览
6. 用户查看：
   - 将新增多少词条
   - 将新增多少错误词
   - 将更新多少现有词条
   - 哪些行有错误
   - 哪些记录冲突或重复
7. 用户确认导入
8. 系统写入正式词条数据
9. 系统生成导入结果与后续审核任务

## 5.3.4 导入结果页

导入结果页应展示：

- 批次 ID
- 导入类型
- 上传文件名
- 上传时间
- 执行人
- 总记录数
- 新增词条数
- 更新词条数
- 新增错误词数
- 错误行数
- 是否需要人工确认

并支持：

- 下载错误明细
- 查看预览快照
- 查看导入后的词条列表
- 重新导入

## 5.4 审核中心

审核中心必须让用户按“待处理任务”工作，而不是按对象逐个查找。

### 页面分区

- 词条审核
- 拼音候选审核
- 发布审核
- 导入确认任务

### 用户动作

- 查看待审项详情
- 审核通过
- 审核驳回
- 填写审核意见
- 跳转到对应词条/导入批次/release

## 5.5 发布中心

发布中心必须展示“为什么能发”或“为什么不能发”。

### 页面内容

- 待发布词条变更
- 当前 release 列表
- 当前 stable / canary 状态
- gate blocker 列表
- 验证结果摘要
- 最近回滚记录

### 用户流程

1. 创建 release
2. 查看 gate
3. 查看验证结果
4. 提交发布审核
5. 灰度
6. 全量
7. 必要时回滚

### 关键要求

验证失败必须可以直接跳到：

- 失败样本
- 对应词条
- 对应修复入口

## 5.6 样本与回流中心

该模块单独承接：

- validation case 管理
- feed 来源样本
- QA 回放样本
- 线上反馈样本
- 发布验证失败样本

### 页面内容

- 样本列表
- 来源筛选
- 状态筛选
- 是否启用
- 关联标准词
- 最近命中结果

### 用户动作

- 新增样本
- 停用样本
- 导入样本
- 查看关联词条
- 查看 release 验证结果

## 6. 导入模板要求

导入模板是新后台必须补齐的正式产品能力。

### 6.1 基本要求

每类导入模板都必须包含：

- 模板名称
- 模板版本
- 字段名
- 字段说明
- 是否必填
- 示例值

### 6.2 页面要求

用户在导入前必须能看到：

- 模板下载按钮
- 字段说明文档
- 示例文件下载
- 常见错误说明

### 6.3 初始建议模板类型

建议第一批模板至少覆盖：

1. 路名原始导入模板
2. 政府部门导入模板
3. 结构化词条导入模板
4. 错误词批量补录模板

## 7. 页面间跳转关系

建议强制保证以下跳转：

- 导入批次 -> 导入生成的词条
- 词条 -> 对应审核任务
- 词条 -> 对应发布记录
- 词条 -> 对应验证样本
- 验证失败样本 -> 对应词条
- release gate blocker -> 对应样本或词条

## 8. 与现有 MVP 的关系

当前 MVP：

- `/admin`
- `/test-client`

继续保留。

新后台：

- 独立页面入口
- 独立前端代码目录
- 独立聚合接口

当前阶段禁止：

- 直接在 MVP 页面上继续叠加复杂新流程
- 让新后台前端与 MVP 前端高度耦合
