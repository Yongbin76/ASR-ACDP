## 2. JOB-022 范围定义

### 2.1 本批目标

围绕以下页面，把当前混在一起的状态信息拆成：

- `/console/releases`
- `/console/releases/:id`
- `/console/reviews` 中的 release publish review
- `/console/reviews/:id` 中的 release publish review 详情

统一按：

1. 版本状态
2. 审批状态
3. 流量状态

并同步收口：

- 列表页展示
- 详情页结构
- 动作区可见性与禁用态
- 聚合返回字段

### 2.2 本批不做

本批不包含：

- release 数据库 schema 变更
- 审批流重构
- 灰度策略模型升级
- rollout / gate / validation 结构重做
- 多环境版本矩阵
- 指定 runtime 节点下发差异化能力

## 3. 交付原则

### 3.1 先收口语义，再收口页面

必须先明确后端聚合语义：

- release status
- approval status
- traffic status

再改页面展示。

### 3.2 不改 release 正式状态枚举

`releases.status` 继续只使用：

- `built`
- `canary`
- `published`

不得新增 `rolled_back`、`reviewing` 之类版本状态值。

### 3.3 列表页与详情页必须复用同一套聚合口径

不能出现：

- 列表页自己推断审批状态
- 详情页自己再拼另一套灰度说明

必须以后端统一聚合结果为准。

## 4.2 T02 后端聚合口径收口

目标：

- 在 console read path 中，把版本状态、审批状态、流量状态分层输出
- 列表页和详情页复用同一套聚合结构

建议文件范围：

- `prototype/src/lib/console-service.js`
- `prototype/src/lib/platform-db.js`
- `prototype/src/http/admin-surface.js`

### 4.2.1 建议实现步骤

1. 明确 release 列表和详情共用的状态聚合 helper。
2. 统一 release 状态输出：
   - `status`
   - `isCurrentPublished`
   - `isCurrentCanary`
3. 统一审批状态输出：
   - `approval.status`
   - `approval.requiredApprovals`
   - `approval.approvedCount`
   - `approval.taskId`
4. 统一流量状态输出：
   - `traffic.grayEnabled`
   - `traffic.isCurrentCanary`
   - `traffic.scopeType`
   - `traffic.percentage`
   - `traffic.releaseId`
5. 确保 release list / detail 不再各自拼一套灰度或审批逻辑。

### 4.2.2 T02 验收点

- `GET /api/console/releases`
  - 每条 release 均可读到分层状态
- `GET /api/console/releases/{releaseId}`
  - 详情可读到相同语义的状态分层
- 不改 `releases.status` 的底层值

## 4.3 T03 Release 列表页收口

目标：

- `/console/releases` 不再把三套状态混成一个展示块

建议文件范围：

- `console/client/app.js`
- `console/client/app.css`

### 4.3.1 建议实现步骤

1. 列表主状态 badge 只显示 release status：
   - `built`
   - `canary`
   - `published`
2. 审批状态单独显示为次级 badge 或摘要行。
3. 当前正式版本、当前灰度版本改成独立标记，不再塞回主 badge。
4. 列表行保留最小关键信息：
   - version
   - version status
   - approval status
   - createdAt

### 4.3.2 T03 验收点

- release 列表中不再出现“一个 badge 同时表达审批+灰度+发布”的情况
- 同一 release 的审批状态变化，不会导致主版本状态 badge 变义

## 4.4 T04 Release 详情页收口

目标：

- `/console/releases/:id` 拆成 3 张主卡：
  - 版本状态
  - 发布审批
  - 流量状态

建议文件范围：

- `console/client/app.js`
- `console/client/app.css`

### 4.4.1 建议实现步骤

1. 顶部 hero 只表达版本状态。
2. 新增或重排 3 个区块：
   - `版本状态`
   - `发布审批`
   - `流量状态`
3. 详情中明确显示：
   - 是否当前正式版本
   - 是否当前灰度版本
   - 当前是否存在启用中的灰度策略
4. 技术值如 taskId、scopeType 继续保留，但下沉为次级信息，不抢主视觉。

### 4.4.2 T04 验收点

- 详情页上用户可以一眼区分：
  - 这个版本现在是不是正式版
  - 这个版本审批是否已满足
  - 这个版本是不是灰度目标

## 4.5 T05 动作区与审核中心收口

目标：

- 发布、灰度、审核动作不再靠混杂状态判断
- 审核中心中的 release publish review 卡片与详情，也使用相同的三层状态语义

建议文件范围：

- `prototype/src/lib/console-service.js`
- `console/client/app.js`
- 需要时补 `prototype/src/http/admin-surface.js`

### 4.5.1 建议实现步骤

1. 统一动作判定条件：
   - `built` 可提交审核、可设为灰度
   - `canary` 可正式发布、可调整/关闭灰度
   - `published` 不再显示“设为灰度”
2. 把动作禁用态说明文案和状态分层语义对齐。
3. 保证审核通过不等于自动发布，仍需保留显式发布动作。
4. `buildReviewTargetSummaryMap()` 对 release 目标输出三层状态摘要，不再把审批和版本状态混成一句。
5. `/console/reviews` 中 release publish review 卡片，至少能清晰显示：
   - 版本状态
   - 审批状态
   - 流量状态
6. `/console/reviews/:id` 中 release publish review 详情，增加与发布中心一致的 release 状态摘要区块。

### 4.5.2 T05 验收点

- 动作区逻辑与状态分层一致
- 用户不会因为审批状态或灰度状态而误判 release status
- 审核中心中的 release publish review 不再沿用混杂的 release 状态文案
