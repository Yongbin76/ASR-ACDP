# 原型实现与当前能力

## 1. 文档目的

本文件用于说明 `ACDP/prototype/` 当前已经实现的能力、运行方式和已知边界，避免平台设计文档与现有代码状态脱节。

## 2. 原型定位

当前原型是“可管理后台 MVP”，重点验证以下链路已经打通：

1. 原始资料导入
2. 种子词典生成
3. SQLite 管理库存储
4. 词条 CRUD
5. 规则持久化
6. 拼音画像治理
7. 审核任务流
8. 审计日志
9. 快照构建
10. 版本发布
11. 版本回滚
12. 灰度策略
13. 运行时匹配
14. 浏览器侧 UTF-8 中文 HTTP 测试

## 3. 当前代码目录

原型目录：`ACDP/prototype/`

关键文件：

- `src/server.js`：本地 HTTP 服务

## 6. 当前规则模型

原型已支持的规则字段：

- `candidateOnly`
- `minTextLen`
- `maxTextLen`
- `boundaryPolicy`
- `leftContextAllow`
- `rightContextAllow`
- `leftContextBlock`
- `rightContextBlock`
- `regexAllow`
- `regexBlock`

运行时行为：

- 命中后先执行规则评估
- 命中阻断上下文时，结果进入 `blocked`
- 命中 `candidateOnly` 时，不自动替换，只进入候选
- 设置 `minTextLen` 后，短于阈值的命中被阻断

## 7. 当前拼音画像治理能力

原型当前已支持词条级拼音画像持久化，核心字段包括：

- `fullPinyinNoTone`
- `initials`
- `syllables`
- `runtimeMode`
- `polyphoneMode`
- `customFullPinyinNoTone`
- `alternativeReadings`
- `notes`

当前实现方式：

- 若词条未配置拼音画像，则按默认拼音生成
- 若词条配置了自定义标准读音，则快照构建时优先使用人工配置
- 若词条配置了备用读音，则会一并写入快照索引
- 浏览器管理页已支持加载、编辑和保存词条级拼音画像
- 浏览器管理页已支持按词条生成多音字候选
- 已支持将候选提交到持久化审批队列
- 审批通过后会自动写入词条 `alternativeReadings`
- 浏览器管理页已支持默认生成拼音与当前画像的对比查看
- 浏览器管理页已支持字级拼音差异查看
- 浏览器管理页已支持拼音冲突列表与详情查看
- 拼音冲突接口已支持严重度分级与排序，返回 `severityLevel`、`severityScore`、`severityReasons`
- 当前冲突排序会优先暴露短词、高重叠、标准读音冲突的词组，便于先处理高风险项

## 9. 当前发布、灰度与运行时能力

当前已实现：

- 从 SQLite 中选取 `approved/published` 词条构建快照
- 生成版本目录、`snapshot.json`、`manifest.json`
- 发布到 `prototype/workspace/releases/latest/`
- 回滚到历史版本
- 维护 `stable` 和 `canary` 双运行版本
- 维护单个启用中的灰度策略
- 基于 `trafficKey` 哈希按百分比分流到 `stable` 或 `canary`
- 发布前显式提交 release review 已支持
- `publish` / `canary` 已要求 release review 审批通过
- 发布审核人与执行人分离约束已支持
- 高风险 release 双人审批已支持
- release 细粒度 gate 已支持：
- 已构建 release 中若存在词条状态失效，则禁止 `publish` / `canary`
- 已构建 release 中若存在待审拼音候选，则禁止 `publish` / `canary`
- 已构建 release 的 snapshot 若无法加载，则禁止 `publish` / `canary`
- 已构建 release 会执行 smoke simulation 校验；若样本词未在该 release snapshot 中被识别，则禁止 `publish` / `canary`
- 已支持基于仓库配置样本集的业务文本回放校验，配置文件位于 `prototype/config/release_validation_cases.json`
- 已支持将业务校验样本持久化到 SQLite，并在首次初始化时从仓库配置导入
- release validation 默认优先读取库内 enabled 业务样本，库内无样本时才 fallback 到仓库配置
- 已支持批量导入业务校验样本 feed，支持 `upsert` 和 `insert_only`
- 已支持基于 connector 配置的外部样本连接器：
- `cg3`
- `qa_feedback`
- `online_feedback`
- connector 默认仍可走 `file_inbox`
- 已支持首轮 `http_pull_json` connector 骨架：
  - delivery receipt 去重
  - cursor 增量拉取
  - 可选 `http_post` ack
  - error payload replay
- 外部 feed 文件导入后会自动归档到 archive 目录
- 不合法或未识别的 feed 文件会自动进入 error 目录
- 单个坏文件不会阻塞整批 feed 导入
- 已支持 source-specific 原始 payload 适配：
- `cg3.records[]`
- `qa_feedback.feedbacks[]`
- `online_feedback.events[]`
- 已提供示例文件 `prototype/config/validation_feed_examples.json`

## 12. 独立新后台 `/console` 当前能力

当前原型已收口到单一后台页面入口：

- `GET /console`

定位说明：

- `/console`：当前正式后台页面入口
- `/admin`：重定向到 `/console`

当前 `/console` 已实现：

- 总览：首页指标卡、版本状态、快速入口、帮助中心入口
- 词典建设：词典记录列表优先布局、显式中文字段标题、词典记录创建/编辑、规则维护、拼音治理、提交审核、停用、查看关联样本
- 批量导入：统一结构化词条导入主路径、业务属性选择、模板/示例附件下载、导入批次创建、预览、确认导入、错误报表下载、影响词典记录和审核任务查看，并可直接跳到“本批词条审核”视图
- 词典审核：审核任务主列表仅保留词条审核与拼音候选审核，详情页已补“本步目的 / 下一步”提示；词条审核已支持“当前页勾选任务”与“当前导入批次待审核任务”两种批量审核方式，且导入批次范围现只处理仍为 `pending` 的任务
- 发布中心：release 构建、提审、灰度、发布、回滚、升级前置条件查看、发布后风险查看；列表页与详情页已按“版本状态 / 审批状态 / 流量状态”分层展示
- 验证与回流：验证样本创建、停用、关联词条查看
- 系统管理：节点备案、运行节点中心、运行验证工具、用户管理、角色管理、权限分配、治理策略
- 帮助中心：`/console/help/*` 内嵌联调说明与试用说明

当前 `/console` 额外补齐的可用性能力：

- 核心表单已全部改为显式 label，不再依赖 placeholder 充当字段标题
- 各工作台页面已补齐“筛选栏 + 摘要卡 + 下一步动作”三段式布局，降低首次使用门槛
- 词条、导入批次、审核任务、版本列表、运行节点、验证样本等主列表页，统计口径已统一为“顶部总览 / 筛选结果 / 当前页”三层结构；值为 0 的次级顶部卡片默认不展示
- 词条、导入批次、审核任务、发布版本、样本列表都已补齐统一分页能力，翻页时会保留当前筛选条件
- 节点备案、运行节点、导入详情预览行、词条审核、发布版本、验证样本等主列表页，当前已统一为“筛选 / 清空 / 翻页只刷新结果区，不整页重刷”
- 主列表页的分页入口已统一收口到各自“结果列表”区块底部，不再漂浮在页面最底部
- 词条中心已支持按来源类型、风险等级、排序方式直接筛选，且筛选与排序已下推到后端真实查询
- 统一准入已补单字标准词特例：单字标准词必须填写错误词 / 别名，且错误词 / 别名中不能出现任意单字；只有错误词均为非单字时才允许通过该条准入校验
- 词条中心已支持“当前页勾选”与“当前筛选结果”两种批量送审/批量停用执行方式
- 词条中心已支持按当前筛选条件导出 CSV
- 导入中心已支持按批次状态、来源类型筛选；导入详情页已支持按行状态、处理决策筛选
- 词条导入主路径已统一到 `structured_terms_csv_v2`；旧版路名 / 政府部门 / 结构化词条模板仅保留 legacy 兼容入口
- 业务属性配置页已支持新增、编辑、启停与删除，并写回统一配置文件
- 词典审核已支持按任务状态、任务类型、目标类型筛选，并增加待审/已通过/已驳回摘要条；词条审核已支持 `importJobId` 批次上下文过滤与批量审核；release publish review 已迁入发布中心内部流程
- 发布中心已支持按发布状态筛选，并补充 built/canary/published 摘要卡；release list / detail 均已分层输出版本状态、审批状态、流量状态，并把 `published` 后的异常收口为“发布后风险”
- 验证与回流页已支持按关键字、来源类型、启用状态筛选，并在列表中直出期望标准词数量
- 验证与回流页已接入正式文件导入入口，可直接走 `validation_cases_csv_v1` 模板上传、预览、确认导入闭环
- 验证与回流页仍保留 JSON 批量导入入口，作为开发调试兼容方式
- 验证与回流页已支持“当前页勾选”与“当前筛选结果”两种批量停用方式
- 验证与回流页已支持按当前筛选条件导出 CSV
- 顶部身份、左侧导航、用户/角色/权限、治理策略都已改为配置驱动，不再继续散落硬编码
- 关键动作完成后会给出“下一步”提示，例如：
- 词条创建后直接进入详情页，并提示提交审核
- 导入确认后提示去看影响词条和审核任务
- 模板、示例、错误报表均以附件下载方式返回
- 宿主反馈中审核中心摘要异常问题已通过聚合摘要修复
- `listConsoleTerms()`、`listConsoleImportJobs()` 等聚合查询已完成一轮 N+1 优化

## 13. 当前验证方式

围绕 `/console` 当前已具备以下验证方式：

- `npm run test:unit`：覆盖服务层和接口级单元测试
- `npm run test:console`：覆盖词条、导入、审核、发布、样本主链路 E2E
- `npm run smoke:console`：优先尝试真实 HTTP smoke；若运行环境禁止端口监听，则自动 fallback 到 inject 模式做服务级 smoke

说明：

- 在真实宿主环境中，仍建议优先验证真实 HTTP 模式的 `/console` 联调结果
- 在当前受限沙箱环境中，`smoke:console` 会自动使用 inject 模式完成可执行检查

## 14. 已知边界

当前运行环境要求：

- 原型依赖 Node 内置 `node:sqlite`
- Ubuntu 开发环境需使用 Node.js `>= 22.13.0`
- 当前仓库已提供 `.nvmrc`
- 每次恢复工作前建议先执行 `npm run check:env`
- 原型当前默认监听 `0.0.0.0:8787`
- 若需通过外网 IP 访问，还需要放通主机防火墙和云安全组中的 TCP `8787`
- 若需要正式对外提供运行时接口，可通过环境变量 `ACDP_RUNTIME_TOKEN` 启用 Bearer Token 鉴权

当前原型新增规则能力：

- `maxTextLen` 已支持
- `boundaryPolicy=char_type` 已支持
- `regexAllow/regexBlock` 已支持

当前原型新增拼音治理能力：

- 词条级自定义标准读音已支持
- 备用读音数组已支持
- `polyphoneMode` 已支持基础持久化
- 拼音冲突严重度排序已支持
- 短词 / 高重叠冲突已优先暴露
- generated candidates 持久化 review queue 已支持
- candidate review 批准后自动写入 `alternativeReadings` 已支持
- character-level pinyin comparison 已支持

当前仍未完成：

- 已有 file-based 连接器，但尚未直连真实外部系统推送/拉取接口
- feed source 当前仅支持本地目录投递，尚未支持鉴权、回执和自动重试策略
- `test:concurrency` 依赖本地端口监听能力，受限沙箱中无法执行，需要在真实主机环境中跑
- 未显式传 `--target-rps` 时，`test:concurrency` 只记录实际吞吐与延迟；需要目标吞吐判定时再显式加 `--target-rps`
- 当前 K8S 资产仍是单副本 MVP 形态，尚未支持多副本共享状态
- `200 并发/秒` 目标尚未在真实主机环境完成实测验收
- WebSocket 已支持与 HTTP 一致的 Bearer Token 鉴权，但尚未加入更复杂的连接配额/黑白名单治理

当前原型仍有以下限制：

- 拼音治理仍是词条级实现，尚未支持更细粒度的字级覆盖
- 多音字治理目前仍是词条级能力；原型虽已支持候选持久化审批与字级对比，但尚未支持更细粒度的字级覆盖编辑
- 规则模型和审批流仍是简化版
- 管理页是单页原型，不是正式管理后台
- 灰度策略当前只支持 `traffic_key_hash` 单策略模型

## 15. 后续建议

下一阶段建议优先补：

1. 多级审批与权限模型
2. 更完整的冲突严重度排序
3. 候选持久化审核队列
4. 更完整的灰度策略模型
5. 正式管理前端
