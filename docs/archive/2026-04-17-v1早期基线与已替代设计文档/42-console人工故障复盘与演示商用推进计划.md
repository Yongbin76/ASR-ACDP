## 4. 明天演示规避方案

明天演示建议只走已经最稳定的主链路，不临时扩展场景。

### 4.1 建议演示页面

- `/console`
- `/console/terms`
- `/console/reviews`
- `/console/releases/{releaseId}`
- `/console/runtime-nodes`

### 4.2 建议演示顺序

1. 从首页工作台进入词条中心
2. 新建词条并送审
3. 切换审核员完成审核
4. 构建 release
5. 提交发布审核并通过
6. 在 release 详情页展示：
   - 发布确认
   - 节点收敛
   - 验证结果
   - 验证证据
7. 进入 runtime 节点页展示当前版本、目标版本和异常说明

### 4.3 演示前检查

建议演示前固定执行：

```bash
cd /Codex/ACDP
npm run smoke:console
npm run test:console
```

如果要演示真实控制面模式，再补：

```bash
ACDP_RUNTIME_TOKEN=... npm run check:control-config -- --require-env-sources
ACDP_RUNTIME_TOKEN=... npm run verify:runtime-control -- --artifact-store-mode=configured
```

### 4.4 演示时规避项

- 不演示真实目标集群部署，因为当前 `kube_context_missing`
- 不把“历史异常卡片”当作当前线上故障来讲，而是说明为“历史异常已纳入统一异常治理”
- runtime 页面默认保持手动刷新，不现场开启持续自动刷新

## 5. 小规模商用前必须关闭项

以下项建议视为“小规模商用前必关”：

### 5.1 P0

- 运行异常生命周期治理
  - 区分历史异常 / 当前异常 / 已恢复异常
- 首页与关键页首屏等待继续压缩
  - 目标是显著低于当前体感等待
- 统一后台交互规范文档化
  - 刷新
  - 异常
  - 加载
  - 滚动

### 5.2 P1

- 把 `/test/ACDP/` 中剩余历史问题逐条复盘并标状态
- 继续扩大“卡片内滚动收边”覆盖范围
- 演示资产转商用资产
  - 固定账号
  - 固定数据
  - 固定检查清单

## 6. 平台推进计划

## 6.1 第一阶段：明天演示保障

目标：

- 明天完整演示稳定、不丢主链

重点：

- 固定演示账号、演示节点、演示 release
- 演示前 smoke 与 E2E
- runtime 页面保持手动刷新
- 只走最稳主链

## 6.2 第二阶段：试运行 / 小规模商用准备

目标：

- 平台可给少量真实用户试运行

重点：

- 完成剩余后台统一治理
- 把历史问题台账化
- 继续优化关键页首屏性能
- 继续优化异常治理闭环

## 6.3 第三阶段：小规模商用

目标：

- 平台具备稳定使用、交接、运维和问题回收能力

重点：

- 统一后台设计规范固定下来
- 演示资产升级为商用资产
- 控制面 / 数据面 / 验证证据 / 故障处理形成完整闭环

## 7. 当前建议

当前建议优先级如下：

1. 明天演示只走稳定主链
2. 演示前固定执行 smoke / E2E
3. 演示后立即把 `/test/ACDP/` 历史问题逐条转成正式台账
4. 再继续推进“小规模商用前必须关闭项”
