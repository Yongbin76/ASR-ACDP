## 1. 文档目的

本文档用于把后台一级、二级目录的命名原则和后续配置驱动方案明确下来，避免后续继续在页面代码中散落硬编码导航名称。

## 2. 设计目标

后续导航需要同时满足：

1. 命名简洁
2. 层级清晰
3. 业务语义稳定
4. 页面标题、breadcrumb、帮助目录可同源
5. 可通过配置文件调整，而不是到处改代码

## 3. 命名原则

### 3.1 一级目录命名原则

- 以“业务域”命名
- 尽量短
- 少用“中心”
- 避免一个名称里同时出现“管理/中心/平台/工具”多个抽象后缀

### 3.2 二级目录命名原则

- 直接表达对象或动作
- 尽量和页面主对象一致
- 避免歧义词

例如：

- `词条`
- `批量导入`
- `词条审核`
- `发布审核`
- `节点备案`

### 3.3 技术词下沉

以下内容不应作为导航主文案：

- 内部枚举
- 字段编码
- 英文 route
- 技术缩写

它们应只出现在：

- 技术详情
- 调试信息
- 开发文档

## 5. 配置驱动建议

### 5.1 建议新增统一导航配置文件

建议路径：

- `prototype/config/console_navigation.json`

### 5.2 建议配置结构

示例：

```json
{
  "groups": [
    {
      "key": "workbench",
      "label": "工作台",
      "order": 10,
      "items": [
        {
          "key": "overview",
          "label": "工作台",
          "path": "/console",
          "pageKey": "/",
          "feature": "overview.view"
        }
      ]
    }
  ]
}
```

### 5.3 每项建议字段

- `key`
- `label`
- `order`
- `path`
- `pageKey`
- `feature`
- 可选 `helpSlug`
- 可选 `group`

## 6. 与页面标题、breadcrumb、帮助的关系

后续应统一成：

- 导航名称来自统一配置
- 页面标题来自统一配置或同一映射表
- breadcrumb 使用同一命名源
- 帮助目录和导航目录保持一致

## 7. 当前建议

如果后续正式进入 `v1.0` 重构，建议把“导航命名收口”和“导航配置化”作为独立工作包处理，而不是在实现过程中边写边改。

当前 `/console` 壳层已先按本文件收口一级、二级目录主文案；后续如果继续做导航配置化，应以这里的命名作为唯一真源。
