# 控制面 / 数据面实施分解与开发前自审

## 1. 目的

本文件用于把已确认的控制面 / 数据面架构方案进一步拆成可执行的开发批次，并在正式编码前完成一轮自我审核。

目标不是再讨论方向，而是回答：

- 先做什么
- 后做什么
- 哪些依赖已经清楚
- 哪些风险仍在，但不阻塞开发
- 是否已经达到“可以开始实际开发”的状态

## 2. 实施工作分解

## 2.1 第一批：制品仓与控制面骨架

覆盖 job：

- `JOB-002B`
- `JOB-002C`
- `JOB-002D` 的最小版本

本批目标：

- 让 admin 具备管理 runtime 节点和发布目标版本的最小控制能力
- 让 MinIO 配置与 artifact metadata 先落地

具体工作项：

1. 新增 artifact store 配置读取模块
2. 新增 MinIO client 封装
3. 新增 artifact key/path 生成规则
4. 新增 runtime node 数据表 / 读写方法
5. 新增 runtime register / heartbeat API
6. 新增 desiredVersion / artifact metadata 存储结构
7. 新增 `GET /api/runtime-control/me`

本批验收：

- admin 能记录 runtime 节点
- admin 能维护 desiredVersion
- MinIO 相关配置全部来自配置文件

## 2.2 第二批：runtime 制品拉取与本地版本切换

覆盖 job：

- `JOB-002E`
- `JOB-002D` 的 apply result 回传部分

本批目标：

- 让 runtime 真正脱离共享 `latest` 目录
- 让 runtime 基于本地制品目录独立运行

具体工作项：

1. 设计本地制品目录结构
2. 实现 artifact 下载器
3. 实现 checksum 校验
4. 实现 currentVersion / desiredVersion 持久化
5. 实现本地原子切换
6. 实现 apply result 回传
7. 保留回滚能力

本批验收：

- runtime 能基于 admin 下发的 artifact metadata 拉取并切换版本
- runtime 启动不再依赖共享 workspace 的 `latest`

## 2.3 第三批：runtime 本地统计缓冲与回传

覆盖 job：

- `JOB-002F`

本批目标：

- 让 runtime 的运行统计脱离 admin 共享库
- 支撑多节点聚合统计

具体工作项：

1. 设计 runtime 本地统计表
2. 实现 300s 定时 flush
3. 实现批次号 / 序列号 / cursor
4. 实现 admin 侧幂等入库
5. 实现节点级和全站聚合读取

本批验收：

- runtime 本地累计统计
- admin 可收到多节点批量回传
- 重复上报不产生重复统计

## 2.4 第四批：`/console/runtime-nodes` 页面

覆盖 job：

- `JOB-002G`

本批目标：

- 让控制面可以查看 runtime 节点与版本分发状态

具体工作项：

1. 新增 runtime 节点列表页
2. 新增节点详情区
3. 展示当前版本 / 目标版本 / apply 结果 / 心跳时间
4. 展示节点统计摘要
5. 预留控制动作入口

本批验收：

- `/console/runtime-nodes` 可查看节点、版本和基本状态

## 2.5 第五批：RBAC 四层重构

覆盖 job：

- `JOB-002H`

本批目标：

- 把 `/console` 的权限模型从“角色散点控制”收敛为统一的 用户-角色-权限-页面功能 四层模型

具体工作项：

1. 定义用户模型
2. 定义角色模型
3. 补齐权限清单
4. 建立页面功能矩阵
5. 把页面按钮/动作映射到权限
6. 调整 `/console` 顶部身份模型

本批验收：

- 页面功能与权限关系清楚
- 高风险操作以“当前用户”作为真实边界主体

## 3. 推荐开发顺序

建议按以下顺序进入编码：

1. `JOB-002B` MinIO 配置与 artifact store 抽象
2. `JOB-002C` runtime node register / heartbeat
3. `JOB-002D` desiredVersion / artifact metadata / apply result
4. `JOB-002E` runtime 拉取制品与本地切换
5. `JOB-002F` runtime 本地统计与回传
6. `JOB-002G` `/console/runtime-nodes`
7. `JOB-002H` RBAC 四层重构

原因：

- 先打通控制链路和制品链路
- 再补统计闭环
- 再补可视化页面
- 最后做权限收口，避免前期为了改权限反复改接口

## 4. 关键依赖检查

本轮自审确认：

### 4.1 已明确，不阻塞开发

- 第一版制品仓：MinIO
- 第一版部署位置：现有服务器
- runtime 本地状态：SQLite
- 统计回传周期：300s
- 新增 `/console/runtime-nodes`
- RBAC 四层重构纳入范围
- 敏感信息第一阶段先放配置文件

### 4.2 仍存在，但不阻塞当前编码

- MinIO 的生产级安全方案
- 真实目标 K8S 集群验证
- 真实镜像仓库接入

这些会影响后续上线质量，但**不阻塞现在开始实现控制面 / 数据面能力本身**。

## 5. 风险自审

### 风险 A：把 MinIO 和制品路径模型做死

当前应对：

- 已独立出 `artifact_store.config.json`
- 后续实现必须只从配置读取

结论：

- 可控，不阻塞开发

### 风险 B：runtime 与共享 workspace 再次耦合

当前应对：

- 第二批实现明确要求 runtime 改为本地拉制品并本地切换

结论：

- 可控，但要在实现时严格避免继续依赖共享 `latest`

### 风险 C：RBAC 重构过晚导致前面返工

当前应对：

- 方案里已把 RBAC 范围和页面功能映射提前定义

结论：

- 有风险，但当前顺序仍可接受
- 编码时要同步维护权限矩阵，不能完全拖到最后才想

### 风险 D：统计回传模型设计不清导致 admin 入库返工

当前应对：

- 已明确按 `nodeId + batchId + sequence` 做幂等

结论：

- 可控，不阻塞开发

## 6. 开发前自审结论

结论：

- **已达到开始实际开发的条件**

原因：

1. 方向已经冻结
2. 关键节点已确认
3. 第一版制品仓、存储方式、回传周期、页面范围、RBAC 范围均已明确
4. 当前剩余的不确定项主要属于“后续上线/安全/真实环境验证”，不阻塞实现

## 7. 建议执行方式

开始实际开发时，建议按“批次完成 + 自测 + 文档同步”的方式推进：

1. 每一批只解决一个闭环
2. 每批完成后更新：
   - `docs/38-项目JobList与状态清单.md`
   - `SESSION_HANDOFF.md`
3. 每批都要补最小测试
