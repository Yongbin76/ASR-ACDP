# 运行时对外服务与后台服务拆分建议

## 1. 当前问题

当前实现把以下能力放在同一个服务进程中：

- 对外 runtime HTTP/WS 接口
- `/admin` 页面与 `/api/admin/*`
- `/console` 页面与 `/api/console/*`

对应现状可见：

- 单一入口文件：`prototype/src/server.js`
- 单一后台守护脚本：`prototype/src/cli/service-manager.js`
- 单一镜像启动方式：`Dockerfile`
- 单一 K8S Deployment：`k8s/deployment.yaml`

这会带来三个直接问题：

1. 改动后台代码时容易连带影响对外 runtime 服务
2. runtime 能力无法独立部署、扩缩容和隔离发布
3. `/console` 与 `/admin` 的后台开发节奏会绑到对外服务发布节奏上

## 2. 建议目标

建议拆成两个独立服务：

### 2.1 Runtime Service

只负责：

- `GET /health`
- `GET /api/runtime/current`
- `GET /api/runtime/stats`
- `POST /api/runtime/reload`
- `POST /api/runtime/correct`
- `GET /ws/runtime/correct`
- 如果保留内部调试能力，可选保留 `/api/simulate`

### 2.2 Admin Service

只负责：

- `/admin`
- `/console`
- `/api/admin/*`
- `/api/console/*`

## 3. 为什么这样拆

这样拆之后：

- runtime 服务可以独立部署为对外服务，发布节奏更稳
- admin 服务可以持续迭代 `/console`，不和 runtime 发布绑定
- `/admin` 与 `/console` 仍可共处于 admin 服务内，保持双轨入口，但不再和外部 runtime 服务耦合

## 4. 当前代码基础是否支持拆分

当前基础已经具备一部分：

- 底层 DB、runtime、release-gate、console service 已在 `prototype/src/lib/*` 分层
- 当前数据库已启用 `PRAGMA journal_mode = WAL`
- 服务已存在 `createPrototypeApp()` 这种可测试入口

但仍缺少：

- runtime 路由与 admin/console 路由的独立 app 装配
- 独立服务入口文件
- 独立服务脚本、镜像和 K8S 部署

## 5. 推荐拆分路径

### 阶段 A：代码层逻辑拆分

先不改对外地址，只做代码重构：

- 从 `prototype/src/server.js` 中抽出 `createRuntimeApp()`
- 从 `prototype/src/server.js` 中抽出 `createAdminApp()`
- 把共享逻辑下沉到 `prototype/src/lib/*`
- 保持现有 API 行为不变

当前进展：

- 已开始第一步代码层拆分
- 已把 `server.js` 中的 runtime 路由和 admin/console 路由拆到独立 surface 模块
- 当前外部入口、URL 和服务进程形态保持不变

目标：

- 先把“一个文件里混着三套路由”的问题拆开
- 降低后续独立进程化的风险

### 阶段 B：进程层拆分

新增两个独立入口：

- `prototype/src/runtime-server.js`
- `prototype/src/admin-server.js`

并新增脚本：

- `npm run start:runtime`
- `npm run start:admin`
- `npm run service:start:runtime`
- `npm run service:start:admin`

当前进展：

- 已提供 `npm run start:runtime` / `npm run start:admin`
- 已提供 `npm run service:start|status|stop:runtime`
- 已提供 `npm run service:start|status|stop:admin`
- 已提供 `npm run smoke:runtime` 与 `npm run smoke:admin`
- 已补充 surface 级测试，确认 runtime 面不暴露 `/admin`，admin 面不暴露 `/api/runtime/*`
- 已补充 Docker/K8S split 资产，并完成本地 `kind` 集群首轮验证

目标：

- 同一台宿主机先跑两个独立进程
- `/admin` 与 `/console` 继续共享 admin 进程
- runtime 对外接口独立为另一个进程

### 阶段 C：部署层拆分

分别提供：

- Runtime Dockerfile / Deployment / Service
- Admin Dockerfile / Deployment / Service

建议部署口径：

- runtime service：对外暴露，优先稳定与低延迟
- admin service：内网/VPN/堡垒机访问，优先管理能力

## 6. 数据与状态建议

短期内可继续共用：

- 同一份 SQLite 管理库
- 同一份 workspace/release snapshot

但要控制边界：

- admin 服务负责词条、审核、发布、版本管理写入
- runtime 服务尽量只读已发布 snapshot
- runtime 侧若继续写统计，建议先保留轻量写入；中期再迁移为独立统计存储

## 7. 我的建议

建议优先级如下：

1. 先完成 `/console` 收尾和宿主验证，不动 `/admin`
2. 紧接着做“代码层逻辑拆分”
3. 再做“进程层拆分”
4. 最后做“镜像/K8S 部署层拆分”

原因：

- 先把 `/console` 收尾，避免在功能仍波动时叠加架构迁移
- 先拆代码再拆进程，风险最小
- `/admin` 与 `/console` 的双轨独立，应该在 admin 服务内部继续保持，不需要把它们再拆成两个服务
