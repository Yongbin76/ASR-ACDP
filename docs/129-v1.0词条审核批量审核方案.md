## 1. 目标

在不放大误审风险的前提下，为 `词条审核` 补齐可落地的批量审核能力，优先解决 CSV 批量导入后生成大量 `term_review` 任务只能逐条处理的问题。

## 2. 当前现状

当前系统存在两个明显事实：

- `主数据 -> 词条` 已支持批量“提交审核”，但不支持批量“审核通过 / 驳回”
- `内容审核 -> 词条审核` 当前只有单任务审批接口和单任务按钮

现有实现上：

- `POST /api/console/terms/batch-submit-review` 已存在
- `POST /api/console/reviews/{taskId}/approve` 已存在
- `POST /api/console/reviews/{taskId}/reject` 已存在
- `/console/reviews?view=terms` 当前没有多选、全选、批量通过、批量驳回

因此，CSV 导入后即使一次生成几十条 `term_review`，审核员也只能逐条点。

## 3. 为什么不能直接做“按当前筛选结果批量审核”

当前 `词条审核` 页的筛选条件只有：

- `view=terms`
- `status`

如果在此基础上直接支持“按当前筛选结果批量通过”，当筛选条件只是 `待审核` 时，实际会把全站待审核词条一起处理，风险过高。

所以第一版必须坚持：

- 不做“宽筛选一键全审”
- 不做 release / 拼音候选的批量审核
- 不做没有上下文约束的批量通过

## 4. 第一版目标范围

第一版只覆盖：

- 对象：`term_review`
- 页面：`/console/reviews?view=terms`
- 上下文：显式选中的任务，或某个明确的导入批次

第一版明确不覆盖：

- `pinyin_candidate_review`
- `release_publish_review`
- 没有批次上下文的“当前筛选结果全部审核”
- 跨多种审核类型混合批量处理

## 5. 设计原则

### 5.1 安全优先

批量审核必须比单条审核更保守，而不是更宽松。

### 5.2 上下文明确

只有在“我知道这批任务来自哪里”的前提下，才允许一键批量处理。

### 5.3 复用现有副作用逻辑

批量审核不应重新实现词条状态流转，应复用：

- `approveReviewTask()`
- `rejectReviewTask()`

这样单条审核与批量审核的副作用、审计、状态语义保持一致。

### 5.4 先解决导入批次场景

CSV 导入是当前最强的批量审核诉求，第一版优先解决这一场景。

## 6. 推荐目标态

### 6.1 导入批次详情页提供批量审核入口

在 `/console/import/jobs/:jobId` 的“生成的审核任务”区块增加入口：

- `前往词条审核本批任务`

跳转到：

- `/console/reviews?view=terms&importJobId={jobId}`

这样审核员不是在一个宽列表里盲审，而是在“这个导入批次”的上下文里处理。

### 6.2 词条审核页支持两种批量作用域

仅在 `view=terms` 下提供批量审核条。

支持两种作用域：

- `selected_tasks`
  - 只处理当前页显式勾选的任务
- `import_job`
  - 只处理当前 `importJobId` 对应批次下的全部待审核词条任务

其中：

- 默认作用域是 `selected_tasks`
- 只有存在 `importJobId` 时，才显示 `import_job`

### 6.3 页面交互建议

在 `词条审核` 页新增：

- 每条任务前的勾选框
- 页头批量操作条
- 当前上下文提示条
- 已勾选数量 / 本批待审核数量摘要

批量操作条建议字段：

- 批量动作：`批量审核通过` / `批量驳回`
- 作用域：`当前页勾选任务` / `当前导入批次待审核任务`
- 驳回备注：仅批量驳回时必填或建议填

### 6.4 上下文提示建议

当页面带 `importJobId` 时，页内必须显式提示：

- 当前只显示某个导入批次关联的词条审核任务
- 批量审核只会作用于该批次的待审核任务

避免用户误以为是全站批量动作。

## 7. 后端接口建议

## 7.1 扩展审核列表查询

现有：

- `GET /api/console/reviews`

建议新增过滤参数：

- `importJobId`

用途：

- 只返回与该导入批次相关的 `term_review`

实现建议：

- 通过 `term_sources.import_job_id` 与 `review_tasks.target_id = term_id` 关联
- 使用 `EXISTS` 或去重查询，避免一个词条多条 source 记录导致重复行

## 7.2 新增批量通过接口

- `POST /api/console/reviews/batch-approve`

建议请求体：

```json
{
  "scope": "selected_tasks",
  "taskIds": ["review_001", "review_002"],
  "importJobId": "",
  "comment": "console batch approve"
}
```

或：

```json
{
  "scope": "import_job",
  "importJobId": "import_job_001",
  "comment": "console batch approve"
}
```

## 7.3 新增批量驳回接口

- `POST /api/console/reviews/batch-reject`

建议请求体：

```json
{
  "scope": "selected_tasks",
  "taskIds": ["review_001", "review_002"],
  "comment": "字段不完整，需回主数据修正"
}
```

或：

```json
{
  "scope": "import_job",
  "importJobId": "import_job_001",
  "comment": "整批分类编码配置错误，退回修正"
}
```

## 7.4 响应建议

返回风格继续保持和现有 console API 一致：

```json
{
  "ok": true,
  "item": {
    "scope": "import_job",
    "decision": "approve",
    "totalRequested": 20,
    "processedCount": 18,
    "approvedCount": 18,
    "rejectedCount": 0,
    "skippedNotFoundCount": 0,
    "skippedNonPendingCount": 2,
    "items": [
      {
        "taskId": "review_001",
        "targetId": "term_001",
        "status": "approved"
      }
    ]
  }
}
```

## 8. 服务层实现建议

建议新增两个层次的能力。

### 8.1 任务解析层

负责把“批量作用域”解析成最终待处理任务集合。

建议新增辅助函数：

- `listPendingTermReviewTasksByImportJobId(db, importJobId)`
- `resolveBatchReviewTasks(db, payload)`

约束：

- 只能解析 `taskType=term_review`
- 只能解析 `targetType=term`
- 只能处理 `status=pending`

### 8.2 批量决策层

建议新增：

- `batchApproveReviewTasks(db, payload, operator)`
- `batchRejectReviewTasks(db, payload, operator)`

实现原则：

- 先解析任务集合
- 在事务中逐条调用现有单任务函数
- 汇总处理结果

不要复制单任务状态流转逻辑。

## 9. 权限与护栏建议

建议新增页面功能点：

- `reviews.bulk.approve`
- `reviews.bulk.reject`

权限仍复用：

- `review.decide`

风险级别建议：

- `reviews.bulk.approve`：`high`
- `reviews.bulk.reject`：`high`

并保持二次确认。

## 10. UI 细节建议

### 10.1 只在词条审核视图展示批量操作

`拼音审核` 不展示批量勾选，不展示批量动作条。

### 10.2 默认不支持跨页记忆勾选

第一版不做复杂的跨页勾选缓存。

支持方式：

- 当前页勾选
- 当前导入批次全部待审核

这样复杂度和风险都可控。

### 10.3 空操作保护

以下情况按钮应禁用：

- 未勾选任何任务，且作用域为 `selected_tasks`
- 当前 `importJobId` 下无待审核任务
- 当前身份无 `review.decide`

### 10.4 文案要求

确认文案必须说清“作用范围”：

- `确认批量审核通过当前勾选的 8 条词条审核任务？`
- `确认批量驳回导入批次 import_job_001 下的 23 条待审核词条任务？`

## 11. 导入批次场景下的最小闭环

建议目标流程：

1. 在 `批量导入` 中确认导入
2. 在导入详情查看“生成的审核任务”
3. 点击“前往词条审核本批任务”
4. 在 `词条审核` 页按导入批次上下文批量通过或批量驳回
5. 返回版本发布继续 build，或返回主数据修正后重新提交

这条链路正好解决“CSV 导入后仍要逐条审核”的核心痛点。

## 12. 第一版不做的事情

以下内容建议明确延后，不塞进第一版：

- 按任意筛选条件批量审核
- 按审核提交人、审核时间、业务属性等复杂条件批量审核
- 拼音候选批量审核
- release 审核批量通过
- 跨页勾选持久化
- 批量审核撤销

## 13. 收尾判定依据

只有同时满足以下条件，才算这批功能做干净：

- `词条审核` 页支持批量勾选和批量决策
- `导入批次详情` 能把用户带到“本批任务”的词条审核视图
- 批量通过 / 驳回只作用于 `term_review`
- 没有“按宽筛选误审全站任务”的入口
- 单条审核与批量审核的状态副作用一致
- 审计日志能保留每条任务的审批记录
- `test:unit`、`test:console`、`smoke:console` 补齐对应回归

## 14. 推荐结论

建议将本项单独立为新批次 `JOB-034`，按“导入批次上下文 + 词条审核页批量审核”推进，不与已关单的 `JOB-031 ~ JOB-033` 混做。
